# ch04-01-总体目标

## 本节目标

本节要写清项目总体想建成什么、服务什么对象、达到什么能力边界和验证水平。它是第四章的总领，不是简单写“建设一套平台”。

## 论证顺序

建议采用：

`建设依据 -> 项目定位 -> 建设对象 -> 关键能力 -> 应用或验证场景 -> 目标水平 -> 与后续问题、任务、成果的接口`

总述段应写清：

- 项目依据什么要求、面向什么建设对象、满足哪些核心指标或产业化指标。
- 项目相对现有能力要实现什么提升，例如效率提升、质量一致性、规范合规、数据沉淀、自主可控、推广复用等。
- 项目完成后形成什么系统、平台、工具、模型、数据集、知识库、报告能力或工程方法。
- 项目验证边界在哪里，哪些内容已经明确，哪些内容应以正式材料为准。

## 必须覆盖的内容

- 项目建设对象。
- 目标能力和能力边界。
- 应用对象、试点场景或验证场景。
- 量化指标或保守边界。
- 与项目解决问题、研发内容、预期成果之间的承接关系。

## 建设目标写法

建设目标应从“为什么建、建成什么、达到什么替代或应用效果”展开。建议按以下链路写：

`外部约束或业务卡点 -> 建设对象 -> 能力提升或自主替代目标 -> 工程应用和推广目标 -> 安全、生态或复用目标`

可写内容包括：

- 建设国产自主可控的软件平台、系统工具、数据底座、模型链路或工程能力。
- 降低外部工具依赖、断供风险、数据泄露风险和专业人员单点依赖。
- 对标国内外成熟工具或行业先进方法，但不得无依据写“国际领先”“国内首创”。
- 明确工程应用范围、验证对象、产品类型、性能目标、标准/专利/软著/报告等成果目标。
- 若用户材料包含指标，可列出技术指标、质量指标、应用指标或产业化指标；没有材料依据时只能用保守边界或“待补充”。

## 深度要求

- 至少落到 3 个以上目标维度，例如效率、质量、数据、合规、安全、推广、生态。
- 每个目标维度都要写成 `目标对象 -> 项目动作 -> 形成能力 -> 验证或应用边界`。
- 量化指标必须与资料口径一致；不得为了显得完整自行补准确率、覆盖率、回收期、成本节约金额等。
- 目标不能提前展开成完整技术路线，但必须给后续问题、研发内容和成果留下接口。

## 常用句式

- `为破解……问题，降低……风险，项目拟联合……力量，建设……。`
- `项目建成后，将形成……能力，支撑……场景，实现……。`
- `围绕……需求，项目将重点形成……、……和……等能力。`
- `相关指标以用户资料和正式验收方案为准，资料未明确部分采用保守表述。`

## 禁止事项

- 不要只写“建设一套平台”“提升智能化水平”。
- 不要把目标写成空泛愿景。
- 不要提前展开后续技术路线细节。
- 不要编造量化指标、推广范围、成果数量和验收结论。
