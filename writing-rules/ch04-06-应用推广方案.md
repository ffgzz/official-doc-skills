# ch04-06-应用推广方案

## 本节目标

本节要说明项目成果如何从试点验证走向内部推广、场景复制和持续迭代，而不是只说明“以后会推广”。推广方案必须服务本项目成果落地，不能写成市场宣传。

## 论证顺序

建议采用：

`推广条件总述 -> 试点验证 -> 示范推广 -> 规模复制 -> 反馈迭代 -> 预期推广效果和边界`

## 推广总述写法

开头应说明：

- 项目成果具备什么推广基础，例如应用场景、平台基础、单位协同、数据基础、业务刚需或已有工具链。
- 推广对象是谁，例如牵头单位内部、联合体成员、集团内部同类场景、产业链上下游或外部行业用户。
- 推广应遵循什么原则，例如先内部验证、再示范复制、最后按条件外部推广。

不得直接写“面向全国推广”“全面商业化”这类无依据口径。

## 推广路径要求

推广路径应写成“由内到外、由示范到规模化”的路径。

### 试点验证

写清：

- 试点对象或典型场景。
- 试点要验证哪些功能、性能、流程或用户体验。
- 如何收集问题和反馈。
- 试点结果如何进入模型、规则、模板、接口和流程优化。

### 示范推广

写清：

- 哪些单位、业务线或场景可以作为示范窗口。
- 示范推广需要满足哪些前置条件，例如数据准备、接口适配、权限开通、人员培训、流程确认。
- 如何形成可复制的部署包、操作规范、培训材料或案例库。

### 规模复制

写清：

- 可复制场景和不可直接复制场景的边界。
- 复制时需要重新适配的数据、规范、接口、模型或模板。
- 规模复制如何避免质量失控和数据安全风险。

### 反馈迭代

写清：

- 用户反馈收集机制。
- 问题分级和修复闭环。
- 模型、规则、模板、接口和文档的版本迭代方式。
- 推广过程中形成的数据和经验如何沉淀。

## 必须覆盖的内容

- 推广对象。
- 推广路径。
- 试点场景。
- 推广条件和边界。
- 反馈与迭代机制。
- 预期推广效果。

## 推广方式

根据资料选择，不得编造：

- 集团或牵头单位组织内部推广。
- 成员单位应用验证和示范复制。
- 培训、操作手册、试用环境或技术支持。
- 标准化部署包和接口适配方案。
- 行业交流、联合研发成果展示或非敏感成果推广。

如果资料不足，只能写原则性路径和待补边界。

## 预期推广效果写法

预期推广效果应与成果、应用场景和验证指标挂钩，可写：

- 提升工程交付效率。
- 降低人工重复劳动。
- 改善报告和校核质量一致性。
- 沉淀数据、知识和案例资产。
- 支撑相邻场景复用。
- 提升内部自主可控能力。

不得写无依据销售收入、市场份额、用户规模和投资回报。

## 深度要求

- 至少写清 3 个阶段。
- 每个阶段都要体现对象、路径、条件和保障。
- 必须说明反馈如何反向推动产品、模型、规则或流程迭代。

## 禁止事项

- 不要写成市场宣传。
- 不要只写“逐步推广应用”。
- 不要编造商业化收益、外部客户和推广规模。
- 不要替代第七章的正式推广治理机制。
