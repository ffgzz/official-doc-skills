# ch04-08-成果管理与内部受控复用策略

## 本节目标

本节要说明项目成果如何分类管理、如何内部受控复用、什么可以共享、什么不能开放、如何防范知识产权和保密风险。它继承旧“攻关成果开源策略”的治理深度，但当前项目口径应优先写“内部受控复用”，不得默认写成全部开源。

## 论证顺序

建议采用：

`成果分类 -> 内部受控复用 -> 对外发布边界 -> 知识产权与保密 -> 质量和安全审查 -> 反馈迭代与闭环管理`

## 成果分类写法

必须先把成果分类，再谈复用和开放。可按以下类别组织：

- 核心算法、模型、规则、知识图谱、模板引擎等核心技术成果。
- 数据集、训练语料、工程案例、历史报告、规范知识等数据和知识资产。
- 软件工具、脚本、接口服务、插件、部署包、报告模板等工程化成果。
- 专利、软著、论文、标准、技术文档、操作指南等知识产权和文档成果。
- 可对外展示的非敏感案例、白皮书、培训材料、接口说明或成果摘要。

每类成果都要说明敏感程度、使用对象、复用条件和管理边界。

## 内部受控复用要求

内部受控复用要写清：

- 复用范围：牵头单位内部、联合体内部、集团内部、特定业务场景或经批准的协作单位。
- 复用对象：模型、规则、模板、脚本、接口、报告框架、非敏感数据、测试用例或操作规范。
- 复用条件：数据脱敏、权限控制、版本确认、适配测试、保密承诺、业务审批或技术评审。
- 复用方式：组件封装、接口调用、部署包复用、模板复用、知识库引用、培训和操作手册。
- 复用反馈：使用问题、适配需求、质量缺陷和新场景需求如何回流。

不得只写“成果可在内部推广复用”。

## 对外发布和开放边界

对外发布必须保守。可写：

- 可公开发布的仅限非敏感成果摘要、论文、标准化方法、脱敏案例、通用接口说明、已授权知识产权信息等。
- 涉及企业内部数据、客户数据、船型数据、历史报告、规范知识库、模型参数、核心算法和未公开技术细节的成果不得直接开放。
- 任何对外发布、开源、交流或交付均应以正式审批和协议约定为准。

如果用户资料没有明确开源要求，不得写“全部开源”“开放源代码”“建立开源社区”等确定表述；只能写“在符合保密、知识产权和安全要求的前提下，可探索非敏感组件、接口文档或通用工具的受控开放”。

## 知识产权与保密要求

本节应说明：

- 成果权属、使用权、转让、许可和收益分配以联合研发协议或正式约定为准。
- 专利、软著、论文、标准等成果申请前应完成内部审查。
- 涉密或敏感数据、模型、报告和规则库应纳入访问控制。
- 成果复用和对外交流应避免泄露企业内部数据、客户信息、核心算法和未公开业务流程。

不得替代第七章正式“知识产权及权益分配”制度；本节只写方案视角的原则和边界。

## 质量和安全审查要求

受控复用前应说明必要审查：

- 功能可用性审查。
- 数据脱敏审查。
- 安全漏洞和依赖组件审查。
- 知识产权和许可证审查。
- 文档完整性和版本一致性审查。
- 典型场景适配测试。

资料不足时，不得补写具体审批部门和流程节点，可写“由项目相关责任主体按正式制度组织审查”。

## 闭环管理要求

成果管理必须形成闭环：

`成果登记 -> 分类定级 -> 权限配置 -> 复用申请或确认 -> 适配验证 -> 使用反馈 -> 版本迭代 -> 风险复盘`

每个环节可用自然段说明，不一定机械列清单，但必须看得出闭环。

## 必须覆盖的内容

- 成果分层分类。
- 内部共享范围。
- 敏感数据和敏感成果边界。
- 对外发布或开放边界。
- 知识产权与保密要求。
- 复用反馈和版本迭代闭环。

## 深度要求

- 至少区分 3 类成果边界。
- 必须写清哪些能复用、在什么范围复用、复用前需满足什么条件。
- 必须写清哪些不能开放或只能脱敏开放。
- 必须写出风险防范和闭环管理。

## 禁止事项

- 不要一刀切写“全部开源”。
- 不要只写原则，不写边界。
- 不要编造具体审批部门、协议文本和备案层级。
- 不要替代第七章的正式权益分配制度。
- 不要把受控复用写成市场推广。
