# ch04-05-技术路线

## 本节目标

本节是第四章最关键的小节，要说明项目准备按什么架构、链路、接口、验证方式和工程应用路径实现前面提出的目标和研发任务。技术路线不是技术名词清单，也不是研发内容的重复。

## 论证顺序

固定建议采用：

`总体技术路线总述 -> 架构分层 -> 数据流/业务流 -> 模型链/工具链 -> 模块和接口关系 -> 任务建设路线 -> 验证路线 -> 工程应用路线 -> 安全与自主可控 -> 合理性和可行性`

## 总体技术路线写法

开头必须先说明：

- 项目采用什么总体路线。
- 为什么该路线能够回应第一章痛点和第四章前序问题。
- 技术路线如何承接研发任务、预期成果和应用验证。
- 哪些能力来自现有基础，哪些能力需要本项目建设。

推荐句式：

- `在充分分析……功能需求的基础上，结合……技术及验证手段，项目总体技术路线按照……展开。`
- `该路线以……为输入，以……为核心处理链路，以……为输出，以……为验证闭环。`
- `按照……实际研发及工程应用流程，项目技术路线重点打通……。`

## 架构分层要求

技术路线必须写出架构分层，至少覆盖以下层级中的 4 类，并说明各层输入输出：

- 数据资源层：原始数据、历史报告、规范标准、业务记录、模型文件、图像或点云等。
- 数据处理层：解析、清洗、标注、特征提取、格式转换、质量校验。
- 模型算法层：小模型、规则模型、识别分类模型、推理模型、生成模型、优化算法等。
- 知识规则层：知识图谱、规范条款、规则引擎、模板规则、经验规则。
- 工程工具层：脚本工具、接口服务、中间件、报告工具、可视化组件、自动化流水线。
- 应用生成层：报告生成、结果校核、图表输出、用户交互、审查辅助。
- 验证运维层：单元测试、集成测试、样例验证、用户试用、反馈迭代、权限与日志。

每层不能只列名称，必须说明 `输入 -> 处理动作 -> 输出 -> 支撑后续环节`。

## 数据流和业务流要求

必须说明原始数据从采集、解析、清洗、特征提取、模型处理、结果生成、规则校核到报告输出的流转过程。

数据流应写清：

- 数据来源和格式。
- 数据进入系统后的处理步骤。
- 中间数据如何结构化、标准化和沉淀。
- 结果数据如何进入报告、知识库、模型训练或后续复用。

业务流应写清：

- 用户或业务人员如何触发流程。
- 人工审核、自动处理、规则校验、结果确认之间如何分工。
- 传统流程中哪些环节被自动化、哪些环节仍需人工复核。
- 业务闭环如何支撑质量一致性和可追溯。

## 模型链和工具链要求

不能把所有能力都交给大模型。必须说明小模型、规则模型、大模型、知识图谱、模板引擎或其他算法之间如何分工。

写作时至少区分：

- 识别、分类、提取、计算等确定性或半确定性任务由什么模型或工具承担。
- 规范、条款、模板、编号、格式等规则性任务由什么知识库或规则引擎承担。
- 报告段落生成、语义改写、说明文字组织等生成性任务由什么大模型或文本生成模块承担。
- 哪些环节必须保留人工审核或专家复核。

## 模块和接口关系要求

必须说明系统或工具模块之间如何连接，与既有软件、平台、数据库、脚本或报告工具如何交互。

至少写清：

- 内部模块之间的输入输出接口。
- 与既有 CAD/CAE/PDM/PLM、有限元软件、知识语料平台、报告工具或业务系统的连接方式。
- 接口数据格式、调用方式、权限边界或待补充边界。
- 接口不稳定、版本差异或数据质量问题如何通过适配层、校验层或人工复核处理。

## 任务建设路线要求

技术路线要逐项写明每项研发任务的建设步骤、关键难点、阶段输出和对后续任务的支撑，但不要重复 `项目研发内容` 的任务介绍。

每项任务路线建议写：

`前置条件 -> 实现步骤 -> 关键难点 -> 中间输出 -> 下游承接 -> 验证方式`

## 验证路线要求

必须说明技术路线如何被验证。可采用：

- 单元测试。
- 集成测试。
- 样例数据验证。
- 典型场景验证。
- 用户试用。
- 专家评审。
- 报告抽检。
- 系统联调。
- 上线试运行。

没有明确验收指标时，不得编造精确值；应写清验证方式和“指标以正式验收方案为准”。

## 工程应用路线要求

必须说明从实验样例到真实项目、从内部试点到多场景推广的实施路径。写清：

- 先在哪类典型场景验证。
- 验证通过后如何进入业务流程。
- 如何收集用户反馈并反向优化模型、规则、模板和接口。
- 哪些场景可以复用，哪些场景需要重新适配。

## 安全和自主可控路线要求

应说明数据安全、权限控制、私有化部署、国产化适配、知识产权和受控复用边界。不得把安全写成一句“加强安全管理”。

至少覆盖：

- 数据存储、处理和传输边界。
- 模型调用和知识库访问权限。
- 敏感数据、内部报告和未公开成果的隔离方式。
- 外部组件、开源组件或第三方平台使用边界。
- 与第八小节成果管理和受控复用策略的衔接。

## 合理性和可行性要求

技术路线末尾应从以下角度论证可行性：

- 技术成熟度。
- 团队基础。
- 数据基础。
- 工程场景基础。
- 单位协同基础。
- 风险防控和验证安排。

不得无依据写“技术成熟可靠、风险可控”。必须说明为什么可行。

## 图表要求

本节通常应包含总体技术路线图、模块关系图、数据流图或技术路线与任务指标对应表。若模板已指定图表，必须按模板执行。

建议表格字段：

`技术路线环节 | 对应研发任务 | 输入数据/条件 | 核心方法 | 输出成果 | 对应指标 | 验证方式`

图表后必须有解释段，说明底部数据资源、中部模型和工具链、上层报告生成与应用验证、旁侧安全保障和反馈迭代之间的关系。

## 深度要求

- 本节通常应是第四章最长的小节之一。
- 至少写清 4 层以上结构关系或等价链路。
- 必须包含架构分层、数据流、模型链、接口链、验证链和工程应用链。
- 不得只写一个概念图说明。

## 禁止事项

- 不要把技术路线写成技术名词清单。
- 不要只写“采用人工智能、大数据、知识图谱等技术”。
- 不要和研发内容完全重复。
- 不要把所有能力都归因于大模型。
- 不要无依据写“完全自动化”“100%准确”“全流程无人干预”。
