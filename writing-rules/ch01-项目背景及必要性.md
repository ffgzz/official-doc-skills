# ch01-项目背景及必要性

## 本章定位

本文件是第一章《项目背景及必要性》的章节写作规则库，不是背景素材库。

本章主任务是回答“为什么现在要做、为什么值得做、为什么必须做”，并为第四章项目建设方案提供问题来源、趋势依据和必要性论证。写作时必须服从 `chapter-prompts/ch01-项目背景及必要性.md` 的固定骨架，不得因为本规则内容较多而新增模板未登记的小节。

## 固定论证结构

本章固定采用以下结构：

- `（一）建设背景`
- `（二）建设意义`
- `（三）国内外发展现状及前景`
- `（四）预期解决重大问题`
- `（五）对产业链供应链韧性及安全的意义`

其中核心论证链必须形成：

`政策/战略牵引 -> 行业系统环境 -> 关键业务场景 -> 现有流程或工具短板 -> 本项目切入 -> 技术/产业/生态意义 -> 国内外比较 -> 痛点和前景 -> 重大问题 -> 安全与韧性价值`

## 资料和事实门禁

写本章前必须先消化 `reference-materials/<project-slug>/` 中与第一章有关的项目事实；若存在 `external-research-notes.md`，必须读取其中“第一章 项目背景及必要性”部分。

项目名称、建设对象、单位、任务、成果、指标、经费、周期、团队等事实，只能来自用户资料或正式项目材料；外部调研只能用于补充政策、行业现状、国内外比较、公开趋势、公开案例和通用方法论，不得覆盖用户给定事实。

政策、文号、发布日期、主管部门表述，优先使用官方来源核验。市场规模、占比、排名、订单量、装机量、市场份额等数字，优先使用主管部门、行业协会、研究机构报告或企业原始披露核验。平台、模型、产品能力和应用案例，优先使用官方产品页、论文原文、项目公告、权威报告或一手技术资料核验。

以下来源不得单独支撑确定性断言：

- 百科词条
- 博客、问答社区、论坛
- 资讯转载站、聚合新闻站
- 单篇营销稿、宣传稿

没有完成核验时，不得写具体文号、发布日期、精确市场数字、排名和明确产品能力断言。应降级为稳健表述，例如“近年国家层面持续推动……”“国外头部工业软件企业已在……方面形成较成熟产品体系”。

## 深度门禁

完整第一章不能写成简版背景说明。若用户未另行给出字数要求，默认按项目复杂度控制篇幅：

- 小型单技术项目：不少于 6000 中文字符。
- 中型三到五项任务项目：建议 9000 至 13000 中文字符。
- 大型平台、联合体或产业链项目：建议 13000 中文字符以上。

若用户提示词给出本章目标字数、建议区间或最低字数，必须优先执行用户要求。凡写明“最低不得少于 Z”，Z 是硬下限，不得按项目规模自行降低。

本章关键段落至少覆盖 `对象 / 现状 / 问题 / 原因 / 影响 / 项目切入 / 能力或前景` 中 4 项。若写完后只剩宏观口号、意义判断或政策摘要，必须回修。

完整第一章必须满足以下深度门禁：

- `建设背景` 必须写出 `政策或战略牵引 -> 行业系统环境 -> 关键业务场景 -> 现有流程或工具短板 -> 本项目切入`。
- `建设意义` 的技术、产业、生态三层意义必须分别说明 `项目动作 -> 形成能力 -> 作用对象 -> 预期影响`。
- `国内发展现状` 至少写出政策/行业应用基础、代表性单位或平台工具、现有能力边界中的 3 类对象。
- `国外发展与应用现状` 至少写出 2 至 3 类代表对象或路线，并分别说明核心能力、适用场景、局限性和对本项目的比较意义。
- `痛点分析` 至少拆成 4 类痛点，每类必须能回指后续第四章中的一项任务、技术路线或成果。
- `发展前景` 至少写出 3 条与本项目直接相关的趋势，每条都要说明本项目如何承接该趋势。
- `预期解决重大问题` 必须把前文痛点转化为项目必要性视角的问题链，不能替代第四章的工程化问题拆解。
- `对产业链供应链韧性及安全的意义` 必须落到自主可控、数据安全、平台安全、协同韧性、知识沉淀或生态支撑等具体对象。

## （一）建设背景写法

`建设背景` 是本章事实和问题底座，不能只写“政策支持、行业需要”。建议按 4 至 6 个自然段展开：

1. 第一段写国家战略、产业地位和宏观环境，可从高端装备、制造强国、海洋强国、工业软件、自主可控、数字化转型、人工智能融合应用等已核验依据切入。
2. 第二段写项目对象所在行业或业务链条的核心短板，落到具体对象，例如核心工业软件、设计制造数据链、仿真分析工具、知识库、规则库、协同平台、报告生成链路等。
3. 第三段写技术演进和行业转型，说明人工智能、数字孪生、知识图谱、模型驱动、数据要素、智能体协作等趋势如何改变本领域研发、设计、制造、管理或服务模式。
4. 第四段写本领域现实状态和建设窗口，既说明已有进展，也指出不平衡、不充分、缺少统一数据底座、缺少工程闭环、基础数据积累不足、集成化水平较低等现实差距。
5. 结尾段必须收束到本项目主题，用“亟须”“有必要”“本项目拟围绕”等正式表述，把建设背景自然导向后续建设目标和项目建设方案。

行业系统环境必须写厚。至少从以下维度选取 4 项展开：

- 产业链位置：本项目对象处于研发设计、仿真分析、生产制造、运维服务、合规审查或数据治理中的哪一环。
- 工具链关系：与 CAD、CAE、PDM/PLM、MES、ERP、CAPP/CAM、知识库、大模型平台等系统之间的连接关系。
- 数据链条：涉及哪些数据类型、数据流向、格式差异、沉淀方式和复用障碍。
- 业务流程：现有人工流程、系统流程、审查流程、协同流程如何运转。
- 标准规范：涉及哪些行业规范、审查规则、接口标准、安全合规要求。
- 组织协同：牵头单位、应用单位、科研单位、软件企业或上下游单位之间存在什么协同需求。
- 产业压力：进口依赖、断供风险、工程效率、质量一致性、专业人才不足、成本控制或交付周期压力。

行业系统分析不能停留在“行业数字化转型需求迫切”，必须写出系统、数据、流程、规范和组织之间的关系。

常用句式：

- `以……为代表的……是……的重要组成部分，也是……的重要支撑。`
- `在……新阶段，随着……在……领域的渗透，……已成为提升……水平的关键因素。`
- `然而，当前……仍存在……问题，特别是在……方面尚难满足……需求。`
- `面对……外部环境和……现实需求，亟须……。`
- `因此，本项目拟围绕……开展研究和建设，解决……关键问题，支撑……。`

## （二）建设意义写法

`建设意义` 不能写成泛泛价值判断。完整小节默认先写 1 个总述段，再按 `技术层面 / 产业层面 / 生态层面` 三个三级项展开。

总述段应先交代项目对象与国家战略、产业转型或行业能力建设之间的关系，再点出产业链自主可控、工程应用需求或行业升级压力，最后引出三层意义。

`技术层面` 应写：

- 项目对象在技术体系中的位置，例如统一模型、单一数据源、全生命周期数据管理、跨系统集成、流程协同、规则化表达、安全保障等。
- 项目解决哪些技术问题，例如多源异构、接口不统一、流程难追溯、知识难复用、专业精度不足、工程数据断链等。
- 项目形成什么技术能力，例如自主技术体系、统一数据模型、数据链路贯通、工作流管理、规则校核、定制化开发、安全防护等。

`产业层面` 应写：

- 项目如何回应工业软件卡点、国外产品依赖、高授权费用、供应链安全、企业成本和市场竞争问题。
- 项目如何服务产业数字化转型，例如设计制造全程数字化、开发周期缩短、市场响应能力提升、上下游协同效率提高。
- 项目如何与 ERP、MES、CAD、CAE、CAPP、CAM、知识平台或业务系统协同，推动产业链上下游整体优化。

`生态层面` 应写：

- 项目如何带动国产软件协同、人才培养、技术积累、标杆应用、接口开放或产学研用协同。
- 项目如何与国产软硬件、设计工具、仿真工具、制造系统和运维系统兼容适配。
- 项目如何由单个系统建设外溢为行业知识、开发能力和工程方法沉淀。

常用句式：

- `在技术层面，……是实现……的重要支撑。项目将围绕……特点，通过……建立……，支撑……。`
- `在产业层面，……有助于破解……问题，降低……依赖，提升……能力。`
- `在生态层面，……将带动……协同，沉淀……能力，推动形成……生态。`

## （三）国内外发展现状及前景写法

本节默认分为 `国内发展现状 / 国外发展与应用现状 / 痛点分析 / 发展前景` 四个三级项。

### 国内发展现状

国内现状按以下顺序写：

1. 先写行业或领域总体发展基础、政策支持和已有建设基础。
2. 再写国内代表性产品、单位、平台、项目或技术路线，说明已经具备哪些能力。
3. 随后写国产能力的现实限制，例如功能深度、工程应用成熟度、复杂场景适配、二次开发、与既有系统集成、行业份额、数据治理等。
4. 结尾必须引出“为什么仍需要本项目”，不能停在成果罗列。

国内现状至少覆盖以下 4 类信息中的 3 类：

- 政策和产业基础。
- 工程应用场景。
- 代表性工具或平台。
- 能力边界。

每类对象采用 `主体或路线 -> 已有能力 -> 应用限制 -> 与本项目关系` 的句式展开。

### 国外发展与应用现状

国外现状按以下顺序写：

1. 先列出代表性产品、企业、研究机构或技术路线。
2. 再分别说明其核心能力、典型应用场景、工程成熟度和生态优势。
3. 然后概括国际发展趋势，例如云化、平台化、模块化、协同化、AI 集成、供应链协同、全球团队协作等。
4. 最后与国内现状形成比较，说明差距在业务功能完整性、定制化能力、工程应用深度、生态集成能力、数据安全或用户体验上的具体表现。

国外现状至少选择以下 2 至 3 类对象展开：

- 头部工业软件或工程软件企业的 CAE、PLM、PDM、仿真后处理或报告模块。
- 国际船级社、行业机构或专业工具在规范校核、疲劳分析、结构评估中的应用。
- 国外研究机构或企业在工程知识图谱、自动化报告生成、AI 辅助仿真分析中的探索。

每类对象必须写出核心功能或技术路线、工程应用成熟度、与本项目场景的重合点、无法直接满足本项目需求的原因。结尾必须形成 3 至 5 个方面的差距归纳，为痛点分析和技术路线提供依据。

### 痛点分析

痛点分析通常先写 1 个总述段，再按若干痛点展开。每个痛点必须回答：

- 当前问题表现是什么。
- 在本项目业务场景中如何发生。
- 对效率、质量、协同、安全、合规、成本、交付或自主可控造成什么影响。
- 为什么需要本项目的统一平台、数据底座、规则体系、工具链或工程验证来解决。

痛点分类不得少于 4 类，优先从以下框架选择：

- 流程效率痛点。
- 数据治理痛点。
- 工具链集成痛点。
- 模型算法痛点。
- 规范合规痛点。
- 工程验证痛点。
- 安全自主痛点。

每类痛点必须写成 `问题表现 -> 成因 -> 影响 -> 对应建设任务`。无法对应后续任务的痛点应删除或改写。

### 发展前景

发展前景不是泛泛展望。每个前景段至少包含：

- 趋势判断。
- 本项目对象如何承接该趋势。
- 对行业流程、产业链协同、数据资产或生态能力的具体影响。

可围绕以下方向展开：

- 面向全产业链的数据一体化管理。
- 支持专业工程场景的智能分析、智能生成和智能校核。
- 整合一体化数字底座平台。
- 引入人工智能、数字孪生或智能决策能力提升产业运行效率。
- 支撑国产工业软件生态、接口开放和产业链协同。

不得写成“未来前景广阔、市场空间巨大、意义重大”。必须说明“广阔在哪里、由什么能力承接、如何作用于业务流程或产业链”。

## （四）预期解决重大问题写法

本节是必要性视角的问题收束，不是第四章 `项目解决的主要问题` 的替代品。写法应为：

`前文痛点归纳 -> 为什么必须由本项目处理 -> 项目拟解决的重大问题方向 -> 与后续建设方案的接口`

每个重大问题必须满足：

- 能回指建设背景或国内外现状中的差距。
- 能说明不解决会造成的效率、质量、安全、合规或产业链影响。
- 能对应第四章中的任务、技术路线或成果。
- 不提前写成完整研发任务、单位分工或验收口径。

问题标题建议写成“围绕……解决……问题”，避免写成口号。

## （五）对产业链供应链韧性及安全的意义写法

本节只写价值和意义，不写正式成果复用制度、推广制度、审批制度。应从 3 至 5 个角度展开：

- 自主可控：减少关键工具、专业知识、模型能力或外部平台依赖。
- 数据安全：推动业务数据、模型数据、规范知识和报告成果在可控环境中沉淀。
- 协同韧性：提升跨单位、跨系统、跨流程协同效率，降低单点人工依赖。
- 知识沉淀：把专家经验、规范规则和工程案例转化为组织级资产。
- 生态支撑：形成可复制方法和标杆场景，支撑产业链上下游协同升级。

每个角度必须写清 `风险或短板 -> 项目动作 -> 形成能力 -> 对产业链供应链韧性及安全的作用`。

## 写作步骤

1. 先识别本次要写的是完整第一章还是其中某个小节。
2. 从 `reference-materials/<project-slug>/` 抽取项目对象、痛点、任务、成果、指标和约束。
3. 从 `external-research-notes.md` 抽取政策、行业现状、国内外比较、公开趋势和公开案例。
4. 将证据分配到建设背景、建设意义、国内外现状、重大问题、安全与韧性五个部分。
5. 按正式公文文风组织正文，确保每个小节有事实、问题、项目切入和后续接口。

## 输出检查

写完后逐项检查：

- 是否服从模板固定标题和编号层级。
- 是否没有新增模板未登记小节。
- 是否形成政策、行业、场景、短板、项目切入的递进链。
- 建设意义是否按技术、产业、生态三层展开。
- 国内外现状是否不是资料堆砌，而是形成比较基线。
- 痛点是否至少形成 4 类，并能回指第四章任务或技术路线。
- 发展前景是否说明本项目如何承接趋势。
- 重大问题是否没有替代第四章工程问题拆解。
- 产业链供应链韧性及安全是否落到具体作用对象。
- 是否避免未经核验的具体数字、文号、日期、排名和市场份额。

## 禁止事项

- 不要只有宏观口号，没有行业和场景落点。
- 不要把建设意义写成单段价值判断。
- 不要把技术、产业、生态意义改成随意的战略、经济、社会三段式，除非用户明确要求。
- 不要把国内外现状写成“国外先进、国内不足”。
- 不要把痛点写成抽象抱怨。
- 不要把发展前景写成空泛展望。
- 不要无依据写“国际领先”“国内首创”“填补空白”。
- 不要新增模板未登记的小节或用连续流水号标题堆篇幅。
