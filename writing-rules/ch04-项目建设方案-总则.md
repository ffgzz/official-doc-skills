# ch04-项目建设方案-总则

## 本章定位

本文件是第四章《项目建设方案》的章级规则库。第四章是整份文稿中最工程化、最具体的一章，必须回答“建什么、解决什么、怎么建、产出什么、如何验证、如何推广、如何协同、成果如何受控复用”。

当前第四章采用拆分文件生成，但章级闭环必须统一。所有第四章子文件必须同时遵守本总则和对应子规则，不得把子文件写成互不关联的独立短文。

## 固定结构

第四章固定采用：

- `（一）总体目标`
- `（二）项目建设方案`

子文件归属固定为：

- `01-04` 属于 `（一）总体目标`
- `05-08` 属于 `（二）项目建设方案`

各子文件不得重复输出不属于自己的上级标题，不得新增模板未登记的二级节或三级项。

## 章级闭环

第四章必须建立完整闭环：

`目标 -> 问题 -> 任务 -> 技术路线 -> 成果 -> 验收 -> 推广 -> 协同 -> 成果管理与受控复用`

最低闭环要求：

- 问题必须能回指第一章痛点和国内外差距。
- 任务必须能回指问题。
- 技术路线必须能回指任务。
- 成果必须能回指任务和验收方式。
- 推广必须能回指成果形态、应用对象和验证场景。
- 协同必须能回指单位角色、任务接口和平台集成关系。
- 成果管理必须能回指成果类型、敏感边界、知识产权和复用范围。

若第四章写完后只能说明“项目准备做什么”，但不能说明“为什么这样做、怎么做、如何验证、谁协同、成果如何应用和受控复用”，应判定为不合格。

## 资料和事实门禁

写第四章前必须先消化 `reference-materials/<project-slug>/` 中与第四章有关的项目事实；若存在 `external-research-notes.md`，必须读取其中“第四章 项目建设方案”部分。

项目目标、建设规模、应用范围、任务划分、单位分工、成果清单、指标、周期、经费、推广范围、审批机制、开源或复用边界等项目事实，只能来自用户资料和正式项目材料。外部调研只能用于补充公开政策、行业趋势、通用技术路线合理性、公开案例和通用治理口径，不得覆盖或补造项目事实。

资料不足时，应写“待补充”“以正式材料为准”或降级为保守表述；不得编造具体指标、成果数量、验收口径、推广范围、审批流程和单位职责。

## 深度门禁

完整第四章不能写成“方案摘要”。若用户未另行给出字数要求，默认按项目复杂度控制篇幅：

- 单一工具或单一系统项目：不少于 15000 中文字符。
- 三到五项子课题或多单位协作项目：建议 24000 至 36000 中文字符。
- 大型平台、联合体或产业链项目：建议 36000 中文字符以上。

若用户提示词给出本章目标字数、建议区间或最低字数，必须优先执行用户要求。凡写明“最低不得少于 Z”，Z 是硬下限，不得按子课题数量、单位数量或资料简略程度自行降低。

第四章关键段落必须至少覆盖 `目标 / 问题 / 任务 / 路线 / 输出 / 验证 / 应用或保障` 中 4 项。若写完后只剩任务名、功能愿望和价值判断，必须回修。

完整章必须满足以下硬门禁：

- `项目研发内容` 与 `技术路线` 不得重复。研发内容写“做什么、分哪些任务、交付什么”，技术路线写“按什么架构、数据流、模型链、接口链、验证链去实现”。
- `技术路线` 必须是第四章最长的小节之一，必须包含总体架构、数据流、模块关系、接口关系、测试验证、工程应用和安全保障。
- 必须建立 `问题 -> 研发任务 -> 指标或目标 -> 成果 -> 验收口径 -> 应用场景` 的对应关系，优先用表格呈现。
- 每项核心研发任务必须写清输入、技术动作、输出、对应指标、验证方式和责任主体或协同主体。
- 每项量化目标或预期成果必须能回指至少一项研发任务和一项验收方式。
- 应用推广、联合研发、平台集成和成果管理不得写成口号，必须写对象、路径、接口、边界和闭环。

## 证据分配要求

各子文件写作前必须抽取本项可直接使用的项目资料。

最低资料抽取要求：

- `建设目标`：项目总体目标、量化目标、应用对象、验证边界和安全边界。
- `项目解决的主要问题`：业务痛点、效率问题、质量问题、数据沉淀问题、工程验证问题和合规问题。
- `项目研发内容`：子课题、子任务、关键技术动作、输入输出和单位或岗位分工。
- `预期成果与产业链供应链韧性及安全保障`：成果清单、成果数量或边界、应用成效、量化成效、自主可控、数据安全和产业协同资料。
- `技术路线`：阶段划分、数据链路、模型链路、系统架构、图示要求、接口关系和验证安排。
- `应用推广方案`：试点场景、推广对象、相邻场景复用、反馈迭代安排和推广边界。
- `联合研发与平台集成方式`：牵头单位、参研单位、平台对象、接口边界、联调机制和交付边界。
- `成果管理与内部受控复用策略`：成果分类、内部复用范围、脱敏共享、对外发布边界、知识产权、保密和审批约束。

若某个子文件在参考资料中已有明确项目事实，但正文没有消费，视为资料消费不合格。

## 统一文风要求

语言应保持正式、工程化、可研/立项/项目建议书口径。允许长段，但每段必须有任务、机制、输出或保障。

推荐句式：

- `本项目依照……要求，面向……场景，拟建设……。`
- `为解决……问题，项目将围绕……开展……，形成……。`
- `基于项目总体目标，遵循……思想，从……角度出发，将项目研发内容分解为……。`
- `针对……特点和……问题，开展……，输出……，支撑……。`
- `按照……实际研发及工程应用流程，项目技术路线……。`
- `通过……形成……，支撑……。`

避免：

- 只有任务标题，没有建设目标和建设方案闭环。
- 把技术路线写成技术名词堆叠。
- 把推广方案写成市场宣传。
- 把协同方式写成单位名单。
- 把成果管理写成一句“加强管理”。
- 无依据写“国际领先”“完全替代”“100%覆盖”等强结论。

## 图表总要求

第四章通常需要图表支撑。若章节模板指定图表，必须按模板执行。若资料足够，优先使用以下图表帮助建立闭环：

- 项目总体目标概念图。
- 主要建设任务分解表。
- 项目整体技术路线图。
- 技术路线与任务指标对应表。
- 任务、成果和验收口径对应表。
- 应用推广路径图。
- 联合研发与平台集成关系图或分工表。
- 成果受控复用边界表。

图表后必须有解释段，说明该图表如何支撑技术路线、任务闭环、验收管理、推广应用或受控复用。不得只放表格或图示不解释。

## 与其他章节边界

- 不重复第一章的宏观背景和必要性，只承接痛点并转化为工程问题。
- 不替代第五章的正式任务进度和里程碑安排。
- 不替代第六章的正式单位任务分工和交付边界。
- 不替代第七章的组织管理制度、权益分配和推广治理机制。
- 不写第八章预算、第九章效益测算、第十章系统风险清单。

## 输出检查

写完第四章后逐项检查：

- 是否服从 `（一）总体目标 / （二）项目建设方案` 的固定结构。
- 是否没有新增模板未登记小节。
- 是否建立 `问题 -> 任务 -> 技术路线 -> 成果 -> 验收 -> 应用` 闭环。
- 研发内容是否写成任务链路，而不是任务名清单。
- 技术路线是否写出架构分层、数据流、模型链、接口链、验证链和工程应用链。
- 应用推广是否写出路径、对象、条件、反馈迭代和预期效果。
- 联合研发与平台集成是否写出分工、接口、联调和交付边界。
- 成果管理与内部受控复用是否写出成果分类、复用范围、发布边界、知识产权和保密闭环。
- 是否避免无依据的具体指标、数量、排名、市场份额和强结论。
