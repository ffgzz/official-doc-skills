# ch04-02-项目解决的主要问题

## 本节目标

本节要把第一章中的背景痛点转化为第四章要解决的工程问题。它不是背景痛点的重复，也不是研发任务清单，而是“为什么这些问题需要在本项目建设方案中被工程化解决”。

## 论证顺序

建议采用：

`问题总述 -> 问题一 -> 问题二 -> 问题三 -> 问题之间的关联 -> 与后续研发内容和技术路线的接口`

每个问题都按以下链路展开：

`问题表现 -> 发生场景 -> 成因 -> 影响 -> 本项目切入方向 -> 对应后续任务或技术路线`

## 问题标题写法

问题标题应写成“动作 + 对象 + 解决什么问题”的形式，例如：

- `构建统一数据处理链路，解决多源工程数据难贯通问题`
- `建立节点识别与分类机制，解决人工判断效率和一致性不足问题`
- `完善规范知识和动态模板体系，解决报告生成与合规校验分离问题`
- `形成工程验证闭环，解决算法原型难以进入真实业务流程问题`

不得只写“效率问题”“质量问题”“数据问题”这类过空标题。

## 必须覆盖的内容

- 至少 3 类关键问题；复杂项目建议 4 至 6 类。
- 每类问题的业务发生场景、数据或系统对象。
- 每类问题的直接后果，例如效率、质量、协同、安全、合规、成本、交付或自主可控影响。
- 每类问题对应的建设方向、研发任务或技术路线接口。

## 深度要求

- 每个问题必须能回指第一章痛点或国内外现状差距。
- 每个问题必须能导向后续 `项目研发内容` 中至少一项任务。
- 不能只写“问题存在”，必须写“为什么现有流程、工具、数据或组织方式解决不了”。
- 问题之间要避免重复：流程效率、数据治理、模型算法、工具集成、规范合规、工程验证、安全自主应各有侧重。

## 可选问题框架

根据项目资料选择，不要机械全用：

- 流程效率问题：人工识别、人工录入、人工复核、人工制图、人工编写或人工审查导致周期长、返工多。
- 数据治理问题：数据格式不一、结构化程度低、历史数据不可复用、数据质量不可控、语料和知识难沉淀。
- 工具链集成问题：CAD/CAE/PDM/PLM/脚本/报告工具之间接口不统一、数据流断裂、模型和结果难贯通。
- 模型算法问题：复杂几何、拓扑、工况、规范规则导致识别、分类、推理或生成准确性不足。
- 规范合规问题：行业规范、船级社规则、报告模板、图表编号、单位和格式要求难以自动校核。
- 工程验证问题：算法原型与真实工程流程脱节，缺少典型场景、验收口径和用户反馈闭环。
- 安全自主问题：关键工具、专业知识、企业数据和模型能力存在外部依赖或信息安全风险。

## 禁止事项

- 不要简单重复第一章痛点。
- 不要把问题写成口号。
- 不要直接展开成正式研发任务。
- 不要把没有后续任务承接的问题硬塞进来。
