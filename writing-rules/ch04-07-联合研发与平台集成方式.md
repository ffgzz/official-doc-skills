# ch04-07-联合研发与平台集成方式

## 本节目标

本节要写清多方如何协同研发，以及成果如何集成到实际平台、系统或工具链中。它强调方案视角下的联合研发逻辑和平台集成关系，不替代第六章正式单位分工表。

## 论证顺序

建议采用：

`联合研发总体模式 -> 各类主体角色 -> 研发分工和接口 -> 平台集成对象 -> 联调验证机制 -> 协同闭环`

## 联合研发总体模式写法

开头应说明：

- 项目为什么需要联合研发。
- 牵头单位、研发单位、应用单位、平台单位、高校或外部支撑单位之间如何形成互补。
- 联合研发如何围绕目标、问题、任务和成果展开，而不是临时拼接资源。

可采用 `用户需求和场景反馈 -> 企业开发测试与集成 -> 科研机构或高校提供关键技术支撑 -> 应用单位示范验证 -> 反馈促进迭代` 的闭环表述。

## 角色分工写法

可参考“产学研用”逻辑，但必须适配现有单位和资料，不得照搬空泛分类。

常用口径：

- `产`：软件设计、开发、测试组织、部署实施、应用推广和运维服务。
- `学`：基础理论、关键算法、人才培养和学科支撑。
- `研`：行业知识、核心模块、专业方法、测试评价和技术攻关。
- `用`：应用场景、工程验证、反馈迭代、示范推广和需求确认。

如果项目资料没有高校、外部企业或特定单位，不得编造；可改写为“牵头单位、研发支撑单位、应用验证单位、平台支撑单位”。

## 研发分工和接口要求

每类主体都要写：

- 承担什么研发或验证任务。
- 输入什么资料、数据、场景或技术能力。
- 输出什么模块、工具、模型、报告、接口或验证结论。
- 与其他主体之间的接口关系。

不得只罗列单位名称和“负责协同攻关”。

## 平台集成方式写法

必须说明成果如何集成到实际平台、系统或工具链中。至少写清：

- 集成对象：知识语料平台、大模型平台、业务系统、有限元软件、报告工具、数据平台、内部门户或其他系统。
- 集成方式：接口调用、插件嵌入、脚本集成、中间件适配、数据同步、模板挂接、服务封装等。
- 集成边界：哪些模块进入平台，哪些模块保持独立工具，哪些能力只在内部受控环境使用。
- 数据和权限边界：输入输出数据、访问权限、日志留痕、敏感数据隔离。

资料不足时，不得补写具体接口协议、审批部门和上线流程，只能写原则性集成方式和待补项。

## 联调验证机制

本节必须写出协同研发如何被验证。可包括：

- 模块级联调。
- 数据接口联调。
- 典型场景联调。
- 报告输出抽检。
- 用户试用反馈。
- 专家或业务部门评审。

每条联调机制要说明验证对象、参与主体、输出结果和问题回流方式。

## 必须覆盖的内容

- 牵头单位与参与单位角色。
- 分工边界。
- 平台接口或集成对象。
- 协同研发和联调机制。
- 集成边界和验证方式。

## 深度要求

- 至少说明 2 条以上协同接口关系。
- 至少说明 2 类平台或工具集成对象，除非资料明确只有单一平台。
- 要写出 `需求 -> 研发 -> 集成 -> 验证 -> 反馈` 的闭环。

## 禁止事项

- 不要写成单位名单。
- 不要只写“联合攻关”“协同推进”。
- 不要替代第六章正式任务分工。
- 不要编造具体协议、审批流程、接口标准和上线安排。
