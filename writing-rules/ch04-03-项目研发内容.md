# ch04-03-项目研发内容

## 本节目标

本节要把项目真正要做的研发任务拆清楚。它回答“做什么、分哪些任务、各任务交付什么”，不回答完整“按什么技术路线实现”，后者属于 `技术路线`。

## 论证顺序

建议采用：

`建设范围或应用范围 -> 研发内容总述 -> 任务分解 -> 子任务展开 -> 任务闭环表 -> 与预期成果和技术路线的接口`

每个任务至少写：

`任务目标 -> 任务输入 -> 关键技术或工程动作 -> 任务输出 -> 对应问题/指标 -> 验证方式 -> 与其他任务关系`

## 任务分解要求

任务分解建议采用 3 至 8 项，具体数量以项目资料为准，不得为了凑结构编造任务。

完整软件、平台、智能工具或工程化项目可从以下方向拆分：

- 顶层设计或总体架构设计。
- 业务场景和应用流程研究。
- 数据架构、数据治理或关键数据技术。
- 模型算法、规则引擎、知识图谱、模板引擎或大模型协同技术。
- 核心功能模块研发。
- 接口集成和工具链贯通。
- 测试验证、工程验证和质量校核。
- 示范应用、部署封装或复用支撑。

每个任务的总段建议写成：

`针对……特点和……问题，项目拟开展……，形成……，为……提供支撑。`

每个子任务段建议写成：

`子任务名称 -> 研究/建设对象 -> 技术动作 -> 输出成果 -> 支撑后续环节`

## 必须覆盖的内容

- 任务边界：本任务负责什么，不负责什么。
- 任务输入：来自哪些数据、系统、规范、场景、用户需求或前序任务。
- 关键动作：采用什么模型、算法、架构、接口、规则、模板、测试或工程方法。
- 任务输出：形成什么算法、模块、脚本、平台、数据集、模板、报告、规范或验证结果。
- 对应目标或指标：支撑效率、质量、准确率、覆盖范围、安全、自主可控或推广目标中的哪一类。
- 验收口径：通过样例数据、典型场景、用户试用、对比测试、专家评审、报告抽检、系统联调或上线运行中的哪种方式确认。

## 任务闭环表

资料足够时，优先形成任务闭环表。建议字段：

`任务编号 | 任务名称 | 对应问题 | 关键技术动作 | 主要交付物 | 对应指标/目标 | 验收口径 | 责任单位或协同主体`

若用户材料没有明确验收指标，不得编造精确指标；可写“建议在验收方案中进一步明确”，但仍要说明验证方式。

## 与技术路线的边界

研发内容写：

- 分哪些任务。
- 各任务做什么。
- 输入输出是什么。
- 交付物是什么。
- 怎样确认任务完成。

技术路线写：

- 总体架构如何分层。
- 数据流、业务流、模型链和接口链如何运转。
- 各任务如何串联实现。
- 如何测试验证、部署应用和安全保障。

如果本节和技术路线出现大段重复，应压缩本节中的实现细节，把详细链路留给技术路线。

## 深度要求

- 至少形成一套任务闭环。
- 每项核心任务必须有输入、动作、输出、验证。
- 任务之间必须体现“前置数据/架构 -> 核心模型/工具 -> 集成验证 -> 应用输出”的链路。
- 任务要能支撑后续成果和验收说明。

## 禁止事项

- 不要只有任务标题。
- 不要只有“研究……技术、实现……功能”。
- 不要和技术路线完全重复。
- 不要编造责任单位、验收指标和交付数量。
