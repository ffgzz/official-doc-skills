# 公文复核 Skill

> 当前复核只依据用户 brief、章节计划、事实台账、正文散件和装配稿。
>
> 当前 review 核心检查点是：
> - `official-doc-research` 是否真的完成了独立调研门禁，而不是只搜两三次
> - `research-evidence.md` 是否为每条有效来源保留了可复用证据，而不是只保存链接列表
> - `project-brief.md` 与 `00-section-plan.md` 是否记录了全文编号方案，正文是否按同一方案执行
> - 用户要求的章节是否写完
> - 五类共性章节是否完成搜索留痕
> - `创新点 / 技术成果 / 技术指标` 是否与项目建设方案中的研发内容和建设任务闭环
> - 五类共性章节是否形成了“背景差距 -> 项目建设方案 -> 创新机制 -> 成果 -> 指标”的论证链
> - 项目建设方案、成果、指标是否符合附件式“针对问题 -> 研究/构建 -> 实现 -> 形成/验证”写法
> - 正文是否仍存在明显 AI 痕迹、列表化正文或加粗小标签
> - 用户要求的表图是否齐
> - 字数是否满足 brief
> - 正文中的具体事实是否已在 `facts-ledger.md` 中完成核验
> - 正式稿中的图表是否装配在对应章节附近，而不是仅在文末集中附后
> - `已核验` 事实所依赖的来源等级是否达标，而不是只靠百科/博客/资讯转载

## 适用时机

以下任一情况出现时，应主动使用本 Skill：
- 用户明确要求检查、复核、体检、定稿前核验
- 当前一轮正文、表格、图示已经生成，需要阶段性检查
- 你准备告诉用户“这一轮已完成”
- `official-doc-assemble` 刚刚完成，需对 `formal-draft` 做装配后 final review

## 核心目标

对已经生成的骨架、正文、表格、图示做结构化体检，输出可执行的问题清单，而不是只给一句泛泛评价。
`review` 不是终点，它要为下一步 `revise` 提供明确输入。

## 使用前必读

开始复核前，至少读取以下文件：
- `workspace/plan/<project-slug>/project-overview.md`
- `workspace/plan/<project-slug>/project-brief.md`
- `workspace/plan/<project-slug>/stage-gates.md`
- `workspace/plan/<project-slug>/research-plan.md`
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/research-notes.md`
- `workspace/plan/<project-slug>/facts-ledger.md`
- `workspace/plan/<project-slug>/progress.md`
- 已生成的 `workspace/outputs/<project-slug>/`、`workspace/tables/<project-slug>/`、`workspace/figures/<project-slug>/` 文件
- `.claude/skills/official-doc-core/references/attachment-writing-patterns.md`

如果 `workspace/assembled/<project-slug>/formal-draft.md` 已存在，复核时也必须读取它，并把它视为“当前对外版本”。
不要只根据旧 review、旧装配说明或记忆中的字节数下结论。
若本轮 review 是由 `official-doc-assemble` 触发的 final review，应以 `formal-draft` 为主检查对象，而不是只看散件文件。

若发现 `workspace/plan/<project-slug>/research-drafts/` 已存在，但 canonical `research-evidence.md` 或 `research-notes.md` 缺失，复核不得自行把分组草稿拼起来视为“已完成”。这属于 Must Fix：调研草稿未合并到正式总账。

若 `project-overview.md` 或 `progress.md` 中记录了目标总字数，本轮 review 还必须检查：
- 当前稿件总字数是否落在用户要求范围内
- 若未落在范围内，必须直接记入 Must Fix，并阻断进入 assemble

## 强制取数规则

review 不得复用旧 review、旧 assembly-notes、progress 中的结论性文字来判断“已通过”。所有结论必须从当前文件重新取数。

### 调研来源取数

来源数量必须从 `workspace/plan/<project-slug>/research-sources.md` 的实际来源行重新统计，而不是从“来源统计”表、`progress.md` 或旧 review 的描述中照抄。统计时按调研组分别计算：
- 有效来源数：该组实际来源记录行数。
- 近 3 年来源数：发布时间或时效标记显示在当前年份及前 2 年内的来源数。
- `A主源` 数、`B辅源` 数：按来源等级字段逐行计数。
- 证据卡数：到 `research-evidence.md` 中查找同编号证据卡，只有存在证据卡且记录抓取/转换状态、正文可读性、结构化证据点、原文位置和使用边界，才算完成。

若 `research-drafts/` 下存在分组来源或证据草稿，而 canonical `research-evidence.md`、`research-notes.md` 缺失或明显落后，review 必须直接记为 Must Fix，不能按草稿目录替代正式总账判通过。

正式可研、立项、项目建议书默认执行以下硬门槛：
- 普通激活组：不少于 8 条有效来源、近 3 年不少于 5 条、`A主源` 不少于 2 条、`B辅源` 不少于 3 条。
- `BG`：不少于 12 条有效来源。
- `TI`：不少于 10 条有效来源。
- `IN`：不少于 8 条有效来源，且必须包含外部方案或技术路线比较。

任一激活组未达标时，必须记为 Must Fix，结论只能是“调研门禁未通过 / 暂不可作为正式交付稿”。不得把全文总来源数相加后冒充某个调研组达标，不得写“BG组5条、其他组共10+条，所以BG达12条”。

### 风格脚本取数

运行 `scripts/style_check.ps1` 后，必须读取脚本输出中的命中行和 `Result`。只要脚本输出 `Result: FAIL`，review 必须记为 Must Fix；不得只写“已执行 style 检查（通过）”。若脚本无法运行，必须用 `rg` 手工检查以下模式，并在 review 中说明替代检查结果：
- `\*\*子任务`
- `\*\*技术描述\*\*|\*\*技术难点\*\*|\*\*作用\*\*`
- 行首项目符号或正文编号清单
- 同一层级编号混用

若要判断 style_check 的命中是否来自 Mermaid 或表格，必须逐行核对命中行是否位于 fenced code block 或 Markdown 表格内。命中行在普通正文中时，不得解释为“Mermaid 误触发”。正式稿中出现 `Result: FAIL` 时，除非所有命中行均在已排除区域内，否则不得标记为通过。
若 `Result: FAIL` 的命中项属于 `markdown heading uses decimal outline`、`shallow slogan phrases`、`mechanical transitions`，且命中行在普通正文中，也不得用“英文术语误报”或“公文常用句式”一笔带过；这类命中默认需要回修。

## 统计口径（强制统一）

凡涉及章节比例、总字数、`【待补】` 数、final review 结论时，必须统一使用以下统计口径：

### 正文字数统计

- 以当前检查对象为准：
  - 普通 review：优先统计 `workspace/outputs/` 散件正文
  - final review：必须统计 `workspace/assembled/<project-slug>/formal-draft.md`
- 以下内容不计入“正文总字数”：
  - Markdown 表格单元格
  - Mermaid 代码块
  - 以 `>` 开头的装配引用提示行
  - 仅用于引图引表的标题提示行
  - notes / assembly-notes / review 文本
- 章节比例的分母必须是“同一统计口径下的全部正文字数”，不得混入表格、Mermaid、notes

### `【待补】` 统计

- 默认同时给出两种口径：
  - `标记数`：字面出现了多少次 `【待补】`
  - `字段数`：有多少个待补字段 / 行 / 单元格组
- 若只写一个数字，必须明确标注是“标记数”还是“字段数”
- final review 与 assembly-notes 必须使用同一种口径，不得一个按字段数、一个按标记数

复核章节重量时，不再套用旧模板名。
应根据以下信息综合判断：
- `project-brief.md` 中的章节要求
- `00-section-plan.md` 中登记的章节顺序与轻重
- 用户明确给出的每章字数
- 当前正文是否真的把重章节写厚、轻章节写短

## 检查维度

### 1. 标题完整性
- 一级标题是否齐全
- 二级标题是否遗漏
- 是否擅自改写模板固定标题
- 编号顺序是否连续
- `project-brief.md` 与 `00-section-plan.md` 是否记录全文编号方案
- `00-section-plan.md` 是否同时记录 `用户章节标记` 与 `正式输出标题`；正式稿一级标题必须使用正式输出标题
- 若目标文体为可研报告、立项申请书、项目建议书或技术总结，且全文编号方案为正式报告型，正式稿不得输出 `# 第一章 概述`、`# 第二章 ...`，应输出 `# 一、概述`、`# 二、...`；出现即 Must Fix
- 若 `project-brief.md` 写成“章节数量：7章（不含第五章）”或类似自相矛盾口径，必须记为 Must Fix，并要求按用户实际提供章节数和原始标记重写
- 同一层级是否混用 `（1）`、`1.1`、`1.1.1` 等编号
- 普通章节的四级标题不得写成 `#### 11.1 ...`、`#### 13.2 ...`；除项目研发内容内部任务化拆解外，十进制目录标题污染全文编号时必须记为 Must Fix
- 项目研发内容内部若采用 `任务1 / 子任务1-1` 等拆解，是否只在该层级内使用，未污染全文其他章节编号
- 若全文二级节方案为 `（一）`，章内不得出现 `### 一、...`、`### 二、...` 这类把一级编号下放到二级标题的情况；出现即 Must Fix
- 若普通章节出现 `### 10.` 及以上的连续流水号标题，或同一二级节下堆叠大量短标题而不是自然段展开，应记为 Must Fix。修复动作是合并标题、压缩为少量必要三级标题，并把内容改写到既有二级节自然段中
- 若出现 `历史意义`、`时代价值`、`远景展望`、`政策建议`、`社会责任`、`一带一路`、`国际合作机遇`、`可持续发展规划`、`人才培养` 等与 `00-section-plan.md` 无关的泛化扩写标题，应记为 Should Fix；如果它们用于凑字数并稀释主章节，应记为 Must Fix
- 若同一文件出现完全相同的 Markdown 标题文本，应记为 Must Fix，并要求合并到第一次出现的位置
- `子任务`、`项目研发内容`、`创新点` 等正式层级不得用 `**加粗文本**` 伪装标题，应使用与 `00-section-plan.md` 匹配的 Markdown 标题层级或正文自然段
- 对每一章，必须把正式稿实际出现的二级节与 `00-section-plan.md` 允许清单逐项比较。正式稿不得新增计划外二级节；例如计划只允许第四章 `（一）总体目标`、`（二）项目建设方案` 时，新增 `（三）技术创新与突破`、`（四）研究方案可行性分析`、`（五）工作基础` 等都属于 Must Fix

### 2. 图表一致性
- 图号、表号是否连续
- caption 是否与目录一致
- 正文是否正确引用图表
- 图表内容是否与正文口径一致
- 已装配进正式稿的表图内容，是否与 `workspace/tables/`、`workspace/figures/` 中最新文件一致

### 3. 事实一致性
- `official-doc-research` 是否为激活调研组建立了调研计划
- 每个激活调研组是否至少完成 4 轮检索，而不是只做 2 到 3 次零散搜索
- `BG`、`TI` 等高风险组是否完成追加近年更新检索
- 每个激活调研组是否达到最低保留来源量，且高风险组是否达到 `BG` 不少于 12 条、`TI` 不少于 10 条、`IN` 不少于 8 条的门槛
- `research-evidence.md` 是否存在并为 `research-sources.md` 中每条有效来源建立同编号证据卡
- 证据卡是否记录抓取/转换状态、正文可读性、结构化证据点、原文位置/页码/章节和使用边界
- 来源是否以近 3 年资料为主，超窗来源是否明确标记为 `历史基线` / `基础规范`
- 项目名称是否统一
- 单位全称/简称是否统一
- 负责人、成员、职责是否冲突
- 起止时间和阶段计划是否冲突
- 成果数量、预算口径是否冲突
- 正文中的具体政策、文号、发布日期、市场数字、比例、排名、产品能力表述，是否能在 `facts-ledger.md` 中找到对应 `已核验` 记录
- 若 `facts-ledger.md` 中状态为 `待核验` 或 `仅作趋势参考`，正式稿是否仍写成了确定性强表述
- `research-sources.md` 中支撑这些事实的来源，是否包含至少 1 条 `A主源`
- 是否把 `C参考` 来源单独当成了 `已核验` 的依据

### 4. 风险项
- 是否存在大量 `【待补】`
- 是否存在明显空话套话
- 是否存在“打造平台、赋能业务、形成闭环、显著提升”等口号化表述，但没有对象、机制、输出或验证方式
- 是否存在正文列表化、演示稿化、答题卡化
- 是否存在 `**技术描述**`、`**技术难点**`、`**作用**` 之类加粗小标签
- 是否存在 `**子任务...**`、`**创新点...**`、`**系统与平台研发类指标**` 等用加粗文本充当小标题的情况；若是正式稿，默认记为 Must Fix
- 是否存在 `首先、其次、最后、此外、另外、接下来、总之` 等机械过渡词
- 是否存在“研究台账里只有 2 到 3 条来源，却已开始大段写正文”
- 是否存在“来源大多是 3 到 4 年前旧信息，却被当成当前现状依据”
- 是否存在“research-plan / research-notes 仍接近空白，但正文已写得很实”
- 是否存在“research-sources 有来源列表，但 research-evidence 缺失或只有链接，没有抓取内容、证据点和使用边界”
- 是否存在“research-evidence 已很厚，但正文关键段落仍主要靠泛化判断，没有把事实、差距、机制、成果或验收口径消费进去”
- 是否存在“图表已有但正文未引用”
- 是否存在“正文提到表图但文件未生成”
- 是否存在“项目建设方案只有标题和功能愿望，没有按模板写清总体目标、主要问题、项目研发内容、预期成果、技术路线、应用推广、产学研用合作和开源策略”
- 是否存在“成果和指标看起来像泛泛清单，没有成果实体、交付形态、测试条件和验证方法”
- 是否存在“正文已写具体事实，但 facts-ledger 无对应 `已核验` 记录”
- 是否存在“应当概括写的趋势性信息，被写成具体文号、具体日期、精确市场数字”
- 是否存在“facts-ledger 标记为 `已核验`，但 research-sources 里主源只有百科、博客、论坛、自媒体或资讯转载”
- 是否存在“散件文件已有明确值，但正式稿回退成 `【待补】`”
- 是否存在“review / assembly notes 中的统计口径明显滞后于当前文件”
- 是否存在“章节比例或阈值明明不满足，但 review 仍误判为通过”
- 是否存在 `workspace/plan/<project-slug>/agent-skill-trace.md` 缺失；若本轮使用过 writing 子代理却没有该文件，应记为 Must Fix
- 若本轮属于整稿生成、多章节生成或章节批量重跑，`agent-skill-trace.md` 是否为空；正常多章节写作下台账为空应直接记为 Must Fix，不得以“主控直接写作”解释通过
- 若本轮属于整稿生成、多章节生成、章节批量重跑或正常项目调研，是否存在 `workspace/plan/<project-slug>/agent-prompts/` 缺失或明显为空；若主控声称派发了子代理却没有任何实例化任务单落盘，应记为 Must Fix
- 若 `agent-prompts/` 目录存在但为空，也应按“未真正派发子代理”处理，不得把“目录已创建”误判为流程合规
- `agent-prompts/*.md` 是否真正保留了任务单硬区块；若任务单被改写成 `任务标识 / 内容要求 / 禁止编造内容` 这类自由说明文，而缺少 `【主控已预载专项 skill】`、`【任务级留痕要求】`、`【提交物】`，应记为 Must Fix：子代理任务单未按模板生成
- 命中正文专项内容的 `agent-prompts/*.md` 中，`【主控已预载专项 skill】` 是否被错误写成 `否`、`无` 或“需要子代理自行加载”；若出现，应记为 Must Fix：主控未先加载对应正文 skill
- `agent-skill-trace.md` 是否为每个 writing 子代理分别记录了任务编号、任务单要求加载的专项 skill、显式 skill 调用指令、主控已预载专项 skill、子代理实际显式加载的专项 skill、显式调用指令原文、skill 加载确认原文和一致性结论
- 是否存在 `progress.md` 已写“已加载 skill / 已完成写作”，但 `agent-skill-trace.md` 没有对应记录，或记录结论仍为 `不匹配 / 缺失`
- 是否存在“命中专项内容的子代理写作结果没有 `已显式加载的 skill 清单与顺序` 留痕，或非专项章节子代理没有留下 `本任务不需要显式加载专项写作 skill` 声明，却被主控当成合格结果”
- 是否存在“命中专项内容的写作/回修任务没有留下 `主控已预载专项 skill`、`本节证据清单摘要`、`本节写作规则摘要`，却已被主控当成合格结果”
- 若本轮 revise 修改了五类正文专项章节，`progress.md` 中是否存在明确的 `本轮 revise 加载的正文 skill` 留痕；若缺失，应记为 Must Fix
- 上述 revise 留痕是否同时写明了受影响文件/小节、证据清单摘要和规则摘要；若只写“已回修完成”而无专项 skill 名称，应记为 Must Fix
- 若 `progress.md` 仅写“若后续需要将重新加载对应 skill”“下轮再补记录”这类未来时说明，而没有本轮实际加载记录，应仍记为 Must Fix；未来承诺不能替代本轮留痕
- 若本轮属于正常多章节写作，而 `progress.md` 中出现“本轮主控直接写作，无需子代理留痕”“agent-skill-trace.md 留空合理”或等价说明，应直接记为 Must Fix，不得把它解释成合理豁免
- 若本轮属于正常项目调研，而 `progress.md` 中出现“主控直接调研，无需 research 子代理”“research 阶段未派发子代理也合理”或等价说明，应直接记为 Must Fix
- 若 `progress.md` 只写了“调用 official-doc-project-background / official-doc-research-content / official-doc-technical-achievements”等主控层 skill 名，而没有任何 research/writing 子代理任务记录，不得把这视为子代理已执行；应直接记为 Must Fix：主控加载 skill 代替了子代理执行
- 若 `formal-draft.md`、章节散件或图示说明稿中出现 `【来源：...】`、`来源：原始资料`、`BG-1-01 / RC-1-01 / IN-1-01` 这类内部来源标记，应直接记为 Must Fix：内部证据标记进入正式正文
- 若 `workspace/outputs/<project-slug>/` 或 formal-draft 中出现 `research-mapping.md`、`检索与资料映射说明`、`调研执行概况`、`来源分组与章节映射` 等内部过程说明，应直接记为 Must Fix：内部过程材料误入正文/装配
- 若任一 `.mmd` 未通过 `python scripts/check_mermaid.py <file>`，或已有图示出现 `Syntax error in text` / Mermaid 解析失败占位，应直接记为 Must Fix：图示语法错误
- 是否存在“表图被整体堆到文末附图 / 附表，而非放在对应章节附近”
- 是否存在“英文术语污染或错词”，例如 `C广告matic` 这类中英文夹杂错写
- 是否存在“复杂度、公式或符号残句”，例如 `在的时间复杂度内`、`复杂度降至约，其中`、`协调层复杂度为约`
- 是否存在“无证据支撑的强结论”，例如 `填补国内技术空白`、`国际领先`、`首创`、`唯一`
- 是否存在“同义反复的大段重复扩写”，尤其在后部长章节或成果/指标密集章节通过重复价值表述凑字数

### 4.5 写作深度
- 背景类章节是否形成 `建设背景 -> 建设意义 -> 国内外发展现状及前景` 的链条；完整背景章是否包含 `技术层面 / 产业层面 / 生态层面` 的建设意义，以及 `国内发展现状 / 国外发展与应用现状 / 痛点分析 / 发展前景` 的现状前景结构
- 项目建设方案是否形成 `总体目标 -> 项目建设方案` 的完整结构；完整章是否包含 `建设目标 / 项目解决的主要问题 / 项目研发内容 / 预期成果 / 产业链供应链韧性及安全保障 / 技术路线 / 应用推广方案 / 产学研用合作方式 / 攻关成果开源策略`
- 项目研发内容是否每项都写出 `问题来源 -> 建设任务 -> 技术或工程动作 -> 输入输出 -> 与其他任务的关系 -> 验证方式`
- 技术路线是否写出 `总体技术方案 -> 架构分层 -> 任务依赖 -> 测试验证 -> 工程应用`，而不是把研发任务改名
- 应用推广、产学研用合作和开源策略是否分别写清对象、路径、机制、风险边界和反馈闭环，而不是只写“推广应用、协同合作、适时开源”
- 创新点是否具备比较基线，而不是只写“创新、先进、突破”
- 技术成果是否能逐项回指项目研发内容或建设任务，并明确成果实体、交付形态和验收方式
- 技术指标是否写清指标对象、定义、目标值或区间、测试条件和验证方法
- 重章节是否存在“标题很多、段落很短、每段只有一句功能愿望”的浅层展开
- 是否存在“现状分析只罗列国内外资料，没有差距原因和本项目切入点”的问题
- 五个正文 skill 对应章节是否真正执行 `attachment-writing-patterns.md` 的硬门禁：背景类关键段落至少覆盖 `对象 / 现状 / 问题 / 影响 / 项目切入 / 能力或前景` 中 4 项，项目建设方案类关键段落至少覆盖 `目标 / 问题 / 任务 / 路线 / 输出 / 应用或保障` 中 4 项，创新类关键段落至少覆盖 `对象 / 问题 / 动作 / 机制 / 输出 / 验证` 中 4 项，成果和指标至少覆盖各自 5 项要素。若只像摘要、清单或口号，应记为 Must Fix，并要求回到对应正文 skill 重写
- 五个正文 skill 对应章节是否真正消费了 `research-evidence.md`、`research-notes.md`、`facts-ledger.md` 中的证据：背景章至少能看出建设背景所需的政策/趋势/短板依据，建设意义所需的技术/产业/生态依据，以及国内外发展现状及前景所需的国内现状、国外现状、痛点或前景依据；项目建设方案至少能看出目标依据、主要问题依据、研发任务依据、技术路线依据、推广或保障依据；创新点/成果/指标至少能看出比较、机制、输出或验收依据。若台账丰富而正文仍无可追溯证据痕迹，应记为 Must Fix：调研未消费
- 若主控或子代理没有在 `agent-skill-trace.md` 中留下标准 `【专项 skill 留痕回执】`，或台账中“任务单要求加载的专项 skill”和“子代理实际显式加载的专项 skill”不一致，或缺少“显式 skill 调用指令/显式调用指令原文/skill 加载确认原文”任一字段，且当前散件又明显违背对应专项 skill 的编号、句式或论证链规则，应记为 Must Fix：专项 skill 调用留痕缺失 / 疑似绕过专项 skill 写作，并要求按正确任务单重派或重写。仅有 `progress.md` 摘要不算通过。
- 若命中专项内容的任务在 `progress.md`、worker 报告或任务回执中缺少“主控已预载专项 skill”“本节证据清单摘要”“本节写作规则摘要”中的任一项，且当前散件又明显违背对应专项 skill 的编号、句式或论证链规则，应同样记为 Must Fix：主控派发前置步骤缺失 / 疑似未按专项 skill 预处理后派发。
- 若本轮 revise 触及五类正文专项章节，但 `progress.md` 中没有明确记录“本轮 revise 加载的正文 skill”，或未写明对应受影响文件/小节、证据清单摘要、规则摘要，应记为 Must Fix：专项回修加载留痕缺失 / 疑似未按对应正文 skill 回修。
- 一旦命中上述任一流程类 Must Fix，review 或 final review 的结论段不得再出现“条件性允许进入 assemble”“可交付正式稿”“Must Fix 已全部清零”等放行表述；若结论段仍放行，应记为 Must Fix：review 结论自相矛盾。
- 具体来说，只要出现以下任一情况，结论段就必须明确写“暂不可进入 assemble / 暂不可作为正式交付稿”：`agent-prompts/` 缺失或为空、`agent-skill-trace.md` 缺失或为空、正常项目调研未见 research 子代理任务条目、正常多章节写作未见 writing 子代理任务条目、专项 revise 未见 `本轮 revise 加载的正文 skill` 实际记录。
- 若当前章节属于非专项章节，则 review 只需检查子代理是否明确声明“本任务不需要显式加载专项写作 skill”，不应把“未调用专项 skill”本身当成问题。
- 若 review 发现本轮本应派发 research / writing 子代理却未派发，不得以“当前内容看起来像对应 skill 写法”代替流程核验；内容像，只能说明路由大体正确，不能说明流程合规。

## 复核阶段不派子代理

`official-doc-review` 必须由主控代理直接完成，不得并行派发复核子代理。原因很简单：复核依赖全局口径、编号方案、来源门禁和当前正式稿的实时状态，拆给多个子代理后很容易出现互相冲突的结论。

因此：
- 不得生成 `workspace/review/<project-slug>-review-worker-*.md` 一类子代理复核文件
- 不得把结构、事实、深度拆给多个子代理分别判定
- 只能由主控代理读取当前正式稿、台账、脚本输出后，生成唯一正式结论文件 `workspace/review/<project-slug>-review.md` 或 `...-final-review.md`

### 5. 章节配比
- 重章节是否写得足够厚，轻章节是否被写得过长
- 是否出现“应该展开的章节只有骨架，应该简写的章节反而堆叠大段正文”
- 是否存在章节内部层级失衡，例如项目建设方案没有下钻到项目研发任务、技术路线、应用推广、产学研用合作和开源策略

### 6. 高优先表图完备性
- 是否已生成 brief 和 `00-section-plan.md` 中要求的表图
- 是否按章节需要完成高优先表图
- 是否存在“正文已稳定但高优先表图还没补”

### 7. 装配就绪度
- 当前产物是否已经达到 `review -> revise -> assemble` 的装配前状态
- 是否仍有 Must Fix 会阻断正式装配
- 是否仍缺少支撑重章节的关键表图
- 若用户给出了总字数要求，当前成稿是否满足总字数约束

## brief 驱动判定

优先根据 `project-brief.md` 与 `00-section-plan.md` 判断：
- 哪些章节是重章节
- 哪些章节是偏短章节
- 哪些图表是必须项
- 哪些字数要求是硬约束

对于章节级 review，默认检查对象应是“当前章节 + 该章必要表图”，不要因为全书尚未写完就把阶段稿误判为失败。
对于全书总 review 或 final review，才按整本比例、总字数和装配就绪度做总判定。

## 输出格式

输出到：
- `workspace/review/<project-slug>-review.md`

若本次是由 `official-doc-assemble` 触发的装配后 final review，还必须额外输出：
- `workspace/review/<project-slug>-final-review.md`

建议结构：
0. 统计依据与复核时间
1. 已通过项
2. 必修问题（Must Fix）
3. 可优化问题（Should Fix）
4. 章节配比结论
5. 高优先表图完备性结论
6. 建议修复顺序
7. 仍需补充材料清单
8. 当前是否允许进入 assemble

每个问题尽量写明：
- 严重级别
- 对应文件
- 对应章节 / 表图编号
- 修复动作

Must Fix 问题优先覆盖以下高风险类型：
- 调研深度明显不足
- 最新来源不足，仍大量依赖过旧资料
- 正文风格明显不像正式公文
- 未核验具体事实
- 来源等级不达标却被标记为已核验
- 章节比例硬阈值不满足
- 表图缺失或装配位置错误
- 正文仍留有引图引表占位
- review 结论与统计结果自相矛盾

“统计依据与复核时间”至少说明：
- 本次统计基于哪些文件
- 是否包含当前 `formal-draft`
- 正文章节比例是按哪一版文件实时计算得出
- 若是装配后 final review，必须明确写出“本次为装配后 final review”
- 若用户给出了总字数要求，必须写出“目标总字数 / 当前统计总字数 / 偏差”
- 必须明确写出“本次正文字数统计是否排除了表格、Mermaid、引用提示行”
- 必须明确写出“`【待补】` 本次采用的是字段数还是标记数”

若本次是装配后 final review：
- `workspace/review/<project-slug>-final-review.md` 必须明确写出“本次为装配后 final review”
- 不要只复用旧的 `workspace/review/<project-slug>-review.md` 而不生成新的 final review 文件
- 若仍存在 Must Fix，结论只能是：
  - `暂不可作为正式交付稿`
  - 或 `需回退 revise -> assemble`
- 不得写成：
  - `可交付正式稿`
  - `可装配正式稿`
  - `可保留为正式稿`

Must Fix 问题应尽量写成：
- `问题`
- `影响文件`
- `修复动作`
- `完成判定`

## 表达要求

问题必须写到可执行，不要只写抽象判断。

例如：
- 不写：`预算有问题`
- 要写：`第9章预算说明中的总预算为120万元，而表6合计为110万元，建议先统一总预算口径，再修订分项。`

对于“章节配比”问题，不要只写“某章偏短”，要写成类似：
- `某“建设方案”章节只有2个短节，未承接模板要求的任务拆解和路线展开；建议先补齐对应子节，再回填该章所需图示。`

对于“高优先表图缺失”问题，不要只写“缺表”，要写成类似：
- `某资金筹措及投资估算章节正文已写到投资结构，但 workspace/tables/ 中仍未生成资金结构表；应先调用 official-doc-table 生成该表，再统一该章正文引用。`

对于“未核验具体事实”问题，要写成类似：
- `某背景章节写入了具体政策文号和发布日期，但 facts-ledger 中无对应已核验记录；应先补官方来源核验，若无法核验则改写为概括性政策背景表述。`

对于“调研过浅”问题，要写成类似：
- `背景类调研组在 research-sources 中仅保留 3 条来源，其中 2 条早于近 3 年，且 research-notes 仍为空表；在这种情况下直接写背景章节会导致现状判断失真。应先回到 official-doc-research 补足多轮检索和近年主源。`

对于“正文风格不正式”问题，要写成类似：
- `某项目建设方案章节仍大量使用项目符号和加粗小标签，如“技术描述”“技术难点”“作用”，整体更像演示稿备注而不是正式公文正文；应改写为连续自然段，并补齐总体目标、研发任务、技术路线、应用推广和协同机制。`

## 风格检查脚本

复核正文风格时，优先运行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/style_check.ps1 -FilePath <文件路径>
```

重点检查：
- 正文列表行
- 正文加粗小标签
- 机械过渡词
- 空壳强调句
- 目录式过密编号
- 文本污染（中英混杂错词）
- 复杂度或公式残句
- 过强结论词（需人工核验证据）

如果脚本命中以下任一项，默认直接记为 Must Fix：
- 正文列表行
- 正文加粗小标签
- 同一章节中高频出现机械过渡词
- `Result: FAIL`

若脚本输出命中了 `**子任务`，不得解释为“只是标题不是小标签”。在正式稿中，子任务标题必须改为规范标题层级或自然段引导句。

对于“来源等级不达标”问题，要写成类似：
- `某背景章节关于船舶行业大模型的具体断言虽然在 facts-ledger 中标记为已核验，但 research-sources 的支撑来源主要是百科和资讯转载，缺少官方公告或一手材料；应补充 A主源后再保留该断言，否则改写为保守趋势表述。`

对于“正式稿与散件不一致”问题，要写成类似：
- `workspace/assembled/.../formal-draft.md 中的表2仍为大量【待补】版本，但 workspace/tables/.../table-02.md 已包含具体成员名单；应停止在 assemble 阶段重写表2，改为直接纳入最新 table-02.md。`

对于“风格”问题，不要只写“语言略夸张”，要写成类似：
- `第10章出现“无懈可击”“一举破解”等明显宣传化词语，与原模板的可研文风不一致；建议改回工程说明句和判断句。`

对于“装配位置错误”问题，要写成类似：
- `某成果或指标章节正文仍保留“此处引用表4”占位，而表4被统一放在文末附表区；应在对应段落后直接插入该表，并删除占位提示。`

对于“术语污染或残句”问题，要写成类似：
- `某现状章节“C广告matic Electrical”为术语污染，应改为“Cadmatic Electrical”，并统一全文该厂商名称。`
- `某困难或风险章节出现“规则推理层在的时间复杂度内”“复杂度降至约，其中”等残句，应改写为完整工程表述或删除不完整公式描述。`

## 交付后动作

- 更新 `workspace/plan/progress.md`
- 若发现大量共性问题，回写 `workspace/plan/facts-ledger.md`、`workspace/plan/research-evidence.md` 或 `workspace/plan/source-materials.md`
- 若存在可修复问题，默认下一步应调用 `official-doc-revise`
- 若判断“不可装配”，必须明确写出阻断装配的 Must Fix 列表
- 若本次是装配后 final review，必须明确写出：当前 `formal-draft` 是否仍可保留为正式稿，还是需要回退到 `revise -> assemble`
