# official-doc-project-background

## 定位

这是“项目背景及必要性”章节的规则库，不是自由发挥的背景素材库。

当前背景章节的主要参照模板为 `产业链项目指南/完整科研项目模板.docx` 的“一、项目背景及必要性”。当用户要求撰写完整背景章，且未另行给定更强模板时，默认采用该模板的章节结构和写法：

```md
一、项目背景及必要性
（一）建设背景
（二）建设意义
1. 技术层面
2. 产业层面
3. 生态层面
（三）国内外发展现状及前景
1. 国内发展现状
2. 国外发展与应用现状
3. 痛点分析
4. 发展前景
```

如果用户只要求其中一部分，只写用户要求的部分，并沿用本 skill 中对应小节的写法。不得因为本 skill 覆盖范围较宽，就自行追加 `00-section-plan.md` 未登记的小节。

本 skill 覆盖的写作对象包括但不限于：
- 建设背景
- 建设意义
- 技术层面、产业层面、生态层面意义
- 国内发展现状
- 国外发展与应用现状
- 国内外发展现状及前景
- 痛点分析、问题分析
- 发展前景、发展趋势
- 攻关必要性、建设必要性、立项必要性
- 产业链安全、自主可控

## 先做什么

开始写之前先检查调研门禁是否完成，而不是从零开始搜索。

必须先读取：
- `workspace/plan/<project-slug>/research-plan.md`
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/research-notes.md`
- `workspace/plan/<project-slug>/facts-ledger.md`
- `workspace/outputs/<project-slug>/00-section-plan.md`

如果背景类调研组仍存在以下任一情况，应先回到 `official-doc-research` 补调研，再回来写：
- 只有零散 2 到 3 条来源
- 多数来源早于近 3 年，且没有标记为 `历史基线` / `基础规范`
- 缺少 `A主源`
- `research-evidence.md` 没有为有效来源建立证据卡，或证据卡只记录标题和链接
- 还没有形成可写正文的 `research-notes.md`
- 无法分别支撑“政策/行业背景、技术或产业现状、国内外比较、痛点或前景判断”

本 skill 不直接联网搜索。只有在背景类台账已经基本齐全、但某个小问题仍缺支撑时，才回到 `official-doc-research` 补充调研，由 `official-doc-research` 使用既定检索工具完成检索、抓取和落账。

补充调研结果必须登记到：
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

登记时至少补齐以下字段：
- `research-sources.md`：来源标题、来源类型、来源等级、发布机构/作者、发布时间、链接、支撑章节、是否可直接支撑正文断言
- `research-evidence.md`：同编号证据卡、抓取状态、结构化证据点、短摘录、使用边界和仍需补证的问题
- `facts-ledger.md`：事实表述、对应章节、来源编号、主源编号、辅源编号、事实状态、备注

事实状态只允许使用：
- `已核验`
- `待核验`
- `仅作趋势参考`

来源等级只允许使用：
- `A主源`
- `B辅源`
- `C参考`

## 事实门禁

政策、文号、发布日期、主管部门表述，优先使用官方来源核验。

市场规模、占比、排名、订单量、装机量、市场份额等数字，优先使用主管部门、行业协会、研究机构报告或企业原始披露核验。

具体平台、模型、产品能力、市场格局和应用案例，优先使用官方产品页、论文原文、项目公告、权威报告或一手技术资料核验。

背景类事实的默认来源等级口径：
- `A主源`：国务院/部委/地方主管部门/标准原文/船级社/行业协会权威统计/论文原文/企业官方公告或产品页
- `B辅源`：主流行业媒体、研究机构报告、行业协会公开材料、权威会议材料
- `C参考`：百科、博客、论坛、自媒体、聚合资讯、转载稿、问答社区

以下来源不得单独支撑 `已核验`：
- 百度百科等百科词条
- 知乎、CSDN 等博客/问答社区
- 资讯转载站、聚合新闻站
- 单篇营销稿、宣传稿

如果某条背景事实只有 `C参考` 来源：
- 不能标记为 `已核验`
- 不能写成具体政策文号、具体发布日期、精确市场数字或明确产品能力断言
- 只能作为检索线索，继续追到 `A主源` 或 `B辅源`

如果没有完成核验：
- 不要写具体文号
- 不要写具体发布日期
- 不要写精确市场数字
- 不要把单一宣传材料中的表述当作确定事实

可以改写为更稳健的概括性表达，例如：
- `近年国家层面持续推动高端装备制造、工业软件和制造业数字化转型。`
- `国外头部工业软件企业已在产品数据管理、协同设计和生命周期管理方面形成较成熟的产品体系。`

不要写成：
- `2025年8月，国务院印发……`
- `2025年全球工业AI软件市场规模已达……`

除非这些表述已经在 `facts-ledger.md` 中标记为 `已核验`。

本章的核心论证顺序是：

```md
建设背景：政策/战略/产业形势 -> 外部环境或行业压力 -> 技术趋势与现实短板 -> 本项目切入

国内外发展现状及前景：国内现状 -> 国外发展与应用现状 -> 痛点分析 -> 发展前景
```

背景类关键段落必须至少覆盖 `对象 / 现状 / 问题 / 影响 / 项目切入 / 能力或前景` 中 4 项。若写完后只剩宏观口号或意义判断，必须先回修再交付。

## 新模板提炼出的章节写法

### 1. 建设背景

`建设背景` 是本章的事实和问题底座，不能只写“政策支持、行业需要”。参照模板中的成熟写法，建议按 4 到 6 个自然段展开：

1. 第一段写国家战略、产业地位和宏观环境。可从高端装备、制造强国、海洋强国、工业软件、自主可控、数字化转型等已核验依据切入，再交代国际竞争、市场波动、技术封锁、供应链风险等外部压力。
2. 第二段写本项目对象所在的核心短板。要把“为什么需要建设”落到具体对象，例如核心工业软件、产品数据管理平台、设计制造数据链、仿真分析工具、知识库、规则库、协同平台等，说明现有依赖、国产替代压力、业务复杂度和已有能力不足。
3. 第三段写技术演进和行业转型。围绕工业互联网、云计算、大数据、人工智能、数字孪生、模型驱动、数据要素等趋势，说明这些趋势如何改变本领域研发、设计、制造、管理或服务模式。
4. 第四段写本领域现实状态和建设窗口。说明行业已有进展，也要指出不平衡、不充分、缺少统一数据底座、缺少工程闭环、基础数据积累不足、集成化水平较低等现实差距。
5. 结尾段必须收束到本项目主题。用“亟须”“有必要”“本项目拟围绕”等正式表述，把建设背景自然导向后续项目建设方案或建设目标。

如果确有必要介绍某类软件、平台或技术对象的功能范围，可以采用模板中的“总述段 + 编号功能项”写法，但功能项必须服务于背景论证，不得把整节写成产品说明书。功能项一般控制在 3 到 6 项，每项说明“功能对象 + 业务作用 + 为什么与本项目有关”。

建设背景常用句式：
- `以...为代表的...是...的重要组成部分，也是...的重要支撑。`
- `在...新阶段，随着...在...领域的渗透，...已成为提升...水平的关键因素。`
- `然而，当前...仍存在...问题，特别是在...方面尚难满足...需求。`
- `面对...外部环境和...现实需求，国内亟须...。`
- `因此，本项目拟围绕...开展研究和建设，解决...关键问题，支撑...。`

### 2. 建设意义

`建设意义` 不是一段泛泛价值判断。完整背景章默认先写 1 个总述段，再按 `技术层面 / 产业层面 / 生态层面` 三个三级标题展开。

总述段写法：
- 先交代项目对象与国家战略、产业转型或行业能力建设之间的关系。
- 再点出国外产品依赖、产业链自主可控、工程应用需求或行业升级压力。
- 最后用一句话引出三层意义，不要直接堆“意义重大、影响深远”。

`技术层面` 应写：
- 项目对象在技术体系中的位置，例如统一模型、单一数据源、全生命周期数据管理、跨系统集成、流程协同、规则化表达、安全保障等。
- 将解决哪些技术问题，例如数据海量高维、多源异构、时变复杂、接口不统一、流程难追溯、知识难复用、设计制造数据断链等。
- 将形成什么技术能力，例如自主技术体系、统一数据模型、数据链路贯通、工作流管理、状态管控、定制化开发、安全防护等。
- 语言上要以“通过建立...实现...”“通过对...进行...支撑...”为主，少写空泛的“提高技术水平”。

`产业层面` 应写：
- 项目如何回应工业软件“卡脖子”、国外垄断、高授权费用、供应链安全、企业成本和市场竞争问题。
- 项目如何服务产业数字化转型，例如设计制造全程数字化、产品开发周期缩短、市场响应能力提升、上下游协同效率提高。
- 项目如何与 ERP、MES、CAD、CAE、CAPP、CAM 等既有工业软件协同，推动产业链上下游整体优化。
- 不能只写“带动产业发展”，必须说明带动路径。

`生态层面` 应写：
- 项目如何构建健康工业软件生态，包括国产软件协同、人才培养、技术积累、标杆应用、开源或接口开放、产学研用协同等。
- 项目如何形成可扩展的应用生态，例如与国产软硬件、设计工具、仿真工具、制造系统和运维系统兼容适配。
- 项目如何由单个系统建设外溢为行业知识、开发能力和工程方法的沉淀。
- 结尾可以保留价值收束，但必须回到具体生态对象和动作。

建设意义常用句式：
- `在技术层面，...是实现...的重要支撑。项目将围绕...特点，通过...建立...，支撑...。`
- `在产业层面，...有助于破解...问题，降低...依赖，提升...能力。`
- `在生态层面，...将带动...协同，沉淀...能力，推动形成...生态。`

### 3. 国内外发展现状及前景

`国内外发展现状及前景` 是背景章中最容易写浅的部分。参照模板，完整小节默认分为 `国内发展现状 / 国外发展与应用现状 / 痛点分析 / 发展前景` 四个三级标题。

#### 国内发展现状

国内现状应按以下顺序写：
- 先写行业或领域总体发展基础、规模地位、政策支持和已有建设基础。
- 再写国内代表性产品、单位、平台、项目或技术路线，说明已经具备哪些能力。
- 随后写国产能力的现实限制，例如功能深度、工程应用成熟度、用户体验、复杂场景适配、二次开发、与既有系统集成、行业份额等。
- 结尾必须引出“为什么仍需要本项目”，不能停在成果罗列。

不要把国内现状写成宣传稿。模板中的有效写法是“先肯定基础，再指出依赖和短板”，而不是简单写“国内快速发展、仍有差距”。

#### 国外发展与应用现状

国外现状应按以下顺序写：
- 先列出代表性产品、企业、研究机构或技术路线。
- 再分别说明其核心能力、典型应用场景、工程成熟度和生态优势。
- 然后概括国际发展趋势，例如云化、平台化、模块化、协同化、AI 集成、供应链协同、全球团队协作等。
- 最后与国内现状形成比较，说明差距不只是“先进/落后”，而是业务功能完整性、定制化能力、工程应用深度、生态集成能力或用户体验上的差距。

国外现状不要写成百科条目堆砌。每个代表性对象只写与本项目有关的能力和局限。

#### 痛点分析

痛点分析通常先写 1 个总述段，再按若干痛点展开。若 `00-section-plan.md` 允许使用四级标题，可采用 `数据有效性和关联性痛点`、`数据格式不一痛点`、`数据应用痛点`、`异构系统集成痛点` 等命名；若计划不允许四级标题，就用连续自然段写。

每个痛点必须回答四件事：
- 当前问题表现是什么。
- 在本项目业务场景中如何发生，最好有具体系统、数据、流程或用户对象。
- 对效率、质量、协同、安全、合规、成本、交付或自主可控造成什么影响。
- 为什么需要本项目的统一平台、数据底座、规则体系、工具链或工程验证来解决。

痛点段落可以比普通段落更长，但不能只写“效率低、成本高、风险大”。每个痛点都应能转化为后续项目研发内容或建设任务。

#### 发展前景

发展前景不是泛泛展望。参照模板，可按 `前景一 / 前景二 / 前景三 / 前景四` 等三级或四级标题展开，但必须先检查 `00-section-plan.md` 是否允许该层级。

前景写法应围绕本项目未来应用方向展开，例如：
- 面向全产业链的数据一体化管理。
- 支持产品定制化的多目标、多学科协同优化。
- 整合一体化数字底座平台。
- 引入人工智能、数字孪生或智能决策能力提升产业运行效率。
- 支撑国产工业软件生态、开源开放接口和产业链协同。

每个前景段至少包含：
- 趋势判断。
- 本项目对象如何承接该趋势。
- 对行业流程、产业链协同、数据资产或生态能力的具体影响。

不要写成“未来前景广阔、市场空间巨大、意义重大”。必须说明“广阔在哪里、由什么能力承接、如何作用于产业链或业务流程”。

## 写作步骤

### 第一步：识别本次要写的结构

从用户 brief 和 `00-section-plan.md` 判断：
- 是否写完整 `项目背景及必要性`
- 是否只写 `建设背景`
- 是否只写 `建设意义`
- 是否只写 `国内外发展现状及前景`
- 是否只写 `痛点分析` 或 `发展前景`
- 是否需要合并到用户已有章节标题下

如果写完整背景章，默认使用“建设背景、建设意义、国内外发展现状及前景”三节结构。若用户或模板已经给出不同结构，优先服从 `00-section-plan.md`，但写法仍按本 skill 的对应规则执行。

### 第二步：抽取证据并分配到小节

写本章前，必须先做“证据抽取”。每个二级节至少从 `research-evidence.md`、`research-notes.md`、`facts-ledger.md` 中整理出 3 个可直接消费的证据点。

完整背景章的最低证据分配：
- `建设背景`：至少包含政策/官方依据、行业或技术趋势依据、项目对象现实短板依据。
- `建设意义`：至少分别支撑技术层面、产业层面、生态层面中的 2 个以上判断。
- `国内外发展现状及前景`：至少包含国内现状依据、国外现状或代表性产品依据、痛点或差距依据、发展前景或趋势依据。

正文不得直接写出 `F-001`、`BG-S03` 这类内部编号，但段落中的政策要求、场景问题、国内外差距、痛点判断和发展前景必须能回溯到这些证据点。

若一个二级节写完后只有宏观形势和价值判断，看不出具体政策、场景问题、国内外比较、影响后果和项目切入，视为“调研未消费”，必须退回重写或补调研。

### 第三步：按新模板文风组织正文

语言要求：
- 正式
- 克制
- 工程化
- 可研/立项口径
- 允许较长自然段，但每段必须有事实、问题或项目切入

参照模板，背景章常见段落长度可比一般正文略长，通常为 180 至 420 字。政策背景、国内外现状和痛点分析可以使用较长复合句，但不能形成无断点的材料堆砌。

推荐使用：
- `当前`
- `在...新阶段`
- `随着...`
- `然而`
- `特别是`
- `由于...`
- `面对...`
- `亟须`
- `通过...`
- `有助于...`

避免：
- 宣传稿口吻
- 夸张修辞
- 无来源的绝对化判断
- 无核验的具体数字、文号、日期、排名、市场份额
- 连续使用 `首先、其次、最后、此外、总之`
- 把意义段写成纯价值词堆叠

## 与后续章节的衔接要求

背景类内容必须为后续章节埋好逻辑前提：
- `建设背景` 为项目总体目标和项目建设方案提供外部前提。
- `建设意义` 为成果价值、产业链安全、自主可控和生态建设提供论证基础。
- `国内外发展现状及前景` 为创新点提供比较基线和趋势依据。
- `痛点分析` 为项目研发内容和技术路线提供问题来源。
- `发展前景` 为预期成果、推广应用和生态建设提供趋势依据。

## 输出与流程要求

本 skill 默认假定以下前置工作已经完成：
- `official-doc-core` 已执行
- `using-official-docs` 已完成工作区初始化
- 工作区目录已创建
- `project-brief.md`、`research-sources.md`、`research-evidence.md`、`facts-ledger.md`、`progress.md` 已存在

因此，本 skill 不负责：
- 创建工作区
- 初始化 plan 文件
- 补建 `project-brief.md`
- 补建 `00-section-plan.md`

本 skill 负责的输出应写入 `workspace/outputs/<project-slug>/` 对应章节文件。
不要把完整背景正文直接输出到终端。
在终端只简要报告：
- 已写入哪个章节文件
- 已消费哪些证据类型
- 主入口下一步应继续补什么

若本章由子代理执行（如 Claude Code 并行任务），该子代理的首个动作必须是显式加载 `official-doc-project-background`，并在开始任何读写前先回报“已显式加载：official-doc-project-background”；未完成这一步，不得读取台账或写稿。

子代理任务单应复用 `.claude/skills/official-doc-core/references/subagent-task-card-template.md`，并写明本章独占可编辑文件。

在正常多章节写作中，主控调用本 skill 后，不应直接把整章正文写完再伪装成“已遵守本 skill”。正确做法是：主控先显式加载本 skill，依据本 skill 生成合格子代理任务单，再派发对应 writing 子代理。若最终 `agent-prompts/*.md` 中看不到本 skill 的调用要求、主控预载记录、任务级留痕要求和提交回执要求，则说明本 skill 并未真正参与子代理执行链。

- 是否需要 `official-doc-table`
- 是否需要 `official-doc-figure`

## 触发样例

以下情形都必须触发本 skill：

### 样例 1

用户写：
- `第一章项目背景及必要性，包括建设背景、建设意义、国内外发展现状及前景`

应触发原因：
- 完整命中新模板背景章结构。

### 样例 2

用户写：
- `建设意义需要从技术层面、产业层面、生态层面展开`

应触发原因：
- 三个层面属于本 skill 的建设意义规则。

### 样例 3

用户写：
- `某一章项目现状和发展趋势，需要写国内发展现状、国外发展与应用现状、痛点分析和发展前景`

应触发原因：
- 四个内容均属于 `国内外发展现状及前景`。

### 样例 4

用户写：
- `第二章只写痛点分析，不单列建设背景`

应触发原因：
- 即使没有出现 `项目背景` 四个字，只要出现 `痛点分析`，仍然必须调用本 skill。

## 输出检查

写完后确认：
- 是否先完成调研门禁并抽取证据
- 是否服从 `00-section-plan.md` 的正式标题和编号层级
- 完整背景章是否采用 `建设背景 / 建设意义 / 国内外发展现状及前景` 三节结构
- `建设意义` 是否按 `技术层面 / 产业层面 / 生态层面` 展开
- `国内外发展现状及前景` 是否按 `国内发展现状 / 国外发展与应用现状 / 痛点分析 / 发展前景` 展开
- 是否写清国内外差距和痛点原因，而非资料堆砌
- 是否给后续项目建设方案、创新点、成果和指标留出逻辑接口
- 是否避免了未经核验的具体数字、文号、日期、排名和市场份额
- 是否没有把整章写成价值词堆叠

## 不要做的事

- 不要只有宏观口号，没有行业和场景落点
- 不要把 `建设意义` 写成单段价值判断
- 不要把 `技术层面 / 产业层面 / 生态层面` 改成随意的战略、经济、社会三段式，除非用户明确要求
- 不要把 `国内外发展现状及前景` 写成简单的“国外先进、国内不足”
- 不要把痛点写成抽象抱怨
- 不要把发展前景写成空泛展望
- 不要无依据写“国际领先”“国内首创”“填补空白”
- 不要新增 `00-section-plan.md` 未登记的二级节或用连续流水号标题堆篇幅
