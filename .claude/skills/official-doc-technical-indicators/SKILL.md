# official-doc-technical-indicators

## 定位

这是“技术指标/量化目标类内容”的规则库。

它回答的是：
- 项目最终要达到什么量化水平
- 这些指标如何分类
- 指标如何对应研究内容与成果
- 指标如何具备验收可操作性

它不是成果清单，也不是背景分析。

本 skill 默认假定：
- `using-official-docs` 已完成工作区和 plan 初始化
- `official-doc-core` 已完成公共校验

本 skill 不负责：
- 创建工作区
- 创建 plan 文件
- 决定是否直接结束整轮任务

本 skill 的正文结果必须写入 `workspace/outputs/<project-slug>/` 对应章节文件。
不要把完整指标正文直接输出到终端。

本 skill 写完后，必须把控制权交还给 `using-official-docs` 继续推进下游流程。

指标正文应先有一段总述，再按类别展开，不要把所有指标直接平铺成零散短句；若用户未要求表格，正文仍应写成正式段落。

正式正文写法必须同时遵守：
- `official-doc-core` 中的公文文风规则
- [../official-doc-core/references/formal-doc-style.md](../official-doc-core/references/formal-doc-style.md)
- [../official-doc-core/references/attachment-writing-patterns.md](../official-doc-core/references/attachment-writing-patterns.md) 中的编号方案和指标写法
- [../official-doc-core/references/depth-writing-rules.md](../official-doc-core/references/depth-writing-rules.md)
- [../official-doc-core/references/depth-writing-examples.md](../official-doc-core/references/depth-writing-examples.md) 中的技术指标示例

`attachment-writing-patterns.md` 是硬门禁，也是必须使用的句式模板，不是可选参考。指标和成效章节必须套用该文件中“技术指标写法”的常用写法和句式骨架，尤其是 `在...输入信息准确、完整的前提下，实现...，...达到...`、`在...测试场景下，系统能够...，...准确率/完整率/通过率达到...`、`通过...测试、抽样复核或示范应用，验证...` 这类句式。每项指标至少使用一处“输入条件 -> 能力动作 -> 目标值 -> 验证口径”的句式链，并覆盖 `指标对象 / 指标定义 / 目标值或区间 / 测试条件 / 验证方法 / 对应成果` 中至少 5 项。只有百分比、提升方向或“达到先进水平”的条目不能作为正式技术指标。

写指标前，必须先从 `research-evidence.md`、`research-notes.md`、`facts-ledger.md` 中为每项指标抽出至少 2 个证据点：1 个用于目标值/区间或校准依据，1 个用于测试条件、适用场景或验证方法。若只有一个孤立百分比而没有条件和口径，不得写入正式指标。

指标正文不得裸露内部来源编号，但每个目标值、区间、测试条件和验证方式都必须能回溯到证据卡或事实台账。若某条指标只能写成“效率提升XX%”“准确率达到XX%”而无法说明在哪种样例、环境、口径下测试，视为未消费调研，应降级为待校准指标或退回补证。

指标文字也要收口。不要用“达到先进水平、显著提升、全面优化”替代测试口径；优先写对象、定义、条件、目标值和验证方式。若一段以价值判断居多，而测试条件和验收方式只有一句甚至没有，说明文风过泛，必须重写。

扩写指标或成效章节时，不得新增 `00-section-plan.md` 未登记的二级节，也不得用连续流水号标题堆篇幅。需要加深时，应补充指标来源、测试场景、验收方法、对应成果和成效解释；指标表内的序号不得反向变成正文标题编号。

若本章由子代理执行（如 Claude Code 并行任务），该子代理的首个动作必须是显式加载 `official-doc-technical-indicators`，并在开始任何读写前先回报“已显式加载：official-doc-technical-indicators”；未完成这一步，不得读取台账或写稿。该子代理也必须完整遵守本 skill 与 `attachment-writing-patterns.md` 的硬规则；每项指标都要具备测试条件和验证口径，不得只保留百分比目标。
若运行环境支持 skill 显式调用，子代理必须在开始读取台账和写作前显式加载 `official-doc-technical-indicators`；仅手动读取本文件路径不算完成本 skill 的调用。
子代理任务单应复用 `.claude/skills/official-doc-core/references/subagent-task-card-template.md`，并写明本章独占可编辑文件。
在正常多章节写作中，主控调用本 skill 后，不应直接把指标章节写完再假定“已经按本 skill 执行”。正确做法是：主控先显式加载本 skill，依据本 skill 提炼指标对象、目标值、测试条件和验证方式，再派发对应 writing 子代理。若最终 `agent-prompts/*.md` 中看不到本 skill 的调用要求、主控预载记录、任务级留痕要求和提交回执要求，则说明本 skill 并未真正进入子代理执行链。

## 先做什么

开始写之前先检查指标类调研门禁是否完成。

必须先读取：
- `workspace/plan/<project-slug>/research-plan.md`
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/research-notes.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

如果指标类调研仍存在以下任一情况，应先回到 `official-doc-research` 补调研：
- 指标区间没有近 3 年基线
- 缺少验收口径或测试场景参照
- 指标数字看起来像拍脑袋，没有主源或辅源支撑
- `research-evidence.md` 没有记录指标目标值、测试条件或验收口径的证据点

本 skill 不直接联网搜索。只有在指标框架已清楚、但个别指标仍需补证时，才回到 `official-doc-research` 补充调研，由 `official-doc-research` 使用本地 MCP 完成检索、抓取和落账。补查重点包括：
- 同类项目的常见指标口径
- 公开可见的行业基准、验收口径、性能口径
- 与本主题相关的准确率、效率、容量、兼容性、应用效果等合理区间

补充调研结果必须记录到：
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

补查完成后，本 skill 优先读取 `research-evidence.md` 的证据卡，再读取调研台账和事实台账，不自行改用其他搜索工具。

## 写作深度门槛

指标类正文不能只列百分比或空泛目标。每项指标必须具备可验收链条：
- `指标对象`：对应哪个系统、模块、模型、数据资源、流程或应用场景。
- `指标定义`：指标衡量的具体含义是什么，避免名称相同但口径不同。
- `目标值或区间`：给出可测试数值、范围或阶段性目标；无依据时用保守区间或 `【待补】`。
- `测试条件`：在什么数据规模、船型范围、样例数量、运行环境或业务场景下测试。
- `验证方法`：通过功能测试、对比测试、抽样复核、专家评审、系统日志或应用示范中的哪种方式验证。
- `依据来源`：目标值来自行业基线、同类项目、标准规范、用户要求还是项目阶段测算。

若无法说明测试条件和验证方法，该项不能作为正式验收指标，只能作为预期方向或待校准指标。

## 从附件提炼出的共性写法

写指标前必须读取 `00-section-plan.md` 中的全文编号方案。指标项可以在表格内用独立序号，但正文标题层级必须服从全文方案；不得把指标条目编号误当成章节标题编号。

指标内容只能写入 `00-section-plan.md` 已登记的指标或成效小节。若计划只允许 `（二）主要技术指标` 或 `（二）预期成效`，不得追加 `项目实施风险与应对`、`可持续发展展望`、`人才队伍建设计划`、`国际推广前景` 等计划外二级节。字数不足时，应补充每项指标的输入条件、测试场景、目标值、验证方法和对应成果。

### 1. 指标通常是分层分类写，不是一个混合清单

附件中常见的指标分组包括：
- 平台/系统研发指标
- 性能指标
- 数据与模型库指标
- 算法指标
- 集成与互操作指标
- 自主可控指标
- 应用验证与效能指标
- 产业化或推广指标

### 2. 指标必须可量化、可验证

高质量指标通常能明确：
- 衡量对象
- 目标值或目标区间
- 测试场景或适用范围
- 验收方式

### 3. 指标要从研究内容和成果中长出来

例如：
- 研究内容做了分类模型，就可以有准确率、召回率、覆盖率
- 做了自动化图像输出，就可以有输出成功率、批处理效率
- 做了报告生成，就可以有模板填充准确率、合规问答准确率、报告生成时间

### 4. 附件式指标口径

附件中的指标常写成“输入条件 + 能力动作 + 目标值 + 验证口径”。不要只写 `准确率达到...`，必须补充适用条件和测试对象。

可直接套用的句式：

- `在...输入信息准确、完整的前提下，实现...，...达到...。`
- `在...测试场景下，系统能够...，...准确率/完整率/通过率达到...。`
- `通过...测试、抽样复核或示范应用，验证...。`

若用户允许表格，指标表字段建议为：

`指标类别 | 指标名称 | 目标值/区间 | 测试条件 | 验证方式 | 对应成果`

## 写作原则

### 原则 1：按对象分组

优先按以下对象写指标：
- 系统平台
- 算法模型
- 数据资源
- 工具链集成
- 应用效果
- 安全与兼容

### 原则 2：每条指标至少包含三个元素

至少写清：
- 指标名称
- 目标值或区间
- 适用对象/测量对象

更理想时再补：
- 测量方法
- 验证场景
- 是否为验收门槛

### 原则 3：没有依据时不要伪精确

如果公开资料无法支撑精确数字，优先采用：
- 合理区间
- 保守表述
- `待项目验证阶段进一步校准`
- `具体数值以测试测算为准`

不要硬写极其精确但没有依据的数值。

## 常见指标分类

根据附件内容，可优先从以下类别中选用：

### A. 平台与系统指标

例如：
- 并发用户数
- 响应时间
- 稳定运行时间
- 模块数量
- 处理吞吐能力

### B. 算法与模型指标

例如：
- 识别准确率
- 分类准确率
- 提取完整率
- 检出率
- 误报率
- 生成通过率

### C. 数据与知识资源指标

例如：
- 样本数量
- 规则条目数
- 语料条目数
- 数据集类别数
- 覆盖场景数

### D. 集成与互操作指标

例如：
- 对接系统数量
- 支持格式数量
- 数据双向打通能力
- 兼容国产软硬件数量

### E. 应用与效能指标

例如：
- 效率提升比例
- 错误率降低比例
- 交付时间缩短比例
- 自动生成成功率
- 审核问题检出率

### F. 自主可控与推广指标

例如：
- 自主可控率
- 试点船型数量
- 推广单位数量
- 部署数量

## 推荐写法

每类指标可以先有一句总述，再列具体指标项。

每个指标项建议采用以下表达：
- `指标名称：目标值；测量对象/场景；说明。`
- `在...条件下，...指标达到...，通过...方式验证。`

如果用户允许表格，可整理成：
- 指标类别
- 指标名称
- 目标值
- 测量方式
- 对应研究内容

## 如何从研究内容反推指标

写指标前，先做一轮映射：
- 研究内容做了什么
- 形成了什么成果
- 这些成果靠什么指标证明有效

例如：
- `目标节点分类模型` -> `分类准确率、覆盖节点类型数、批量处理速度`
- `应力云图自动输出工具` -> `自动输出成功率、单模型处理时长`
- `报告生成智能体` -> `模板填充准确率、报告生成时间、合规问答准确率`

## 与成果、成效的边界

### 指标不等于成果

`知识图谱` 是成果，`知识条目数` 是指标。

### 指标也不完全等于成效

`效率提升 50%` 可以是应用效能指标，但如果用户要写“预期成效”，还要把指标转成业务价值说明。

## 触发样例

以下情形都必须触发本 skill：

### 样例 1

用户写：
- `某一章预期目标，需要写预期成效，尽量量化`

应触发原因：
- `预期成效`、`量化` 都属于指标类规则

### 样例 2

用户写：
- `请补主要技术指标，包括准确率、效率提升和数据规模`

应触发原因：
- `主要技术指标` 直接命中，后面的准确率、效率、规模都是典型指标项

### 样例 3

用户写：
- `单独列一个考核指标体系，分平台能力、算法能力、应用效果三类`

应触发原因：
- `考核指标体系` 是本 skill 的典型写法场景

### 样例 4

用户写：
- `请给出验收指标和应用指标，最好可测试`

应触发原因：
- `验收指标`、`应用指标` 都属于本 skill 的直接命中项

## 输出检查

写完后确认：
- 是否先做了搜索并登记来源
- 指标是否按对象分组，而非杂乱堆叠
- 每条指标是否可验证
- 是否和研究内容、成果建立了映射
- 是否避免了无依据的伪精确数字

## 不要做的事

- 不要把成果名直接当指标
- 不要只写“达到先进水平”
- 不要所有指标都只有一个百分比
- 不要无依据写不现实的高目标
- 不要遗漏应用效能和验收可操作性
