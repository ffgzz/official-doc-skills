# official-doc-research-content

## 定位

这是“项目建设方案”章的写作规则库，不再是旧版泛化内容规则库。

当前“项目建设方案”章的默认规则结构如下。当用户要求撰写完整项目建设方案章，且 `00-section-plan.md` 未另行给定更强章节结构时，默认采用该结构和写法：

```md
四、项目建设方案
（一）总体目标
1. 建设目标
2. 项目解决的主要问题
3. 项目研发内容
4. 预期成果
5. 产业链供应链韧性及安全保障
（二）项目建设方案
1. 技术路线
2. 应用推广方案
3. 产学研用合作方式
4. 攻关成果开源策略
```

如果用户只要求其中一部分，只写用户要求的部分，并沿用本 skill 中对应小节的写法。不得因为本 skill 覆盖完整章，就自行追加 `00-section-plan.md` 未登记的小节。

本 skill 覆盖的写作对象包括但不限于：
- 项目建设方案
- 总体目标、建设目标
- 项目解决的主要问题
- 项目研发内容、建设任务、任务分解
- 预期成果
- 产业链供应链韧性及安全保障
- 技术路线、总体技术方案、技术路线合理性、可行性、自主性
- 应用推广方案、应用推广路径、预期应用推广效果
- 产学研用合作方式、联合体合作模式、任务分配
- 攻关成果开源策略、开源方向、开源协作模式、开源风险防范、开源闭环管理

## 先做什么

开始写之前先检查建设方案类调研门禁是否完成，而不是直接编任务。

必须先读取：
- `workspace/plan/<project-slug>/source-materials.md`
- `workspace/plan/<project-slug>/project-brief.md`
- `workspace/plan/<project-slug>/research-plan.md`
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/research-notes.md`
- `workspace/plan/<project-slug>/facts-ledger.md`
- `workspace/outputs/<project-slug>/00-section-plan.md`

如果建设方案类调研仍存在以下任一情况，应先回到 `official-doc-research` 补调研：
- 项目目标、建设规模、应用范围或技术指标没有依据
- “项目解决的主要问题”无法回指背景章中的痛点和现状差距
- 研发内容拆分只有任务名，缺少任务边界、输入输出和对应成果
- 技术路线缺少总体架构、模块依赖、测试验证或工程应用链路
- 应用推广、产学研用、开源策略没有单位、路径、机制或风险防范依据
- `research-evidence.md` 缺少同类项目建设方案、技术路线或任务分解的证据卡
- `research-notes.md` 尚未沉淀出“目标 -> 问题 -> 任务 -> 技术路线 -> 成果/推广”的摘记

本 skill 不直接联网搜索。只有在建设方案框架已建立、但某个目标值、任务边界、推广路径或开源策略仍缺支撑时，才回到 `official-doc-research` 补充调研。

用户提示词提供的项目资料必须先从 `source-materials.md` 消费。项目目标、任务、子课题、单位分工、团队基础、经费、进度、预期成果、指标和风险，若已在 `source-materials.md` 或 `project-brief.md` 中给出，应作为项目事实使用；外部调研只用于补充公开政策、行业趋势、国内外公开案例和技术路线合理性，不得覆盖用户给定事实。若用户明确禁止参照外部模板文件，本 skill 不得要求打开、读取或模仿任何模板文件。

补充调研结果必须记录到：
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

正式正文写法必须同时遵守：
- `official-doc-core` 中的公文文风规则
- [../official-doc-core/references/formal-doc-style.md](../official-doc-core/references/formal-doc-style.md)
- [../official-doc-core/references/attachment-writing-patterns.md](../official-doc-core/references/attachment-writing-patterns.md) 中的编号方案和项目建设方案写法
- [../official-doc-core/references/depth-writing-rules.md](../official-doc-core/references/depth-writing-rules.md)
- [../official-doc-core/references/depth-writing-examples.md](../official-doc-core/references/depth-writing-examples.md) 中的项目建设方案示例

`attachment-writing-patterns.md` 是硬门禁，不是可选参考。项目建设方案章必须写成“总体目标 -> 项目建设方案”的递进结构，而不是只列任务、技术名词和功能愿望。

本章的核心论证顺序是：

```md
总体目标：建设目标 -> 项目解决的主要问题 -> 项目研发内容 -> 预期成果 -> 产业链供应链韧性及安全保障
项目建设方案：技术路线 -> 应用推广方案 -> 产学研用合作方式 -> 攻关成果开源策略
```

建设方案类关键段落必须至少覆盖 `目标 / 问题 / 任务 / 路线 / 输出 / 应用或保障` 中 4 项。若写完后只剩任务名、功能愿望和价值判断，必须先回修再交付。

## 深度门禁

完整 `项目建设方案` 章不能写成“方案摘要”。默认按项目复杂度控制篇幅：
- 单一工具或单一系统项目：不少于 15000 中文字符。
- 三到五项子课题或多单位协作项目：建议 24000 到 36000 中文字符。
- 大型平台、联合体或产业链项目：建议 36000 中文字符以上。

若用户提示词或 `00-section-plan.md` 给出本章目标字数、建议区间或最低字数，必须优先执行用户要求；上述默认篇幅只在用户未给出字数时使用。凡 prompt 写明“建议X至Y、最低不得少于Z”，Z 是硬下限，X-Y 是目标区间，不得按子课题数量、单位数量或“简版材料”自行降低。技术路线和任务闭环不足时，不能用概括性语言压缩。

本章写作完成后必须自检字数：运行 `python scripts/count_chars.py <本章文件>` 或等效统计，确认本章中文字符数达到 `00-section-plan.md` 的最低字数后，才允许标记为完成；未达标时只能加深已登记小节的目标、问题、研发内容、技术路线、验证口径、应用推广、单位分工和成果闭环，不得新增计划外二级节凑字数。

完整章必须满足以下硬门禁：
- `技术路线` 必须是本章最长的小节之一。三到五项子课题项目中，技术路线建议不少于 8000 中文字符；大型项目建议不少于 12000 中文字符。
- `项目研发内容` 与 `技术路线` 不得重复。研发内容写“做什么、分哪些任务、交付什么”，技术路线写“按什么架构、数据流、模型链、接口链、验证链去实现”。
- 必须建立 `问题 -> 研发任务 -> 指标或目标 -> 成果 -> 验收口径 -> 应用场景` 的对应关系，优先用表格呈现。
- 每项核心研发任务必须写清输入、技术动作、输出、对应指标、验证方式和责任主体。
- 每项量化目标或预期成果必须能回指至少一项研发任务和一项验收方式。
- 技术路线必须包含总体架构、数据流、模块关系、接口关系、测试验证、工程应用和安全保障，不能只写技术名词。

若第四章写完后只能说明“项目准备做什么”，但不能说明“为什么这样做、怎么做、如何验证、谁负责、成果如何验收和应用”，应判定为不合格并重写。

## 章节写法

### 1. 总体目标

`总体目标` 是本章的总领，不是简单写“建设一套平台”。通常先写 1 个总述段，再按 5 个三级小节展开。

总述段应写：
- 项目依据什么要求、面向什么建设对象、满足哪些核心指标或产业化指标。
- 项目相对要求是否有超额或强化安排，例如支持更多系统接口、覆盖更多工程应用场景、推广更多单位。
- 总述段可以出现关键数量指标，但必须来自 `facts-ledger.md` 或用户已给定材料。

#### 建设目标

建设目标应从“为什么建、建成什么、达到什么替代或应用效果”展开。建议按以下链路写：

`外部约束或卡点 -> 建设对象 -> 自主替代或能力提升目标 -> 工程应用和推广目标 -> 生态或开源目标`

可写内容包括：
- 打破国外软件垄断、降低断供风险和数据泄密风险。
- 建设国产自主可控的软件平台、系统工具、数据底座或工程能力。
- 对标国内外成熟系统或行业先进水平，但不得无依据写“国际领先”。
- 明确工程应用范围、推广单位、产品类型、性能目标、标准/专利/软著/论文等成果目标。
- 若用户材料包含指标，可在建设目标下用编号列出“技术指标”和“产业化指标”，但这些编号项必须服务目标论证。

常用句式：
- `为破除...依赖，降低...风险，项目拟联合...力量，打造...。`
- `项目建成后，将形成...，支撑...，实现...。`
- `通过工程应用及产业化推广，项目将...。`

#### 项目解决的主要问题

该小节不是背景痛点的重复，而是把背景问题转化为建设方案要解决的工程问题。常用写法是“总述段 + 若干问题标题 + 解释段”。

每个问题标题应写成“动作 + 对象 + 解决什么问题”的形式，例如：
- `设计一套完整的多层架构和标准，解决集成与适配难问题`
- `引入工程经验优化算法，解决核心算法工程化应用问题`
- `制定数据标准与规范，解决产品数字化定义不统一的问题`
- `建立软件平台测试，解决大体量多接口测试效率与质量问题`

每个问题段至少写清：
- 问题为什么存在。
- 对项目建设造成什么影响。
- 本项目准备用什么架构、标准、算法、平台、测试或工程应用方式解决。
- 该问题与后续研发内容或技术路线如何对应。

不要只写“效率低、质量差、协同难”。问题段要能直接引出任务分解。

#### 项目研发内容

`项目研发内容` 是本章的任务分解核心。推荐写法是：

1. 先写建设规模、建设地点或应用范围。
2. 再写技术水平和自主替代目标。
3. 随后用总述段说明“基于项目总体目标，遵循现代软件工程思想，从架构、关键技术、接口、功能、测试验证和工程应用等角度，将项目研发内容分解为若干任务”。
4. 之后按 `任务1 / 子任务1-1` 的方式展开。

任务分解建议采用 5 到 8 项，不宜少于 3 项。完整软件/平台类项目可拆成：
- 顶层设计或总体架构设计。
- 业务架构或应用场景研究。
- 数据架构或关键数据技术。
- 技术架构或基础平台开发技术。
- 核心业务功能研发。
- 接口集成研发。
- 测试验证。
- 工程应用或示范应用。

每个任务的总段应写：

`针对...特点和...问题，项目拟开展...，形成...，为...提供支撑。`

每个子任务段应写：

`子任务名称 -> 研究/建设对象 -> 技术动作 -> 输出成果 -> 支撑后续环节`

如果项目技术路线复杂，应在 `项目研发内容` 中预留或要求生成“项目主要建设任务分解图/表”。任务表建议字段：

`任务编号 | 任务名称 | 主要建设内容 | 关键输出 | 对应问题/指标`

禁止把项目研发内容写成只有任务标题和一句愿望。任务之间必须体现“顶层设计 -> 架构/关键技术 -> 功能/接口 -> 测试验证 -> 工程应用”的链路。

研发内容必须同步建立指标和验收接口。每项任务写完后，至少补齐以下内容：
- `任务输入`：来自哪些数据、系统、规范、场景、用户需求或前序任务。
- `关键技术动作`：采用什么模型、算法、架构、接口、规则、模板、测试或工程方法。
- `任务输出`：形成什么算法、模块、脚本、平台、数据集、模板、报告、规范或验证结果。
- `对应目标或指标`：支撑效率、质量、准确率、覆盖范围、安全、自主可控或推广目标中的哪一类。
- `验收口径`：通过样例数据、典型场景、用户试用、对比测试、专家评审、报告抽检、系统联调或上线运行中的哪种方式确认。

必须优先生成任务闭环表，字段建议为：

`任务编号 | 任务名称 | 对应问题 | 关键技术动作 | 主要交付物 | 对应指标/目标 | 验收口径 | 责任单位`

如果用户材料没有明确验收指标，不得编造精确指标；可写 `建议在验收方案中进一步明确`，但仍要说明验证方式。

#### 预期成果

在“项目建设方案”章内，`预期成果` 是对本章任务的总体成果收束，应写清可交付实体、成果类型、数量分布和任务对应关系。

应写清：
- 软件/平台/系统成果。
- 信创环境、测试环境、数据资源或标准体系。
- 报告、标准、专利、软著、论文等成果数量或分布。
- 成果与任务之间的对应关系。

如果本项目另有独立成果章节，当前小节只做摘要和分布，不替代详细成果章节；若没有独立成果章节，则应在本小节补足交付形态和应用场景。

预期成果必须和研发任务逐项对应。禁止只列“形成平台、形成算法、形成报告”等泛泛成果。每项成果至少写清：
- 成果名称和形态。
- 来源任务或子课题。
- 主要能力边界。
- 应用场景。
- 验收或确认方式。

建议生成成果对应表：

`成果编号 | 成果名称 | 成果形态 | 对应任务 | 支撑指标 | 应用场景 | 验收方式`

#### 产业链供应链韧性及安全保障

该小节不是泛泛安全意义，而是把建设方案与产业链供应链安全直接对应。常用写法是“总述段 + 4 到 5 个保障角度”：
- 发挥统筹管理优势，加速研制与应用，保障链路安全可控。
- 掌握核心算法与技术，推进技术生态双引流，降低断供风险。
- 强化数据管理及传递控制，减少产业链信息安全隐患。
- 打通全产业链数据流，提升产业链协同能力和韧性。
- 打造自主平台，提高产业链研发创新能力。

每个保障角度必须写清：
- 哪个链条风险或安全问题。
- 项目通过什么技术、平台、标准、接口、应用或管理动作降低风险。
- 对供应链自主可控、信息安全、协同效率或创新能力的作用。

### 2. 项目建设方案

`项目建设方案` 是本章后半部分，不能和前面的“项目研发内容”重复。前半写“要达成什么和做哪些任务”，后半写“按什么技术路线、怎样推广、怎样协同、怎样开源”。

#### 技术路线

技术路线应先写“总体技术方案及技术路线”，再展开具体任务建设路线，必要时追加合理性、可行性、自主性和创新性分析。

技术路线写法应具有以下特征：
- 先把总体技术架构分成若干部分，例如基础环境底座、数据底座、基础服务组件、业务平台、扩展接口平台。
- 再说明应用测试、接口测试、国产化适配测试、工程应用如何支撑总体路线。
- 随后按任务逐项写建设路线，任务之间要有信息输入、支撑关系和输出关系。
- 对每个任务可继续写 `子任务 / 要点 / 图示说明`，但必须服从 `00-section-plan.md` 的层级方案。
- 技术路线章节通常应有总体技术路线图、任务建设路线图、模块关系图或数据流图。若图示未生成，应在正文中预留图题并回传给 `official-doc-figure`。

技术路线段落常用链路：

`总体架构 -> 任务链路 -> 子任务建设路线 -> 测试验证 -> 工程应用 -> 合理性/可行性/自主性`

常用句式：
- `在充分分析...功能需求的基础上，结合...技术及验证手段，将总体技术架构分为...。`
- `按照...实际研发及工程应用流程，项目技术路线见下图。`
- `该任务的完成将为后续...提供顶层策划和指导。`
- `基于完成的...，本任务还将...，以统一...。`

技术路线不能只写“采用先进技术”。必须写清架构分层、模块关系、数据或业务流、测试验证和工程应用。

技术路线必须深写，按以下结构组织：

1. `总体技术路线总述`：说明项目采用什么总体路线，为什么该路线能够回应背景章痛点和总体目标。
2. `架构分层`：至少写出数据资源层、数据处理层、模型算法层、工程工具层、知识规则层、应用生成层、验证运维层中的若干层级，并说明各层输入输出。
3. `数据流和业务流`：说明原始数据从采集、解析、清洗、特征提取、模型处理、结果生成、规则校核到报告输出的流转过程。
4. `模型链路`：说明小模型、规则模型、大模型、知识图谱、模板引擎或其他算法之间如何分工，避免把所有能力都交给大模型。
5. `模块和接口关系`：说明系统或工具模块之间如何连接，与既有软件、平台、数据库、脚本或报告工具如何交互。
6. `任务建设路线`：逐项写明每项研发任务的建设步骤、关键难点、阶段输出和对后续任务的支撑。
7. `验证路线`：说明单元测试、集成测试、样例验证、典型场景验证、用户试用、专家评审或上线试运行如何组织。
8. `工程应用路线`：说明从实验样例到真实项目、从内部试点到多场景推广的实施路径。
9. `安全和自主可控路线`：说明数据安全、权限控制、私有化部署、国产化适配、知识产权和开源边界。
10. `路线合理性和可行性`：从技术成熟度、团队基础、数据基础、工程场景、单位分工和风险防控角度论证为什么可行。

技术路线图必须有正文解释。若无法直接绘图，至少写出图题和图示说明，例如：

`图4-1 项目总体技术路线图`

图示说明应写清：底部数据资源、中部模型和工具链、上层报告生成与应用验证、旁侧安全保障和反馈迭代之间的关系。不能只写“见图”或“此处插图”。

技术路线中必须设置一张 `技术路线与任务指标对应表`，建议字段为：

`技术路线环节 | 对应研发任务 | 输入数据/条件 | 核心方法 | 输出成果 | 对应指标 | 验证方式`

#### 应用推广方案

应用推广方案应写成“由内到外、由示范到规模化”的路径，而不是市场宣传。

推荐结构：
- 总述：联合体覆盖哪些产业链环节，为什么具备推广条件。
- 推广路径：先联合体内部应用，再示范单位窗口推广，再行业规模化推广，必要时向其他行业扩展。
- 用户反馈与迭代：跟踪应用方案、收集反馈、支撑二次开发、形成产品优化闭环。
- 推广方式：集团组织、成员单位辐射、高校合作、开源社区、线上线下宣传、培训和体验中心。
- 预期推广效果：国产替代、信创适配、生态构建、产业升级、辐射推广、产学结合。

每个推广段要回答：
- 面向哪些用户或单位。
- 通过什么路径推广。
- 如何形成反馈和持续改进。
- 预期达到什么应用效果。

#### 产学研用合作方式

产学研用合作方式要写清“谁负责什么、如何协同、如何闭环”。常用结构：
- 先写联合体覆盖范围和全流程能力。
- 再写高校、科研院所、企业、应用单位各自优势。
- 然后写联合体任务分配、合作模式、技术协同攻关、高校支持、任务比重等。

常用分工口径：
- `产`：软件设计、开发、测试组织、应用推广和运维服务。
- `学`：基础理论、关键技术研究、人才培养和学科支撑。
- `研`：行业知识、核心模块、关键技术攻关和测试评价。
- `用`：应用场景、工程验证、反馈迭代和示范推广。

合作机制应写成：

`用户提出需求和反馈 -> 企业开发测试与集成 -> 研究机构提供行业知识和验证支撑 -> 高校参与关键技术突破 -> 应用单位示范验证 -> 反馈促进产品迭代`

不得只罗列单位名称。每类单位都要有任务、接口和产出。

#### 攻关成果开源策略

开源策略应围绕“为什么开源、开源什么、怎么开源、如何防风险、如何闭环”展开。常用结构为：
- 攻关成果开源总体方向。
- 攻关成果总体开源策略。
- 攻关成果开源协作模式。
- 开源技术方案。
- 攻关成果开源推广方式。
- 攻关成果开源风险防范。
- 攻关成果开源闭环管理。

常见开源方向包括：
- 优化软件迭代策略，提升软件产品质量。
- 构建工业软件生态，形成行业知识合力。
- 引导行业应用软件创新，推动新型标准制定。

开源策略段落必须同时写：
- 开源范围和边界，例如组件、功能模块、接口、文档、测试工具或非敏感代码。
- 开源平台和协作机制，例如国内开源组织、社区协作、开发平台、API 和二次开发文档。
- 质量保障，例如单元测试、集成测试、用户验收测试、自动化测试和持续集成。
- 风险防范，例如知识产权、保密边界、安全审查、许可证选择和贡献审核。
- 闭环管理，例如用户反馈、版本迭代、社区治理、标准沉淀和推广应用。

不要把开源策略写成一句“部分代码开源”。必须形成策略链条。

## 写作步骤

### 第一步：识别本次要写的结构

从用户 brief 和 `00-section-plan.md` 判断：
- 是否写完整 `项目建设方案` 章
- 是否只写 `总体目标`
- 是否只写 `项目研发内容`
- 是否只写 `技术路线`
- 是否只写 `应用推广方案 / 产学研用 / 开源策略`
- 是否只命中本章规则，或需要按普通章节要求另写非专项内容

如果写完整章，默认使用“总体目标、项目建设方案”两个二级节及其九个三级项。若用户提示词或 `00-section-plan.md` 已经给出不同结构，优先服从 `00-section-plan.md`，但写法仍按本 skill 对应规则执行。

### 第二步：抽取证据并分配到小节

每个三级项写作前，必须先从 `source-materials.md`、`project-brief.md` 和 `facts-ledger.md` 的 USER 系列记录中抽取本项可直接使用的用户提示词项目资料，再从外部调研台账中抽取补充依据。

用户提示词项目资料抽取最低要求：
- `建设目标`：抽取项目总体目标、量化目标、应用对象、周期和安全边界。
- `项目解决的主要问题`：抽取用户给出的业务痛点、效率问题、错误率问题、数据沉淀问题和工程验证问题。
- `项目研发内容`：抽取三大子课题、子任务、关键技术动作、输入输出和单位分工。
- `预期成果`：抽取成果清单、成果数量、应用成效和量化成效。
- `产业链供应链韧性及安全保障`：抽取用户给出的自主可控、数据安全、合规、产业协同相关资料。
- `技术路线`：抽取阶段划分、数据链路、模型链路、系统架构、图示要求和验证安排。
- `应用推广方案`：抽取试点场景、推广对象、相邻场景复用和反馈迭代安排。
- `产学研用合作方式`：抽取牵头单位、参研单位、团队角色和分工边界。
- `攻关成果开源策略`：抽取开源边界、内部受控复用、脱敏共享和审批约束。

如果某个三级项在 `source-materials.md` 中已有明确项目资料，但证据清单没有摘录，视为资料消费不合格，不得开写。

随后，每个三级项必须从 `research-evidence.md`、`research-notes.md`、`facts-ledger.md` 中整理至少 2 个可消费证据点。完整章最低证据分配：
- `建设目标`：目标依据、指标依据、应用范围依据。
- `项目解决的主要问题`：背景痛点依据、问题成因依据、解决路径依据。
- `项目研发内容`：任务拆分依据、技术机制依据、输入输出或验证依据。
- `预期成果`：成果实体依据、成果数量或形态依据。
- `产业链供应链韧性及安全保障`：供应链风险依据、安全保障或自主可控依据。
- `技术路线`：总体架构依据、任务链路依据、测试验证或工程应用依据。
- `应用推广方案`：推广对象、推广路径、反馈迭代或效果指标依据。
- `产学研用合作方式`：成员能力、分工边界、协作机制依据。
- `攻关成果开源策略`：开源范围、许可证/平台、治理和风险防范依据。

证据清单必须同时区分：
- `用户给定项目事实`：来自 `source-materials.md`、`project-brief.md`、`facts-ledger.md` USER 系列记录，用于支撑本项目做什么、谁来做、投入多少、何时做、形成什么成果和达到什么目标。
- `外部调研补充依据`：来自公开来源，用于支撑政策、行业趋势、国内外案例、技术路线合理性、推广和开源治理口径。

正文不得直接写出 `RC-S01`、`F-001` 这类内部编号，但段落中的目标、任务、指标、推广和开源判断必须能回溯到这些证据点。

若第四章写完后只看得出外部行业分析，看不出用户提示词给出的三大子课题、任务分工、经费、周期、成果清单、量化成效和风险边界，视为“用户项目资料未消费”，必须退回重写。

### 第三步：按章级规则组织正文

语言要求：
- 正式
- 工程化
- 可研/立项/项目建议书口径
- 允许长段，但每段必须有任务、机制、输出或保障

推荐使用：
- `本项目依照...要求`
- `为破除...`
- `项目针对...挑战，重点研究并解决...`
- `基于项目总体目标，遵循...思想，从...角度出发，将项目研发内容分解为...`
- `针对...特点和...问题，开展...`
- `按照...实际研发及工程应用流程，项目技术路线...`
- `通过...形成...，支撑...`

避免：
- 只有任务标题，没有建设目标和建设方案闭环
- 把技术路线写成技术名词堆叠
- 把推广方案写成市场宣传
- 把产学研用写成单位名单
- 把开源策略写成一句口号
- 未核验的具体指标、数量、排名、市场份额和强结论

图表后必须有解释段，说明该图表如何支撑技术路线、任务闭环、验收管理或推广应用。不得只放表格不解释。
