# official-doc-research-content

## 定位

这是“研究内容/研发任务类内容”的规则库。

它负责回答：
- 项目研究什么
- 研究内容如何分组
- 每组下有哪些子课题或子任务
- 每项关键技术解决什么问题
- 最终形成哪些中间输出

如果用户只要求 `关键技术`，就只调用其中的关键技术规则。
如果用户要求 `研究内容 + 子课题 + 关键技术`，就按整套规则展开。

本 skill 默认假定：
- `using-official-docs` 已完成工作区和 plan 初始化
- `official-doc-core` 已完成公共校验

本 skill 不负责：
- 创建工作区
- 创建 plan 文件
- 决定是否直接进入 review / revise / assemble

本 skill 的正文结果必须写入 `workspace/outputs/<project-slug>/` 对应章节文件。
不要把完整研究内容大段文本直接输出到终端。

本 skill 写完后，必须把控制权交还给 `using-official-docs` 继续推进下游流程。

正式正文写法必须同时遵守：
- `official-doc-core` 中的公文文风规则
- [../official-doc-core/references/formal-doc-style.md](../official-doc-core/references/formal-doc-style.md)
- [../official-doc-core/references/attachment-writing-patterns.md](../official-doc-core/references/attachment-writing-patterns.md) 中的编号方案和研究内容写法
- [../official-doc-core/references/depth-writing-rules.md](../official-doc-core/references/depth-writing-rules.md)
- [../official-doc-core/references/depth-writing-examples.md](../official-doc-core/references/depth-writing-examples.md) 中的研究内容示例

`attachment-writing-patterns.md` 是硬门禁，也是必须使用的句式模板，不是可选参考。研究内容、子课题、关键技术段落必须套用该文件中“研究内容与关键技术”的常用写法和句式骨架，尤其是 `本项目围绕...目标，拟按照...主线组织研究内容`、`针对...问题，本研究拟开展...研究，开发/构建...，实现...`、`在...方面，研究...方法，形成...规则/模型/接口`、`最终形成...，为后续...提供...`、`...的关键不在于...，而在于...` 这类句式。每项研究内容、每个重点子课题和每项关键技术都要至少套用一处对应句式，并覆盖 `对象 / 问题 / 动作 / 机制 / 输出 / 验证` 中至少 4 项；重章节应覆盖 5 项以上。

研究内容类章节若属于可研/立项/任务书中的核心技术章节，不能只写“做什么”和“形成什么能力”，还必须下钻到“具体怎么做”。同级人工公文中的成熟写法通常会继续落到以下至少 2 类技术细节：
- 算法或处理链路：如 `先...再...最后...` 的步骤链、逐级求解逻辑、数据流转顺序。
- 可计算对象：如约束条件、规则字段、输入输出变量、数据标准化流程、坐标/距离/聚类/评分等计算关系。
- 工程实现单元：如脚本、接口、推理模块、规则引擎、模板库、验证环境、仿真平台。
- 验证边界：如典型船型、样例集、平台环境、准确率/召回率/时间消耗/约束满足率。

因此，若用户材料或调研证据里已经出现公式、伪代码、流程步骤、字段结构、规则项或验证工况，本 skill 必须优先把这些细节转写为正式技术叙述，必要时可在正文中使用简洁公式、变量关系或步骤描述。不得把现成的技术细节全部抹平为“多模态融合、几何计算、Agent协同、智能优化”这类抽象框架词。

对技术性较强的研究内容章节，公式和图示不是“可有可无的装饰”，而是技术深度证据。触发规则如下：
- 若本章存在 `计算 / 评分 / 距离 / 聚类 / 拓扑 / 优化 / 约束求解 / 标准化 / 语义映射 / 空间关系 / 参数化建模 / 拟合` 这类可计算机制，并且材料中已有变量关系、规则字段、步骤链或算法描述，则正文全章至少要出现 1 处公式、变量关系式或等价的伪代码式步骤说明。
- 若本章存在 3 个及以上前后依赖的研究模块、子课题或处理阶段，则正文或配图说明中至少要出现 1 张技术路线图、数据流图、模块关系图或流程图。没有图时，至少要在正文中明确预留图题与插入位置，并把配图需求回传给 `official-doc-figure`。
- 若章内同时具备“多阶段处理链路 + 可计算机制”，优先同时给出 `表/分解表 + 图示 + 公式/变量关系` 三者中的至少两类，不得只剩纯描述性自然段。
- 若本章存在 3 个及以上核心技术点，则其中至少 1 个技术点必须下钻到公式级细节，至少 1 个技术点必须下钻到伪代码式步骤、处理流程或参数化规则级细节；不能把所有技术点都写成同一种抽象概述。

写研究内容前，必须先从 `research-evidence.md`、`research-notes.md`、`facts-ledger.md` 中为每项研究内容或关键技术抽出至少 2 个证据点，其中至少 1 个用于支撑“问题来源/任务拆法”，1 个用于支撑“技术机制/工程边界”。没有这些证据点，不得直接写“研究A、研究B、研究C”。

研究内容类正文必须把调研消费成“为什么这样拆任务、为什么这样命名关键技术、为什么这样安排前后链路”。正文不得裸露内部来源编号，但每个任务包、子课题和关键技术都应能回溯到对应证据卡。若写完后只剩任务名、功能愿望和抽象能力词，而看不到问题来源、机制和输入输出，视为绕过调研写稿，必须退回回修。

研究内容类段落也要收口。不要反复堆写“提升能力、形成体系、实现协同、增强支撑”来补字数；优先补 `针对什么问题、研究什么对象、通过什么机制、形成什么输出、如何验证`。如果一个子课题段落中的“意义/价值”句超过主体机制句，说明写偏了。

研究内容类重章节还要防止“概念充足、技术发虚”。如果一段删掉 `多模态 / 几何计算 / Agent / 优化 / 闭环 / 体系` 这些抽象词后，仍看不出具体算法、规则、处理步骤或工程动作，说明技术深度不够，必须继续下钻到方法层。人工模板中常见的强写法不是简单报技术名，而是把 `处理对象 -> 规则/算法 -> 运行位置 -> 中间输出 -> 验证方式` 说透。

若用户提供的人工模板、历史项目材料或事实台账里已经出现了技术公式、变量定义、标准化流程、伪代码、参数化规则或流程图，本 skill 应视其为“应保留的技术密度”，而不是自动简化对象。除非用户明确要求写成管理摘要，否则生成稿应尽量保留这些技术痕迹，只做公文化转写，不做技术去势。

扩写研究内容时，不得新增 `00-section-plan.md` 未登记的二级节，也不得在既定小节末尾连续追加 `### 11.`、`### 12.`、`### 13.` 这类目录标题。需要加深时，应补子课题机制、输入输出、验证方式、研究内容表和表后关系说明。

若本章由子代理执行（如 Claude Code 并行任务），该子代理的首个动作必须是显式加载 `official-doc-research-content`，并在开始任何读写前先回报“已显式加载：official-doc-research-content”；未完成这一步，不得读取台账或写稿。该子代理也必须完整遵守本 skill 与 `attachment-writing-patterns.md` 的硬规则；不得通过追加流水号标题、弱化课题边界或省略验证口径来换取速度。
若运行环境支持 skill 显式调用，子代理必须在开始读取台账和写作前显式加载 `official-doc-research-content`；若当前被派发任务还同时覆盖创新点等其他内容，则继续按该任务实际命中的内容类型追加加载对应 skill。仅手动读取本文件路径不算完成本 skill 的调用。
子代理任务单应复用 `.claude/skills/official-doc-core/references/subagent-task-card-template.md`，并写明本章独占可编辑文件。
在正常多章节写作中，主控调用本 skill 后，不应直接把整章研究内容写完再伪装成“已按本 skill 执行”。正确做法是：主控先显式加载本 skill，依据本 skill 和调研台账提炼课题结构、编号规则与写法骨架，再派发对应 writing 子代理。若最终 `agent-prompts/*.md` 中看不到本 skill 的调用要求、主控预载记录、任务级留痕要求和提交回执要求，则说明本 skill 并未真正进入子代理执行链。

## 先做什么

开始写之前，先检查研究内容类调研门禁是否完成。

必须先读取：
- `workspace/plan/<project-slug>/research-plan.md`
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/research-notes.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

如果研究内容类调研仍存在以下任一情况，应先回到 `official-doc-research` 补调研：
- 同类任务拆法还只有一两个零散样例
- 关键技术命名缺少近 3 年公开资料校准
- 研究内容边界仍明显拍脑袋
- `research-evidence.md` 缺少同类任务、技术路线或子课题拆分的证据卡
- `research-notes.md` 尚未沉淀出“问题 -> 方法 -> 输出”的摘记

本 skill 不直接联网搜索。只有在研究内容大框架已建立、但某个子课题仍需补证时，才回到 `official-doc-research` 补充调研，由 `official-doc-research` 使用本地 MCP 完成检索、抓取和落账。补查重点包括：
- 同领域项目指南、攻关任务书、立项书
- 相关技术路线论文、白皮书、行业报告
- 头部企业、研究机构公开方案

补充调研结果必须记录到：
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

搜索目的不是抄材料，而是校准三件事：
- 该主题通常拆成哪些研究方向
- 关键技术通常如何命名才专业且可落地
- 子课题边界如何划分才合理

补查完成后，本 skill 优先读取 `research-evidence.md` 的证据卡，再读取调研台账和事实台账，不自行改用其他搜索工具。

## 写作深度门槛

研究内容类正文不能写成“研究A、研究B、研究C”的名词清单。每项研究内容必须建立任务包论证链：
- `问题来源`：对应背景、痛点或现状差距中的哪一类问题。
- `研究对象`：处理的是数据、规则、模型、系统、流程、设备、报告还是场景。
- `技术机制`：采用什么方法、模型、规则、算法、工具链或流程编排来解决问题。
- `子课题边界`：每个子课题解决的具体环节是什么，彼此如何衔接。
- `输入输出`：该研究内容需要什么输入，形成什么中间成果或交付件。
- `验证方式`：后续如何通过指标、场景、样例、系统测试或报告验收证明有效。

每个研究内容段落至少写出“问题 -> 方法 -> 输出”三项，重章节还应写出“原因 -> 机制 -> 验证”。如果只能写出任务名称，说明调研或设计仍不足，应先补 `research-notes.md`，不得靠套话扩写。

## 从附件提炼出的共性写法

写作前必须先读取 `00-section-plan.md` 中的全文编号方案。若本章采用 `课题1 / 子课题1.1 / 1.1.1`，则该研究内容内部不再把同级条目写成 `（1）`；若全文方案要求正式报告型编号，则章、节标题仍按 `一、`、`（一）` 统筹，课题编号只用于研究内容内部下钻。

正式报告型文本中，章内二级标题必须服从全文方案。例如第三章 `研发内容及技术关键` 下应写 `### （一）项目研究的内容`、`### （二）关键技术及创新点`，不得写 `### 一、项目研究的内容`。课题和子课题属于研究内容内部层级，可以保留 `课题1`、`子课题1.1` 这样的技术编号，但不能替代全文标题编号。

研究内容内部的编号必须在父级内重新起号，不能跨课题、跨二级节累计。若采用正式报告型正文编号，推荐严格按以下结构执行：

```md
### （一）项目研究的内容

#### 1. 研究内容总述

#### 2. 研究内容分解表

#### 3. 研究内容关系说明

#### 4. 课题一：...

课题一研究背景与问题来源：用课题标题下的首段展开，或写成 `##### （1）课题一研究背景与问题来源`
##### （2）...
##### （3）...

#### 5. 课题二：...

课题二研究背景与问题来源：再次从首段或 `##### （1）...` 重新起号
##### （2）...
##### （3）...
```

也就是说：
- `#### 4. 课题一`、`#### 5. 课题二` 这一级必须使用带点号的 `1.`、`2.`、`3.` 体系。
- 每个课题内部的四级标题都必须重新从 `（1）` 起号。
- 不得出现 `#### （9）课题一研究背景与问题来源`、`#### （10）课题二研究背景与问题来源` 这类跨课题累计编号。
- 若“课题研究背景与问题来源”只是引入性说明，优先写成课题标题下首段，而不是专门占用一个带编号的四级标题。

本 skill 只能在 `00-section-plan.md` 已登记的二级节内写作。若计划只允许 `（一）项目研究的内容`、`（二）关键技术及创新点`，不得追加 `（三）技术创新与突破`、`（四）研究方案可行性分析`、`（五）课题设置与任务分工` 等二级节。字数不足时，应加深子课题、关键技术、创新点和技术路线图说明，不得新增二级节凑字数。

### 1. 先总后分，再层层下钻

附件中的成熟写法基本都是：
- 先写项目总体目标或总述
- 再写若干研究内容
- 每个研究内容下继续拆子课题或子任务
- 最后单独提炼关键技术

### 2. 研究内容不是“名词清单”，而是“任务包”

高质量文本通常会写成：
- 解决什么问题
- 开发/构建/研究什么
- 实现什么能力
- 最终形成什么输出

### 3. 子课题之间要有链路关系

附件里常见的拆法不是平铺重复，而是体现：
- 前置数据准备
- 核心算法或模型
- 工具或平台实现
- 验证与集成

也就是说，子课题应体现依赖关系、数据流或功能链路。

### 4. 常见句式是“针对痛点 -> 拟开发/研究 -> 实现 -> 形成”

例如附件中大量使用类似结构：
- `针对……核心痛点，本研究拟开发……`
- `研究……方法，实现……能力`
- `最终形成……`

### 5. 附件式研究内容模板

重章节优先采用以下结构，而不是直接列标题：

1. 先写一段总述，说明项目按什么技术主线组织研究内容。
2. 若内容较多，生成或引用一张“研究内容-子研究内容”表，字段建议为 `序号 / 研究内容 / 子研究内容 / 对应输出`。
3. 表后用一段说明各研究内容之间的关系，如数据底座、规则建模、智能生成、校核验证、平台集成之间的前后依赖。
4. 每项研究内容按 `针对问题 -> 研究/开发/构建对象 -> 技术机制 -> 实现能力 -> 形成输出` 展开。

若章节技术性较强，再补一层“技术细节锚点”：
- 至少 1 处公式、变量关系或矩阵/评分/距离类计算表达；
- 至少 1 处伪代码式步骤、参数化规则链或标准化处理流程；
- 至少 1 处图示、图位说明或模块关系图说明；
- 至少 1 处验证环境、平台或样例边界说明。

顺序要求：正式可研/建议书中，研究内容表应放在详细展开之前。不要先写完所有课题再补表；这会更像生成稿而不是附件中的项目材料。推荐顺序为：

```md
### （一）项目研究的内容

#### 1. 研究内容总述

#### 2. 研究内容分解表

#### 3. 研究内容关系说明

#### 4. 课题1：...

##### 子课题1.1：...
```

禁止用 `**子课题1.1：...**` 充当标题。子课题若需要单独成段，必须使用标题层级；若不需要标题，则写成自然段开头 `子课题1.1主要面向...`，不得加粗。

可直接套用的段落骨架：

```md
针对...导致的...问题，本研究拟开展...研究，构建/开发...，实现...。在...方面，研究...方法，形成...规则/模型/接口；在...基础上，开发...模块，支撑...。最终形成...，为后续...提供...。
```

关键技术单独提炼时使用：

```md
...的关键不在于...，而在于...。项目拟通过...，建立...，实现...。该技术位于...与...之间，承担从...到...的转换作用，是支撑...的核心环节。
```

这是可研和任务书中非常稳定的工程化写法。

### 6. 从人工模板提炼出的技术深写法

同级人工材料里的“主要研究内容 / 关键技术”有两个稳定特征，本 skill 必须吸收：

1. 课题与子课题命名先落对象，再落动作，不用空泛能力词。优先写 `基于...的...自动生成`、`...识别与语义建模`、`...规则体系`、`...验证与集成`，少写 `智能升级`、`平台建设`、`能力提升`。
2. 每项关键技术通常不是一句话，而是两层展开：
   - 第一层写 `围绕什么对象，构建什么框架/模型/规则体系，通过什么动作实现什么处理能力`。
   - 第二层写 `针对什么约束/工况/边界，再把规则、字段、空间关系、求解策略或验证环境落细`。

当章节技术性较强时，优先采用下面这种工程化骨架，而不是只写概念解释：

```md
针对...场景中...不足的问题，本课题拟开展...研究。首先，围绕...输入建立...解析与标准化流程，形成...结构化数据；其次，针对...约束/目标，研究...算法/规则表达方法，实现...；最后，在...平台/样例/典型船型上完成...验证，输出...模块/脚本/模型/报告。
```

关键技术段落优先采用下面这种“两段式”：

```md
（1）...技术
围绕...对象，构建...框架。通过...、...和...，实现...，形成...输入基础/计算基础。
针对...工况/约束，建立...规则/变量关系/求解策略，并在...环境中完成...验证，使...能够直接支撑后续...。
```

如果用户上传材料、历史公文或 facts-ledger 中已有更细的公式、伪代码、字段表、流程步骤，本 skill 应把它们转写成正式技术语言，而不是一律概括成“采用先进算法”。对核心研究内容，宁可少写空泛价值句，也要多写一层可执行技术动作。

### 7. 公式、图示与技术细节的落位规则

人工模板和高质量技术方案类材料的第三章，常见的“技术深度信号”不只是字数，而是正文里出现了可核对的技术锚点。本 skill 应按下面的优先级落位：

1. **研究内容分解表**：先建立总体任务结构。
2. **关系说明或技术路线图**：说明模块依赖、数据流或求解顺序。
3. **子课题/关键技术正文**：在至少 1 至 3 个核心技术点中补入公式、变量关系、字段结构、标准化流程或伪代码式步骤。

推荐写法：

```md
图X 项目总体技术路线图
输入数据 -> 结构化解析 -> 规则/约束编码 -> 智能求解 -> 平台验证 -> 成果输出
```

注意：
- 公式不要求堆很多，但至少要体现“这个技术点是可计算的”，不能只剩概念名词。
- 伪代码不要求写成程序代码块，但至少要达到“输入 -> 处理 -> 约束/参数 -> 输出”的步骤级细节，不能只写“采用某算法”。
- 图示不要求复杂，但必须体现模块关系、数据流或求解链路，不能只是装饰性图题。
- 若材料里没有现成公式，可用“变量关系 + 步骤链”替代；若材料里已有公式、标准化流程或伪代码，则优先保留并公文化转写。
- 若人工模板中已有公式、变量关系、标准化流程、参数化规则或伪代码级细节，而生成稿只剩概念说明，应直接判定为技术深度不足，不得视为“风格差异可接受”。

## 基本写作框架

### A. 总体目标或研究总述

先用一段到两段写清：
- 围绕什么总体目标
- 解决什么核心问题
- 构建什么能力主线

不要一开头就直接列 1、2、3，而没有总述。

### B. 研究内容分组

一般建议拆成 3 到 7 项研究内容。

命名要像任务包，而不是口号。

较好的命名方式：
- `基于……的……技术研究`
- `……模型构建`
- `……平台研发`
- `……自动化方法研究`

较差的命名方式：
- `智能升级`
- `平台建设`
- `能力提升`

### C. 每项研究内容内部继续拆子课题

每项研究内容建议拆 2 到 5 个子课题或子任务。

每个子课题至少回答四件事：
- 解决什么问题
- 具体做什么
- 关键技术是什么
- 形成什么输出

但这四件事应融入自然段，不要写成：
- `**技术描述**：`
- `**技术难点**：`
- `**作用**：`
- 或大段项目符号

### D. 关键技术单独提炼

如果用户要求 `关键技术`、`技术关键`、`核心技术`，要从研究内容中提炼而来，而不是另起一套完全无关的话术。

## 研究内容的详细写法

### 1. 研究内容总述

推荐写法：
- 先概括项目全链路或能力主线
- 再说明研究内容的总体分工逻辑
- 最后说明这些内容如何共同支撑总目标

常见主线类型：
- `数据准备 -> 特征解析 -> 模型/算法 -> 工具链实现 -> 平台集成 -> 工程验证`
- `规则/知识建模 -> 数据治理 -> 智能生成 -> 校核修正 -> 自动交付`

### 2. 单项研究内容的推荐结构

对于每一项研究内容，优先采用以下顺序：
1. 问题与目标
2. 研究对象和范围
3. 关键方法或技术路线
4. 中间输出或阶段成果

可直接借鉴的工程化句式：
- `针对……导致的……问题，本研究拟开展……`
- `通过……实现……`
- `重点突破……`
- `最终形成……能力/模块/方法/数据集`

### 3. 子课题的推荐结构

子课题要写出清晰边界，避免重复套话。

可从以下维度选用：
- 数据层
- 模型层
- 规则层
- 工具层
- 平台层
- 验证层

例如：
- 子课题 1.1：数据采集与标准化
- 子课题 1.2：特征表达与标签体系
- 子课题 1.3：分类或生成模型
- 子课题 1.4：工具接口与批处理能力

### 4. 关键技术的写法

关键技术不是简单把研究内容改个标题。

每项关键技术要回答：
- 技术边界是什么
- 解决的核心难点是什么
- 采用什么技术机制
- 在整个项目链路中起什么作用
- 带来什么工程价值

更接近附件风格的写法是：
- 先点明瓶颈
- 再写机制
- 再写应用位置
- 最后写成效

## 正文呈现规则

### 1. 不要把提示词原样铺开成目录

用户给出的子课题清单，是写作输入，不是最终正文格式。

除非用户明确要求保留多级编号目录，或 `00-section-plan.md` 已明确采用课题化编号，否则不要机械输出：
- `1.1`
- `1.1.1`
- `1.1.2`

再逐条配上一句短说明。

更好的做法是：
- 先写一段总述
- 再按 3 到 5 个研究方向分节
- 每节下用连续自然段展开关键任务和技术链路

### 2. 不要写成“答题卡体”

错误示例：

```md
**技术描述**：研究自动化训练样本生成技术。

**技术难点**：保证样本与真实工程一致。

**作用**：为分类模型训练提供支撑。
```

正确示例：

```md
针对训练样本不足的问题，本研究拟建立自动化训练样本生成技术，通过参数化模型模板和批量化变参机制快速构造训练样本。该技术的关键在于保持样本与真实工程场景的一致性，并控制自动标注误差。相关成果将为后续分类模型训练提供稳定的数据支撑。
```

### 3. 不要写成项目符号正文

错误示例：

```md
本研究内容包括：
- 数据标准化
- 规则建模
- 模型训练
- 平台实现
```

正确示例：

```md
本研究围绕数据标准化、规则建模、模型训练和平台实现四个环节展开。其中，数据标准化负责统一多源输入口径，规则建模负责沉淀行业知识和设计约束，模型训练负责形成核心识别与生成能力，平台实现则承担流程集成与工程验证任务。
```

## 从附件归纳出的常见拆分维度

按主题不同，可组合使用以下研究内容维度：
- 数据标准化与样本构建
- 特征体系与标签体系
- 分类/识别/生成模型
- 规则建模与知识图谱
- 自动化脚本与二次开发
- 模板引擎与文档生成
- 平台集成与流程编排
- 测试验证与工程应用

## 与其他章节的边界

### 与 `official-doc-innovation` 的边界

本 skill 重点写“做什么、怎么做、怎么拆任务”。

不要把创新点单独大段写在这里，除非用户明确要求“关键技术及创新点”并且当前章确实要同时写创新内容。

### 与 `official-doc-technical-achievements` 的边界

本 skill 只写研究任务和中间输出逻辑，不直接替代最终成果章节。

### 与 `official-doc-technical-indicators` 的边界

本 skill 可埋下可量化考核点，但不要把指标表完整写进来，除非用户明确要求。

## 触发样例

以下情形都必须触发本 skill：

### 样例 1

用户写：
- `某一章研发内容及技术关键，需要拆成三个子课题，每个子课题写解决什么问题、关键技术是什么`

应触发原因：
- `研究内容`、`子课题`、`关键技术` 同时命中

### 样例 2

用户写：
- `某一章实施方案里先写任务设置、子任务分解和专题安排`

应触发原因：
- 即使章名是 `实施方案`，但 `任务设置`、`子任务`、`专题安排` 本质上仍属于研究内容规则

### 样例 3

用户写：
- `请围绕主题列出主要攻关内容，每项再展开2-3个子任务`

应触发原因：
- `主要攻关内容` 和 `子任务` 是本 skill 的直接关键词

### 样例 4

用户写：
- `项目建设方案里需要写建设内容、关键技术和阶段任务`

应触发原因：
- `建设内容`、`关键技术`、`阶段任务` 仍然需要调用本 skill 的任务分解规则

## 输出检查

写完后确认：
- 是否先做了搜索并登记来源
- 是否形成“总述 -> 研究内容 -> 子课题 -> 关键技术”的层级
- 每项研究内容是否回答了“问题/动作/技术/输出”
- 子课题之间是否有边界和顺序
- 是否能映射到后续成果和指标

## 不要做的事

- 不要只列几个名词
- 不要只有研究内容标题，没有展开
- 不要只有研究内容，没有子课题
- 不要把关键技术写成空泛套话
- 不要让各研究内容之间相互重复
