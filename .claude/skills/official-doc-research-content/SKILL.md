# official-doc-research-content

## 定位

这是“项目建设方案”章的写作规则库，不再是旧版泛化内容规则库。

当前主要参照模板为 `产业链项目指南/完整科研项目模板.docx` 的“四、项目建设方案”。当用户要求撰写完整项目建设方案章，且未另行给定更强模板时，默认采用该模板的章节结构和写法：

```md
四、项目建设方案
（一）总体目标
1. 建设目标
2. 项目解决的主要问题
3. 项目研发内容
4. 预期成果
5. 产业链供应链韧性及安全保障
（二）项目建设方案
1. 技术路线
2. 应用推广方案
3. 产学研用合作方式
4. 攻关成果开源策略
```

如果用户只要求其中一部分，只写用户要求的部分，并沿用本 skill 中对应小节的写法。不得因为本 skill 覆盖完整章，就自行追加 `00-section-plan.md` 未登记的小节。

本 skill 覆盖的写作对象包括但不限于：
- 项目建设方案
- 总体目标、建设目标
- 项目解决的主要问题
- 项目研发内容、建设任务、任务分解
- 预期成果
- 产业链供应链韧性及安全保障
- 技术路线、总体技术方案、技术路线合理性、可行性、自主性
- 应用推广方案、应用推广路径、预期应用推广效果
- 产学研用合作方式、联合体合作模式、任务分配
- 攻关成果开源策略、开源方向、开源协作模式、开源风险防范、开源闭环管理

## 先做什么

开始写之前先检查建设方案类调研门禁是否完成，而不是直接编任务。

必须先读取：
- `workspace/plan/<project-slug>/research-plan.md`
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/research-notes.md`
- `workspace/plan/<project-slug>/facts-ledger.md`
- `workspace/outputs/<project-slug>/00-section-plan.md`

如果建设方案类调研仍存在以下任一情况，应先回到 `official-doc-research` 补调研：
- 项目目标、建设规模、应用范围或技术指标没有依据
- “项目解决的主要问题”无法回指背景章中的痛点和现状差距
- 研发内容拆分只有任务名，缺少任务边界、输入输出和对应成果
- 技术路线缺少总体架构、模块依赖、测试验证或工程应用链路
- 应用推广、产学研用、开源策略没有单位、路径、机制或风险防范依据
- `research-evidence.md` 缺少同类项目建设方案、技术路线或任务分解的证据卡
- `research-notes.md` 尚未沉淀出“目标 -> 问题 -> 任务 -> 技术路线 -> 成果/推广”的摘记

本 skill 不直接联网搜索。只有在建设方案框架已建立、但某个目标值、任务边界、推广路径或开源策略仍缺支撑时，才回到 `official-doc-research` 补充调研。

补充调研结果必须记录到：
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

`attachment-writing-patterns.md` 是硬门禁，不是可选参考。项目建设方案章必须写成“总体目标 -> 项目建设方案”的递进结构，而不是只列任务、技术名词和功能愿望。

本章的核心论证顺序是：

```md
总体目标：建设目标 -> 项目解决的主要问题 -> 项目研发内容 -> 预期成果 -> 产业链供应链韧性及安全保障
项目建设方案：技术路线 -> 应用推广方案 -> 产学研用合作方式 -> 攻关成果开源策略
```

建设方案类关键段落必须至少覆盖 `目标 / 问题 / 任务 / 路线 / 输出 / 应用或保障` 中 4 项。若写完后只剩任务名、功能愿望和价值判断，必须先回修再交付。

## 新模板提炼出的章节写法

### 1. 总体目标

`总体目标` 是本章的总领，不是简单写“建设一套平台”。参照模板，通常先写 1 个总述段，再按 5 个三级小节展开。

总述段应写：
- 项目依据什么要求、面向什么建设对象、满足哪些核心指标或产业化指标。
- 项目相对要求是否有超额或强化安排，例如支持更多系统接口、覆盖更多工程应用场景、推广更多单位。
- 总述段可以出现关键数量指标，但必须来自 `facts-ledger.md` 或用户已给定材料。

#### 建设目标

建设目标应从“为什么建、建成什么、达到什么替代或应用效果”展开。建议按以下链路写：

`外部约束或卡点 -> 建设对象 -> 自主替代或能力提升目标 -> 工程应用和推广目标 -> 生态或开源目标`

可写内容包括：
- 打破国外软件垄断、降低断供风险和数据泄密风险。
- 建设国产自主可控的软件平台、系统工具、数据底座或工程能力。
- 对标国内外成熟系统或行业先进水平，但不得无依据写“国际领先”。
- 明确工程应用范围、推广单位、产品类型、性能目标、标准/专利/软著/论文等成果目标。
- 若用户材料包含指标，可在建设目标下用编号列出“技术指标”和“产业化指标”，但这些编号项必须服务目标论证。

常用句式：
- `为破除...依赖，降低...风险，项目拟联合...力量，打造...。`
- `项目建成后，将形成...，支撑...，实现...。`
- `通过工程应用及产业化推广，项目将...。`

#### 项目解决的主要问题

该小节不是背景痛点的重复，而是把背景问题转化为建设方案要解决的工程问题。模板常用写法是“总述段 + 若干问题标题 + 解释段”。

每个问题标题应写成“动作 + 对象 + 解决什么问题”的形式，例如：
- `设计一套完整的多层架构和标准，解决集成与适配难问题`
- `引入工程经验优化算法，解决核心算法工程化应用问题`
- `制定数据标准与规范，解决产品数字化定义不统一的问题`
- `建立软件平台测试，解决大体量多接口测试效率与质量问题`

每个问题段至少写清：
- 问题为什么存在。
- 对项目建设造成什么影响。
- 本项目准备用什么架构、标准、算法、平台、测试或工程应用方式解决。
- 该问题与后续研发内容或技术路线如何对应。

不要只写“效率低、质量差、协同难”。问题段要能直接引出任务分解。

#### 项目研发内容

`项目研发内容` 是本章的任务分解核心。模板写法是：

1. 先写建设规模、建设地点或应用范围。
2. 再写技术水平和自主替代目标。
3. 随后用总述段说明“基于项目总体目标，遵循现代软件工程思想，从架构、关键技术、接口、功能、测试验证和工程应用等角度，将项目研发内容分解为若干任务”。
4. 之后按 `任务1 / 子任务1-1` 的方式展开。

任务分解建议采用 5 到 8 项，不宜少于 3 项。完整软件/平台类项目可参考模板拆成：
- 顶层设计或总体架构设计。
- 业务架构或应用场景研究。
- 数据架构或关键数据技术。
- 技术架构或基础平台开发技术。
- 核心业务功能研发。
- 接口集成研发。
- 测试验证。
- 工程应用或示范应用。

每个任务的总段应写：

`针对...特点和...问题，项目拟开展...，形成...，为...提供支撑。`

每个子任务段应写：

`子任务名称 -> 研究/建设对象 -> 技术动作 -> 输出成果 -> 支撑后续环节`

如果项目技术路线复杂，应在 `项目研发内容` 中预留或要求生成“项目主要建设任务分解图/表”。任务表建议字段：

`任务编号 | 任务名称 | 主要建设内容 | 关键输出 | 对应问题/指标`

禁止把项目研发内容写成只有任务标题和一句愿望。任务之间必须体现“顶层设计 -> 架构/关键技术 -> 功能/接口 -> 测试验证 -> 工程应用”的链路。

#### 预期成果

在“项目建设方案”章内，`预期成果` 是对本章任务的总体成果收束，可以比 `official-doc-technical-achievements` 更概括，但仍必须有可交付实体。

应写清：
- 软件/平台/系统成果。
- 信创环境、测试环境、数据资源或标准体系。
- 报告、标准、专利、软著、论文等成果数量或分布。
- 成果与任务之间的对应关系。

如果本项目另有独立“主要技术成果”章节，当前小节只做摘要和分布，不替代详细成果章节；若没有独立成果章节，则应按技术成果 skill 的成果实体规则补足交付形态。

#### 产业链供应链韧性及安全保障

该小节不是泛泛安全意义，而是把建设方案与产业链供应链安全直接对应。模板常用写法是“总述段 + 4 到 5 个保障角度”：
- 发挥统筹管理优势，加速研制与应用，保障链路安全可控。
- 掌握核心算法与技术，推进技术生态双引流，降低断供风险。
- 强化数据管理及传递控制，减少产业链信息安全隐患。
- 打通全产业链数据流，提升产业链协同能力和韧性。
- 打造自主平台，提高产业链研发创新能力。

每个保障角度必须写清：
- 哪个链条风险或安全问题。
- 项目通过什么技术、平台、标准、接口、应用或管理动作降低风险。
- 对供应链自主可控、信息安全、协同效率或创新能力的作用。

### 2. 项目建设方案

`项目建设方案` 是本章后半部分，不能和前面的“项目研发内容”重复。前半写“要达成什么和做哪些任务”，后半写“按什么技术路线、怎样推广、怎样协同、怎样开源”。

#### 技术路线

技术路线应先写“总体技术方案及技术路线”，再展开具体任务建设路线，必要时追加合理性、可行性、自主性和创新性分析。

模板中的技术路线写法具有以下特征：
- 先把总体技术架构分成若干部分，例如基础环境底座、数据底座、基础服务组件、业务平台、扩展接口平台。
- 再说明应用测试、接口测试、国产化适配测试、工程应用如何支撑总体路线。
- 随后按任务逐项写建设路线，任务之间要有信息输入、支撑关系和输出关系。
- 对每个任务可继续写 `子任务 / 要点 / 图示说明`，但必须服从 `00-section-plan.md` 的层级方案。
- 技术路线章节通常应有总体技术路线图、任务建设路线图、模块关系图或数据流图。若图示未生成，应在正文中预留图题并回传给 `official-doc-figure`。

技术路线段落常用链路：

`总体架构 -> 任务链路 -> 子任务建设路线 -> 测试验证 -> 工程应用 -> 合理性/可行性/自主性`

常用句式：
- `在充分分析...功能需求的基础上，结合...技术及验证手段，将总体技术架构分为...。`
- `按照...实际研发及工程应用流程，项目技术路线见下图。`
- `该任务的完成将为后续...提供顶层策划和指导。`
- `基于完成的...，本任务还将...，以统一...。`

技术路线不能只写“采用先进技术”。必须写清架构分层、模块关系、数据或业务流、测试验证和工程应用。

#### 应用推广方案

应用推广方案应写成“由内到外、由示范到规模化”的路径，而不是市场宣传。

推荐结构：
- 总述：联合体覆盖哪些产业链环节，为什么具备推广条件。
- 推广路径：先联合体内部应用，再示范单位窗口推广，再行业规模化推广，必要时向其他行业扩展。
- 用户反馈与迭代：跟踪应用方案、收集反馈、支撑二次开发、形成产品优化闭环。
- 推广方式：集团组织、成员单位辐射、高校合作、开源社区、线上线下宣传、培训和体验中心。
- 预期推广效果：国产替代、信创适配、生态构建、产业升级、辐射推广、产学结合。

每个推广段要回答：
- 面向哪些用户或单位。
- 通过什么路径推广。
- 如何形成反馈和持续改进。
- 预期达到什么应用效果。

#### 产学研用合作方式

产学研用合作方式要写清“谁负责什么、如何协同、如何闭环”。模板常用结构：
- 先写联合体覆盖范围和全流程能力。
- 再写高校、科研院所、企业、应用单位各自优势。
- 然后写联合体任务分配、合作模式、技术协同攻关、高校支持、任务比重等。

常用分工口径：
- `产`：软件设计、开发、测试组织、应用推广和运维服务。
- `学`：基础理论、关键技术研究、人才培养和学科支撑。
- `研`：行业知识、核心模块、关键技术攻关和测试评价。
- `用`：应用场景、工程验证、反馈迭代和示范推广。

合作机制应写成：

`用户提出需求和反馈 -> 企业开发测试与集成 -> 研究机构提供行业知识和验证支撑 -> 高校参与关键技术突破 -> 应用单位示范验证 -> 反馈促进产品迭代`

不得只罗列单位名称。每类单位都要有任务、接口和产出。

#### 攻关成果开源策略

开源策略应围绕“为什么开源、开源什么、怎么开源、如何防风险、如何闭环”展开。模板常用结构为：
- 攻关成果开源总体方向。
- 攻关成果总体开源策略。
- 攻关成果开源协作模式。
- 开源技术方案。
- 攻关成果开源推广方式。
- 攻关成果开源风险防范。
- 攻关成果开源闭环管理。

常见开源方向包括：
- 优化软件迭代策略，提升软件产品质量。
- 构建工业软件生态，形成行业知识合力。
- 引导行业应用软件创新，推动新型标准制定。

开源策略段落必须同时写：
- 开源范围和边界，例如组件、功能模块、接口、文档、测试工具或非敏感代码。
- 开源平台和协作机制，例如国内开源组织、社区协作、开发平台、API 和二次开发文档。
- 质量保障，例如单元测试、集成测试、用户验收测试、自动化测试和持续集成。
- 风险防范，例如知识产权、保密边界、安全审查、许可证选择和贡献审核。
- 闭环管理，例如用户反馈、版本迭代、社区治理、标准沉淀和推广应用。

不要把开源策略写成一句“部分代码开源”。必须形成策略链条。

## 写作步骤

### 第一步：识别本次要写的结构

从用户 brief 和 `00-section-plan.md` 判断：
- 是否写完整 `项目建设方案` 章
- 是否只写 `总体目标`
- 是否只写 `项目研发内容`
- 是否只写 `技术路线`
- 是否只写 `应用推广方案 / 产学研用 / 开源策略`
- 是否需要与 `official-doc-innovation`、`official-doc-technical-achievements`、`official-doc-technical-indicators` 同章叠加

如果写完整章，默认使用“总体目标、项目建设方案”两个二级节及其九个三级项。若用户或模板已经给出不同结构，优先服从 `00-section-plan.md`，但写法仍按本 skill 对应规则执行。

### 第二步：抽取证据并分配到小节

每个三级项写作前，必须从 `research-evidence.md`、`research-notes.md`、`facts-ledger.md` 中整理至少 2 个可消费证据点。完整章最低证据分配：
- `建设目标`：目标依据、指标依据、应用范围依据。
- `项目解决的主要问题`：背景痛点依据、问题成因依据、解决路径依据。
- `项目研发内容`：任务拆分依据、技术机制依据、输入输出或验证依据。
- `预期成果`：成果实体依据、成果数量或形态依据。
- `产业链供应链韧性及安全保障`：供应链风险依据、安全保障或自主可控依据。
- `技术路线`：总体架构依据、任务链路依据、测试验证或工程应用依据。
- `应用推广方案`：推广对象、推广路径、反馈迭代或效果指标依据。
- `产学研用合作方式`：成员能力、分工边界、协作机制依据。
- `攻关成果开源策略`：开源范围、许可证/平台、治理和风险防范依据。

正文不得直接写出 `RC-S01`、`F-001` 这类内部编号，但段落中的目标、任务、指标、推广和开源判断必须能回溯到这些证据点。

### 第三步：按章级模板组织正文

语言要求：
- 正式
- 工程化
- 可研/立项/项目建议书口径
- 允许长段，但每段必须有任务、机制、输出或保障

推荐使用：
- `本项目依照...要求`
- `为破除...`
- `项目针对...挑战，重点研究并解决...`
- `基于项目总体目标，遵循...思想，从...角度出发，将项目研发内容分解为...`
- `针对...特点和...问题，开展...`
- `按照...实际研发及工程应用流程，项目技术路线...`
- `通过...形成...，支撑...`

避免：
- 只有任务标题，没有建设目标和建设方案闭环
- 把技术路线写成技术名词堆叠
- 把推广方案写成市场宣传
- 把产学研用写成单位名单
- 把开源策略写成一句口号
- 未核验的具体指标、数量、排名、市场份额和强结论

## 与其他专项 skill 的边界

### 与 `official-doc-innovation`

本 skill 可以写技术路线创新性或方案自主性，但不得把独立“创新点”章节全部并入本章。若用户明确要求在技术路线中写创新性，可先按本 skill 写项目建设方案，再追加加载 `official-doc-innovation` 校准创新点表达。

### 与 `official-doc-technical-achievements`

本章内的 `预期成果` 是项目建设方案的成果摘要。若用户另有独立“主要技术成果/预期技术成果”章节，详细成果仍由 `official-doc-technical-achievements` 负责；本章只写成果类型、数量、分布和与任务的对应关系。

### 与 `official-doc-technical-indicators`

本章内的建设目标可列技术指标和产业化指标，但完整指标定义、测试条件和验收方法仍由 `official-doc-technical-indicators` 负责。没有依据的精确指标必须保留 `【待补】` 或降级为保守表述。

不要把完整项目建设方案正文直接输出到终端。

## 输出检查

- 开源策略是否写出范围、协作、质量、风险和闭环
