# 正式公文深度写作规则

本文件用于解决“材料搜到了，但正文仍然浅”的问题。正式项目公文的深度不来自术语堆砌，而来自清晰的论证链、可追溯依据、工程机制和可验证输出。

配套示例见 [depth-writing-examples.md](./depth-writing-examples.md)。写作时优先学习示例的论证方式，不要照抄示例主题、项目名称或指标数值。

## 一、深度写作的基本定义

正文不能只回答“有什么”，还要回答：
- 为什么现在需要做。
- 现有状态的差距在哪里。
- 差距产生的原因是什么。
- 本项目准备用什么技术机制解决。
- 预期形成什么可交付输出。
- 如何证明成果有效、可验收、可推广。

如果一段文字只是在重复“建设平台、提升能力、形成体系、赋能业务”，但没有写清对象、场景、机制、输出和验证方式，默认视为浅写法。

## 二、写作前必须建立“章节论证链”

两类共性章节正式起草前，先从 `research-plan.md`、`research-evidence.md`、`research-notes.md`、`facts-ledger.md` 和 `00-section-plan.md` 中抽取本章论证链。论证链至少包含：
- 核心判断：本章要证明什么。
- 事实依据：哪些已核验事实可以支撑。
- 差距或问题：现有做法哪里不足。
- 技术机制：项目采用什么方法、模型、流程或工具链。
- 输出结果：形成什么成果、能力、指标或交付件。
- 边界条件：哪些内容只能保守写，哪些仍需 `【待补】`。

若上述六项中少于四项能被填实，不得直接扩写正文。应回到 `official-doc-research` 补调研，或按“信息不足反馈规则”向用户问一个关键问题。

写正文前，必须先做一次“证据抽取”，而不是边写边猜。最低要求如下：
- 背景类小节：先列出 3 个以上可直接消费的证据点。完整背景章还要按 `建设背景 / 建设意义 / 国内外发展现状及前景` 分配证据，不能只把全部来源堆到“背景”一节。
- 项目建设方案：每个待写条目先列出 2 个以上可直接消费的证据点，并按 `目标 / 问题 / 任务 / 路线 / 输出 / 应用或保障` 分配证据，不能只为“研发内容”堆材料。
- 证据点必须能区分为：`事实/现状`、`差距/比较`、`机制/输出/验证` 中的至少两类。

若只能列出来源标题，列不出可进入正文的证据点，视为调研未转化，不能直接写稿。

## 三、段落信息密度

正式段落通常应包含三个以上“信息原子”：
- 对象：围绕哪个系统、设备、工艺、流程、数据或场景。
- 问题：当前瓶颈、风险、低效或不确定性是什么。
- 原因：为什么现有方式不能充分解决。
- 方法：采用什么技术路线、组织方式或工程措施。
- 机制：方法为什么能起作用。
- 输出：形成什么模块、模型、报告、规范、平台或数据资源。
- 验证：通过什么测试、指标、场景或交付件证明有效。
- 约束：安全、合规、标准、兼容、成本、周期等边界。

单段只写“项目将提升效率、优化流程、增强能力”而没有对象和机制，默认不合格。

## 三点五、写得太满时如何收口

正式公文不是把价值判断写满一页，而是把论证写实。出现以下迹象时，说明段落已经过满：
- 连续出现多个大词：`战略支撑、技术赋能、生态构建、能力提升、产业带动、自主可控`
- 同一段反复表达“意义重大、作用显著、价值突出”，但没有新增事实
- 只写“将形成...能力”，不写形成路径、成果形态和验证方式

收口方法：
- 删除重复的价值判断，保留第一次出现且最贴近章节主题的一处。
- 把抽象价值词替换为具体对象和动作，例如把“提升行业能力”改成“支撑电气原理图自动生成与校核”。
- 把“意义段”压缩为结尾收束句，不要让它占满主体段落。
- 如果一个判断不能回指 `research-evidence.md`、`research-notes.md` 或 `facts-ledger.md` 中的证据点，优先删掉或降级。

## 四、信息不足时如何处理

不要用泛泛的“请提供更多资料”打断用户。只在缺口会改变写作方向时提问，且一次只问一个问题。

应提问的情况：
- 不确定最终文体，可能导致章节口径不同。
- 不确定项目技术对象或应用场景，可能导致项目建设方案的目标、任务和技术路线写偏。
- 不确定必须写入的硬指标、周期、单位、验收口径。
- 用户要求写具体数字、政策文号、预算、负责人，但台账没有依据。

不必提问的情况：
- 只是缺少一个非关键数字，可先写保守表述并标 `【待补】`。
- 只是某个来源没有达到强断言标准，可降级为趋势性表述。
- 用户没有指定子标题，但 `00-section-plan.md` 已能推导出合理结构。

反馈格式应具体：

```md
当前缺口：缺少项目适用船型和试点单位。
影响：会影响应用场景、指标阈值和成果验收口径。
需要补充：请确认本项目优先面向新造船设计、在役船改造，还是两者兼顾。
可先保守处理：若暂不确认，正文将按“船舶设计与分析通用场景”写，并在指标处保留 `【待补】`。
```

## 五、浅写法禁区

以下表达不是绝对禁用，但若没有对象、机制和依据支撑，默认需要改写：
- `提升效率`
- `降低成本`
- `优化流程`
- `赋能业务`
- `打造平台`
- `构建体系`
- `形成闭环`
- `达到先进水平`
- `实现智能化`
- `推动高质量发展`

错误写法：

```md
项目将打造智能化平台，赋能船舶设计业务，显著提升设计效率，形成完整闭环。
```

较好写法：

```md
项目拟面向船舶设计过程中规范查询、方案校核和报告编制效率不足的问题，构建由知识抽取、规则推理、模板生成和结果校核组成的自动化处理链路。该链路以船级社规范、企业历史案例和标准报告模板为输入，形成可复用的规则库和报告生成工具，使设计人员能够在统一平台内完成资料检索、计算结果整理和合规性复核，从而减少人工转录和重复校对环节。
```

## 六、两类章节的深度门槛

### 项目背景

至少形成以下链路：

`建设背景 -> 建设意义 -> 国内外发展现状及前景`。

完整背景章默认展开为：
- `建设背景`：政策/战略/产业地位 -> 外部环境或行业压力 -> 技术演进 -> 本领域现实短板 -> 本项目切入。
- `建设意义`：总述段 -> 技术层面 -> 产业层面 -> 生态层面。
- `国内外发展现状及前景`：国内发展现状 -> 国外发展与应用现状 -> 痛点分析 -> 发展前景。

不得只写“国外先进、国内不足”。必须按技术路线、工程成熟度、应用场景、标准合规、数据积累等维度比较。

`建设意义` 不得写成“意义重大”的单段价值判断。技术层面要写清项目解决的技术体系、数据模型、接口集成、流程协同或安全保障问题；产业层面要写清对国产替代、成本、供应链安全、产业链协同和市场响应的作用路径；生态层面要写清对国产软件协同、人才培养、技术积累、开放接口、产学研用协同或标杆应用的支撑。

`痛点分析` 不得只列“效率低、成本高、风险大”。每个痛点至少写出问题表现、发生场景、影响后果和本项目切入方式。`发展前景` 不得写成空泛展望，每个前景判断都要说明由什么技术或平台能力承接，并如何作用于产业链、业务流程或数据资产。

### 项目建设方案

完整项目建设方案章至少形成以下链路：

`总体目标 -> 项目建设方案`。

默认展开为：
- `总体目标`：建设目标 -> 项目解决的主要问题 -> 项目研发内容 -> 预期成果 -> 产业链供应链韧性及安全保障。
- `项目建设方案`：技术路线 -> 应用推广方案 -> 产学研用合作方式 -> 攻关成果开源策略。

`建设目标` 不得只写“建设平台、提升能力”。应按外部约束、建设对象、自主替代或能力提升、工程应用和生态支撑的顺序写清目标链。

`项目解决的主要问题` 应采用“总述段 + 问题标题 + 解释段”的写法，每个问题至少写清问题表现、产生原因、影响后果和本项目拟解决的方向。

`项目研发内容` 不得写成任务名称清单。每项任务至少写清：
`问题来源 -> 建设任务 -> 技术或工程动作 -> 输入输出 -> 与其他任务的关系 -> 验证方式`。

`技术路线` 不得只画概念图或写“采用人工智能、大数据等技术”。应写清总体技术方案、架构分层、任务依赖、数据或业务流转关系、测试验证和工程应用闭环。

`应用推广方案` 要写清应用对象、推广路径、场景验证、反馈迭代和预期效果；`产学研用合作方式` 要写清各方角色、任务接口、协同机制和成果转化闭环；`攻关成果开源策略` 要写清开源范围、开源方式、质量控制、风险边界和持续迭代机制。

## 七、写作后的自检

每个章节写完后至少自查：
- 是否有论证链，而不是材料堆砌。
- 是否有现状差距和原因分析，而不只是背景介绍。
- 是否写出了技术机制，而不只是功能愿望。
- 是否能从项目研发内容和建设任务推到预期成果、推广应用或保障机制。
- 是否仍有口号式短句、项目符号正文、加粗小标签。
- 是否对未核验信息降级处理或标记 `【待补】`。
- 是否先完成了证据抽取，并在段落中真正消费这些证据，而不是只读过台账。
- 是否有整段被“意义、价值、作用、支撑”占满，而缺少对象、机制、输出和验证。
