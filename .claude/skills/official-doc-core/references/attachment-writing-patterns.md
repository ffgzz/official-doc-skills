# 公文结构与句式规则

## 一、文档级编号方案

编号必须由 `using-official-docs` 在初始化阶段统一确定，并写入 `project-brief.md` 与 `workspace/outputs/<project-slug>/00-section-plan.md`。两个专项章节 skill 只执行该方案，不得各自临场选择编号。

默认按文体选择：

- 可研报告、立项申请书、项目建议书、技术总结：采用正式报告型编号。一级章用 `一、`，二级节用 `（一）`，三级项用 `1.`，四级点用 `（1）`。
- 项目指南、攻关任务书、任务清单型材料：采用指南任务型编号。一级任务用 `1.`，二级要求用 `（1）`，三级动作或技术点用 `（a）`。
- 项目建设方案中的项目研发内容需要深度拆成任务/子任务时：章级仍服从全文编号，在“项目研发内容/建设任务”内部可采用 `任务1`、`子任务1-1`、`（1）` 的任务分解编号。

硬规则：

- 同一层级不得混用 `（1）` 和 `1.1`。
- 同一章节内若已采用 `任务1 / 子任务1-1`，则不要再把同级条目写成另一套编号。
- 编号只用于标题层级和必要的表格/指标项，不得把正文写成编号清单。
- 标题下必须有完整自然段展开，不能只有标题和一句功能愿望。
- 若用户或附件模板已经固定编号，优先服从模板；若未固定，按上述默认规则选型，并在计划文件中留痕。
- 扩写章节时不得靠连续追加 `### 11.`、`### 12.`、`### 13.` 这类流水号标题增加篇幅。正式可研/立项/项目建议书的主体章节应优先在既定二级节内用自然段、表格和少量必要三级标题展开。
- 普通章节不得出现 `### 一、...`、`### 二、...` 这类把一级编号降级使用的标题。正式报告型正文中，三级标题用 `1.`，四级标题用 `（1）`；项目研发内容内部的 `任务1 / 子任务1-1` 是唯一例外。
- 正式报告型正文中，三级标题必须写成 `1.`、`2.`、`3.`，不得写成 `1 `、`2 ` 这类无点号标题。
- 正式报告型正文中，四级标题必须在各自父级小节内重新起号，写成 `（1）`、`（2）`、`（3）`。不得把四级编号跨整个章节或全文连续累计成 `（9）`、`（10）`、`（21）`、`（33）`。
- 若项目研发内容内部已经使用 `任务一 / 任务二`、`任务1 / 任务2` 作为任务包层级，则每个任务内部的四级条目应重新从 `（1）` 起号，或改为 `子任务1-1` 体系；不得一边使用任务化标题，一边继续沿用全章累计的 `（14）`、`（19）` 等编号。
- 同一章节不得出现重复标题。若需要补充相近内容，应合并到原有标题下，不得另起同名或近似同名小节。

章节计划中必须同时记录“用户章节标记”和“正式输出标题”。例如用户提示词写“第一章 概述”，但目标文体为可行性报告且编号方案为正式报告型时，正式输出标题应写成 `一、概述`，不得在正式稿中输出 `第一章 概述`。`第一章/第二章` 只能作为用户原始标记或内部排序信息，不能覆盖 `一、/二、` 的正式输出编号。

正式稿不得新增 `00-section-plan.md` 中没有登记的二级节。若用户只要求第四章包含 `（一）总体目标`、`（二）项目建设方案`，不得为了补字数追加 `（三）研究方案可行性分析`、`（四）工作基础` 等计划外小节。字数不足时，只能在既定小节内加深“目标、问题、任务、路线、输出、应用或保障”和表图说明，不能靠增设新节凑篇幅。

三级、四级标题也要服从当前层级方案。正式报告型正文中，三级标题用 `1.`、四级标题用 `（1）`；只有项目研发内容内部的任务化拆解可以使用 `子任务1-1`。在实施方案、风险等普通章节中，不得出现 `11.1`、`13.2` 这类十进制目录标题。

## 二、附件式写法硬门禁

本文件中的“常用写法”是两个正文 skill 的硬约束，不是可选参考。凡涉及 `official-doc-project-background`、`official-doc-research-content` 的正文段落，写完后必须自检是否满足以下要求；不满足时先回修，再交给 review 或 assemble。

背景类关键段落至少覆盖以下 6 个要素中的 4 个：

`对象 / 现状 / 问题 / 影响 / 项目切入 / 能力或前景`

项目建设方案类关键段落至少覆盖以下 6 个要素中的 4 个：

`目标 / 问题 / 任务 / 路线 / 输出 / 应用或保障`

重章节原则上应覆盖 5 个以上要素。若段落只能看出“做什么”，看不出“为什么做、怎么做、形成什么、如何验”，视为不合格。

两类正文的强制句式骨架如下：

- 背景与必要性：完整背景章默认按 `建设背景 -> 建设意义 -> 国内外发展现状及前景` 展开；建设背景写 `政策/战略/产业形势 -> 外部环境或行业压力 -> 技术趋势与现实短板 -> 本项目切入`，建设意义写 `总体意义 -> 技术层面 -> 产业层面 -> 生态层面`，国内外发展现状及前景写 `国内发展现状 -> 国外发展与应用现状 -> 痛点分析 -> 发展前景`。
- 项目建设方案：完整章默认按 `总体目标 -> 项目建设方案` 展开；总体目标写 `建设目标 -> 项目解决的主要问题 -> 项目研发内容 -> 预期成果 -> 产业链供应链韧性及安全保障`，项目建设方案写 `技术路线 -> 应用推广方案 -> 产学研用合作方式 -> 攻关成果开源策略`。

禁止把上述骨架压缩成口号式短句。每个正文 skill 可以根据章节实际调整措辞，但背景类段落不能删掉“现状、问题、影响、项目切入”，项目建设方案段落不能删掉“目标、问题、任务、路线、输出、应用或保障”的核心信息。

扩写规则：

- 字数不足时，优先补充事实依据、差距原因、技术机制、输出边界、测试验证和应用口径。
- 不得新增 `00-section-plan.md` 没有登记的二级节。
- 不得新增一组连续流水号三级标题来替代正文扩写。
- 不得追加 `历史意义`、`时代价值`、`远景展望`、`政策建议`、`社会责任`、`一带一路`、`国际合作机遇`、`可持续发展规划`、`人才培养` 等与用户章节计划无关的泛化标题。
- 需要组织长内容时，优先使用“总述段 + 表格 + 表后说明 + 少量必要三级标题 + 自然段”的附件式结构。

### 调研证据消费硬门禁

两个正文 skill 不仅要“读取”调研台账，还必须把可用证据转化为正文中的论证链。只调研不消费，视为写作未完成。

写每个二级节前，必须先从 `research-evidence.md`、`research-notes.md`、`facts-ledger.md` 中抽取本节证据清单，至少整理出：

- `已核验事实`：可以直接进入正文的事实、口径、趋势判断或边界条件。
- `比较/差距证据`：可以支撑“现状不足、传统路线局限、同类方案差异”的证据点。
- `机制/输出/验证证据`：可以支撑“怎么做、形成什么、如何验证”的证据点。

消费规则：

- 背景与必要性：每个二级节至少消费 3 个证据点。`建设背景` 至少覆盖政策/官方、行业或技术趋势、现实短板；`建设意义` 至少覆盖技术、产业、生态中的 2 类支撑依据；`国内外发展现状及前景` 至少覆盖国内现状、国外现状或代表性产品、痛点或差距、发展前景中的 3 类依据。
- 项目建设方案：每个三级项至少消费 2 个证据点。`建设目标` 至少覆盖目标依据和指标/应用范围依据；`项目解决的主要问题` 至少覆盖背景痛点和解决路径依据；`项目研发内容` 至少覆盖任务拆分、技术机制、输入输出或验证依据；`技术路线` 至少覆盖总体架构、任务链路、测试验证或工程应用依据；`应用推广方案 / 产学研用合作方式 / 攻关成果开源策略` 至少覆盖对象、路径、机制、风险防范或效果依据。

消费方式要求：

- 正文不得直接裸露 `F-001`、`RC-S01` 之类内部编号，但句子内容必须能回溯到这些证据卡。
- 正文、图示说明稿和正式总稿中同样不得出现 `【来源：...】`、`来源：原始资料`、`BG-1-01`、`RC-1-01` 这类内部来源标记；这些编号只属于台账，不属于正式公文。
- `检索与资料映射说明`、搜索执行概况、来源分组与章节映射等内部过程说明不得写入章节正文，不得作为补字数段落，也不得进入正式总稿。
- 若某节只有宏观判断、价值判断或任务愿望，而看不到具体事实、差距、机制或验收口径，说明证据未真正消费，应先回修。
- 若某节缺少足够证据点，不得硬写，应回到 `official-doc-research` 补调研，再继续正文写作。

### 文风收口硬门禁

附件式公文要求“克制、具体、可辩护”，不是把价值词堆满。

出现以下情况时，应视为“写得泛、写得满”，必须回修：

- 一段里连续堆叠 `产业链安全 / 技术赋能 / 核心能力建设 / 生态体系构建 / 高质量发展 / 战略支撑` 等价值词，但没有对应对象、机制和输出。
- 一段里只有“意义、价值、作用、支撑”，却看不到具体场景问题、实施动作或验收边界。
- 为补字数反复改写同一个判断，只是换同义词重复“提升、优化、增强、支撑、保障”。

收口规则：

- 背景类段落优先保留 `对象、现状、问题、影响、项目切入`，项目建设方案段落优先保留 `目标、问题、任务、路线、输出、应用或保障`，其他正文段落优先保留 `对象、问题、机制、输出、验证`，压缩纯价值判断。
- 每个二级节的“意义/价值”段原则上不超过该节正文的三分之一。
- 遇到“大词连写”时，至少落到一个具体对象和一个具体动作，例如“支撑船舶电气原理图生成”“沉淀规则库与模板库”“通过样例船型验证”。
- 能写成“解决什么问题、形成什么能力、以什么形态交付”的，不写成“意义重大、作用显著、价值突出”。

## 三、正文骨架

正式公文的常用骨架不是“概念堆叠”，而是把对象、问题、研究动作和效果串成闭环。

### 1. 背景与必要性

推荐链路：

`建设背景 -> 建设意义 -> 国内外发展现状及前景`

完整背景章默认结构：

```md
一、项目背景及必要性
（一）建设背景
（二）建设意义
1. 技术层面
2. 产业层面
3. 生态层面
（三）国内外发展现状及前景
1. 国内发展现状
2. 国外发展与应用现状
3. 痛点分析
4. 发展前景
```

`建设背景` 的内部链路为：

`政策/战略/产业地位 -> 国际环境或行业压力 -> 技术演进 -> 本领域现实短板 -> 本项目切入`

`建设意义` 的内部链路为：

`总述段 -> 技术层面 -> 产业层面 -> 生态层面`

`国内外发展现状及前景` 的内部链路为：

`国内基础和短板 -> 国外产品/应用/趋势 -> 痛点表现和影响 -> 发展前景与本项目承接`

常用写法：

- `以...为代表的...是...的重要组成部分，也是...的重要支撑。`
- `在...新阶段，随着...在...领域的渗透，...已成为提升...水平的关键因素。`
- `然而，当前...仍存在...问题，特别是在...方面尚难满足...需求。`
- `在技术层面，...是实现...的重要支撑，项目将通过...建立...。`
- `在产业层面，...有助于破解...问题，降低...依赖，提升...能力。`
- `在生态层面，...将带动...协同，沉淀...能力，推动形成...生态。`
- `从国内发展现状看，...已具备...基础，但在...方面仍存在不足。`
- `从国外发展与应用现状看，...已在...方面形成较成熟能力，其优势主要体现在...。`
- `上述痛点集中表现为...，直接影响...，亟须通过...予以解决。`

意义类内容宜按 `技术层面 / 产业层面 / 生态层面` 组织，每个层面都要写“解决什么问题、形成什么能力、带动什么协同”，不要只写“意义重大”。若用户另有章节安排，可合并层面标题，但不得丢掉三类论证。

### 2. 项目建设方案

推荐链路：

`总体目标 -> 项目建设方案`

完整项目建设方案章默认结构：

```md
四、项目建设方案
（一）总体目标
1. 建设目标
2. 项目解决的主要问题
3. 项目研发内容
4. 预期成果
5. 产业链供应链韧性及安全保障
（二）项目建设方案
1. 技术路线
2. 应用推广方案
3. 产学研用合作方式
4. 攻关成果开源策略
```

`总体目标` 的内部链路为：

`建设目标 -> 项目解决的主要问题 -> 项目研发内容 -> 预期成果 -> 产业链供应链韧性及安全保障`

`项目建设方案` 的内部链路为：

`总体技术路线 -> 任务建设路线 -> 应用推广路径 -> 产学研用协同 -> 开源策略闭环`

推荐句式：

- `本项目依照...要求，制定项目目标，满足...指标要求。`
- `为破除...依赖，降低...风险，项目拟联合...力量，打造...。`
- `项目针对...挑战，重点研究并解决...问题。`
- `基于项目总体目标，遵循...思想，从...角度出发，将项目研发内容分解为...等任务。`
- `针对...特点和...问题，开展...建设，形成...，为...提供支撑。`
- `在充分分析...功能需求的基础上，结合...技术及验证手段，将总体技术架构分为...。`
- `按照...实际研发及工程应用流程，项目技术路线见下图。`
- `首先在...内部推广应用项目成果，并将其应用于...；在此基础上，依托...开展规模化推广。`
- `联合体将围绕...，形成“产、学、研、用”协同机制。`
- `开源策略应围绕开源范围、协作模式、质量保障、风险防范和闭环管理展开。`

## 四、常见不合格形态

出现以下情况时应回修：

- 各章标题层级自己选择编号，导致同一份稿件混用 `（1）`、`1.1`、`1.1.1`。
- 用大量连续的三级、四级标题堆出篇幅，正文没有在既定小节内形成连续论证。
- 项目建设方案只有目标和任务名，没有“建设目标、主要问题、研发任务、技术路线、应用推广、协同合作、开源策略”闭环。
- 技术路线写成“采用AI技术、实现智能化”这类空话，没有架构分层、任务依赖、测试验证和工程应用。
