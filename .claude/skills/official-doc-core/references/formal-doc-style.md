# 正式公文文风参考

本文件依据以下材料归纳：
- `完整科研项目模板.docx`
- `产业链项目指南.docx`
- `产业链项目攻关任务书.docx`
- `ZS-项目可行性报告.docx`
- `ZS项目立项申请书.docx`
- `research-writing-skill` 中关于去 AI 化和风格检查的约束
- `nsfc-agent-skills` 中关于正式申报写作的限制

## 一、正文形态

正式公文正文默认采用连续自然段，不采用演示稿式项目符号堆叠。

允许使用列表的场景：
- `plan/` 台账
- 检查清单
- 表格字段说明
- 操作步骤

不建议在正文中出现：
- `- ...`
- `1. ... 2. ... 3. ...` 的大段列表
- `**技术描述**：...`
- `**技术难点**：...`
- `**作用**：...`

## 二、典型行文方式

从公文样本看，常见的有效写法不是“逐点罗列”，而是“分层展开”。

常见展开方式：
- 先交代大背景，再收束到本项目
- 先说明问题，再说明影响，再提出方案
- 先给总体判断，再展开具体环节
- 背景章常用“建设背景 -> 建设意义 -> 国内外发展现状及前景”的层级展开，其中建设意义进一步按“技术层面、产业层面、生态层面”组织，现状前景进一步按“国内发展现状、国外发展与应用现状、痛点分析、发展前景”组织
- 项目建设方案章常用“总体目标 -> 项目建设方案”的层级展开，其中总体目标进一步按“建设目标、项目解决的主要问题、项目研发内容、预期成果、产业链供应链韧性及安全保障”组织，项目建设方案进一步按“技术路线、应用推广方案、产学研用合作方式、攻关成果开源策略”组织

常见句式：
- `当前，……`
- `在……层面，……`
- `围绕……需求，……`
- `针对……问题，……`
- `项目拟通过……，实现……，形成……`
- `项目建设内容拟按照“……、……、……”的主线组织。`
- `该任务重点解决……问题，形成……输出，并支撑……环节。`
- `技术路线拟采用“……层、……层、……层”的分层架构。`
- `应用推广宜按照“内部试点、示范验证、规模推广、反馈迭代”的路径推进。`
- `以……为代表的……是……的重要组成部分，也是……的重要支撑。`
- `然而，当前……仍存在……问题，特别是在……方面尚难满足……需求。`
- `面对……外部环境和……现实需求，亟须……。`

## 三、禁止的 AI 痕迹

正文默认避免以下痕迹：
- 机械过渡词：`首先、其次、最后、此外、另外、接下来、总之`
- 空壳强调句：`值得注意的是、需要指出的是、重要的是、必须强调的是`
- PPT 化粗标签：`**技术描述**`、`**技术难点**`、`**作用**`
- 目录式堆砌：连续出现 `1.1 / 1.1.1 / 1.1.2` 而没有实质段落展开

## 四、正确与错误示例

### 示例 1：列表化正文

错误：

```md
本项目建设内容如下：
- 建设数据底座
- 建设规则知识库
- 建设报告生成系统
```

正确：

```md
本项目拟围绕数据底座建设、规则知识库建设和报告生成系统建设三个环节展开。数据底座重点解决源数据格式不统一、版本关系不清晰的问题，形成可供后续调用的标准化输入；规则知识库重点沉淀规范条款、报告模板和结论语句之间的映射关系，为自动生成和一致性校核提供约束；报告生成系统则完成模块集成、人工复核和场景验证，形成可部署、可迭代的工程化解决方案。
```

### 示例 2：加粗小标签

错误：

```md
**技术描述**：针对样本不足问题，研究自动化训练样本生成方法。

**技术难点**：需要保证生成样本与工程场景一致。

**作用**：为后续模型训练提供支撑。
```

正确：

```md
针对训练样本不足的问题，项目拟研究自动化训练样本生成方法，通过参数化模板和批处理机制提升样本构建效率。该技术的关键在于保持生成样本与实际工程场景的一致性，避免样本分布偏移。相关成果将为后续模型训练提供稳定的数据基础，并支撑分类精度的持续提升。
```

### 示例 3：目录式机械下钻

错误：

```md
1.1 船舶配电知识建模
1.1.1 规范抽取
1.1.2 规则建模
1.1.3 策略建模
```

正确：

```md
围绕船舶配电知识建模，项目将从规范抽取、规则建模和供电策略表达三个层面展开。首先对船级社规范和企业设计规则进行结构化抽取，形成可计算的规范底座；在此基础上建立负荷类别、舱室属性和供电模式之间的规则关系；进一步结合典型设计场景构建供电策略模型，为后续配电拓扑自动生成提供推理依据。
```

## 五、段落长度与结构

单段通常应包含：
- 主题句
- 支撑句
- 收束句

中文正式正文的常见段长可控制在 120 至 280 字左右。背景章、国内外现状和痛点分析因需要同时交代政策、行业状态、代表性对象和项目切入，单段可适度放宽到 180 至 420 字，但每段必须有明确主题、事实支撑和收束点。过短易显碎片化，过长且没有递进关系则易显材料堆砌。

背景章允许在自然段后使用少量编号条目介绍功能范围、痛点类型或发展前景，但编号条目必须服务于正文论证。不要把全文写成演示稿清单，也不要把“技术层面、产业层面、生态层面”下面继续拆成空泛口号。

## 六、章节内部编号

如用户明确要求保留“第一章、第二章”或“1.1、1.2”之类结构，应保留章节层级。若用户未指定，按 [attachment-writing-patterns.md](./attachment-writing-patterns.md) 中的文档级编号方案执行。

但在层级标题之下，正文仍应写成自然段，不要把小任务继续拆成项目符号列表。

同一份稿件必须保持同层级编号一致。正式报告型文本默认采用 `一、`、`（一）`、`1.`、`（1）`；项目建设方案中的项目研发内容需要任务化拆解时，可以在该节内部采用 `任务1`、`子任务1-1` 等写法，但不能把同一层级同时写成 `（1）` 和 `1.1`。

## 七、总原则

正式公文不是 PPT，也不是技术说明书碎片。

当一段文字看起来像：
- 演示稿备注
- 评审答辩提纲
- AI 自动补全的格式化片段

就说明需要回到“连续自然段 + 论证展开”的公文写法。
