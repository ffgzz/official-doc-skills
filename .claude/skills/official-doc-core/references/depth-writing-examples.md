# 正式公文深度写作示例

本文件只提供写法样例，不是固定主题模板。使用时应学习其论证方式：先交代对象和问题，再说明原因、机制、输出和验证，不要照抄示例中的项目名称、技术对象或指标数值。

## 一、项目背景示例

### 示例 1：建设背景

浅写法：

```md
随着人工智能技术快速发展，船舶行业也需要加快智能化转型。本项目建设具有重要意义，可以提升企业研发能力，推动产业高质量发展。
```

深写法：

```md
当前，船舶设计、结构分析和合规报告编制正在从单一软件工具支撑向多源数据协同、规范知识复用和自动化交付方向演进。传统工作模式下，设计人员需要在规范文件、计算结果、历史报告和工程经验之间频繁切换，报告编制环节仍依赖人工摘录、复核和排版，容易造成数据口径不一致、重复劳动较多和交付周期拉长等问题。围绕这一场景开展智能化报告生成能力建设，不仅是提高单项文档编制效率的问题，更是将计算结果、规范约束和企业知识沉淀为可复用工程资产的需要。
```

为什么更好：
- 有具体业务场景，不停留在“行业智能化”。
- 写出了传统模式的问题表现和原因。
- 把项目必要性落到数据、规范、知识沉淀和交付链路。

### 示例 2：国内外现状

浅写法：

```md
国外在工业软件和人工智能方面起步较早，技术比较先进。国内近年来发展很快，但整体水平仍有差距，需要通过本项目进一步追赶。
```

深写法：

```md
从公开资料看，国外工业软件厂商已开始将仿真数据管理、自动化后处理和报告模板化能力嵌入工程软件生态，相关能力更强调与既有工具链、企业数据管理平台和标准化交付流程的集成。国内在船舶设计软件、结构计算工具和工程数据治理方面已有一定积累，但面向船体结构分析报告的自动生成能力仍多依赖项目组经验、脚本二次开发和人工校核，尚未充分形成从计算结果读取、规范条款映射、图表生成、文字说明到合规复核的一体化闭环。差距的核心不在于单点算法是否可用，而在于行业知识、报告模板、数据接口和校核规则能否被统一组织并稳定服务于工程交付。
```

为什么更好：
- 国内外不是简单强弱对比，而是按工具链、数据管理、模板化、合规交付比较。
- 差距原因不是“技术落后”，而是闭环能力不足。
- 后文自然能引出研究内容。

### 示例 3：痛点和必要性

浅写法：

```md
目前报告编写效率低，人工成本高，因此有必要建设自动生成系统。
```

深写法：

```md
船体结构疲劳分析报告通常涉及模型信息、载荷工况、计算结果、云图文件、规范条款和结论说明等多类内容。现有流程中，计算结果与报告文本之间缺少稳定映射，工程人员需要反复核对文件版本、复制关键数值、整理图表并补写判断说明。一旦模型版本、计算参数或审查意见发生变化，报告往往需要重新校对，既影响交付效率，也增加了错填、漏填和口径不一致的风险。因此，本项目有必要围绕结构化数据提取、报告模板引擎、规范化语句生成和结果一致性校核开展攻关，使报告生成从经验驱动的人工编写转向规则约束下的自动化生成和可追溯复核。
```

为什么更好：
- 写清了报告包含哪些对象。
- 写清了低效的机制性原因。
- 必要性直接转化为后续攻关方向。

## 二、研究内容示例

### 示例 1：研究内容总述

浅写法：

```md
本项目主要研究数据处理、模型训练、系统开发和应用验证，形成船体结构疲劳分析报告自动生成系统。
```

深写法：

```md
本项目围绕船体结构疲劳分析报告自动生成这一目标，拟按照“数据结构化、知识规则化、内容生成化、校核自动化、系统工程化”的主线组织研究内容。前端重点解决有限元计算结果、结构图表和历史报告等多源数据的统一读取与标识问题；中间环节建立规范条款、报告章节、图表模板和结论语句之间的映射关系；后端通过报告生成引擎和一致性校核机制完成自动编制、人工复核和版本追溯。各项研究内容之间形成从输入治理到报告交付的连续链路，而不是相互独立的功能堆叠。
```

为什么更好：
- 有技术主线。
- 解释了各环节关系。
- 明确不是功能清单，而是输入到交付的链路。

### 示例 2：单项研究内容

浅写法：

```md
研究数据处理技术，对有限元结果进行清洗和标准化，为后续模型训练提供数据基础。
```

深写法：

```md
针对有限元结果文件、应力云图、工况参数和报告文本之间格式不统一、语义关系不清晰的问题，项目拟开展船体结构疲劳分析数据结构化处理技术研究。该部分将建立面向报告生成的字段体系和数据映射规则，对模型编号、结构部位、载荷工况、热点应力、疲劳寿命和图表文件进行统一标识，并研究异常值识别、版本匹配和缺失项提示方法。通过该研究形成标准化数据接口和结构化结果库，为后续章节填充、图表调用和结论生成提供稳定输入。
```

为什么更好：
- 明确研究对象和问题来源。
- 写出方法机制：字段体系、映射规则、异常识别、版本匹配。
- 写出输出：接口和结果库。

### 示例 3：关键技术

浅写法：

```md
关键技术是报告自动生成技术，可以自动生成报告，提高效率。
```

深写法：

```md
报告自动生成的关键不在于简单替换模板字段，而在于建立计算结果、章节语义和审查口径之间的稳定映射关系。项目拟研究约束驱动的报告生成技术，将结构化计算结果与报告模板中的章节、表格、图示和结论语句进行绑定，并通过规则校核机制检查关键数值、图表编号和结论表述的一致性。该技术位于数据治理和最终交付之间，承担从“计算结果可读”到“报告内容可审”的转换作用，是实现工程化应用的核心环节。
```

为什么更好：
- 解释了为什么难。
- 写出技术机制和链路位置。
- 将“效率提升”转为“可审、可交付”的工程价值。

## 三、创新点示例

### 示例 1：有比较基线的创新点

浅写法：

```md
本项目创新性地构建智能报告生成平台，实现报告自动生成，具有明显创新性。
```

深写法：

```md
区别于传统基于固定模板的字段替换方式，本项目拟构建面向船体结构疲劳分析场景的规则约束型报告生成机制。该机制不是单纯将计算结果填入模板，而是将结构部位、载荷工况、疲劳寿命、云图文件和规范判断语句统一纳入章节语义模型，通过规则库约束生成内容的取值来源、表达口径和校核关系。由此，报告生成过程从“文本拼接”转向“数据、图表、结论同步受控”的工程交付过程，可降低因人工转录和模板复用不当造成的口径偏差。
```

为什么更好：
- 明确比较对象：固定模板字段替换。
- 写出新机制：章节语义模型和规则库约束。
- 创新落到工程价值：降低口径偏差。

### 示例 2：从研究内容提升为创新点

研究内容式写法：

```md
研究疲劳分析知识库构建技术，形成报告生成所需的知识库。
```

创新点写法：

```md
在知识组织方式上，项目拟将规范条款、历史报告、典型结构部位和计算结果字段统一组织为面向报告生成的疲劳分析知识库。相较于传统以文件夹、人工经验和单份报告模板为主的知识复用方式，该知识库强调规则条款与报告章节、图表类型和结论语句之间的可计算关联，使工程经验能够从个人经验沉淀转为可检索、可调用、可校核的组织级知识资产。
```

为什么更好：
- 不是只说“构建知识库”，而是说明组织方式变化。
- 有旧方式和新方式对比。
- 体现工程知识资产化价值。

### 示例 3：不能强写“首创”

不建议写法：

```md
本项目首次提出船舶报告自动生成技术，达到国内领先水平。
```

稳健写法：

```md
在公开资料尚未充分显示同类能力已形成成熟工程闭环的情况下，本项目将重点探索船体结构疲劳分析报告从数据读取、图表调用、章节生成到一致性校核的端到端实现路径。相关创新应表述为面向特定船舶工程场景的集成创新和流程创新，不宜在缺少充分主源支撑时使用“首次”“领先”等绝对化判断。
```

为什么更好：
- 避免无依据强词。
- 说明能写成什么创新，不能写成什么创新。

### 示例 4：写得太满时如何收口

过满写法：

```md
本项目的实施将全面提升船舶电气设计领域的数字化、智能化、自主化和协同化水平，显著增强产业链安全保障能力、核心技术支撑能力、工程交付能力和行业生态构建能力，为推动高质量发展、服务国家战略和促进产业升级提供坚实支撑。
```

收口后写法：

```md
本项目拟面向船舶电气设计过程中原理图绘制依赖人工、校核重复、知识难以复用的问题，构建集知识抽取、规则校核、模板生成和结果追溯于一体的智能设计支撑能力。项目实施后，预期形成电气原理图生成与校核工具、规则库和模板库，并通过典型船型样例验证其对设计效率和一致性复核的支撑作用。
```

为什么更好：
- 删除了重复价值词，保留了章节真正需要证明的内容。
- 把“支撑什么”落到具体对象、机制、成果和验证。
- 仍然保留正式公文语气，但不再空泛发散。

## 四、主要技术成果示例

### 示例 1：成果实体

浅写法：

```md
形成智能化报告生成能力，提升企业设计水平。
```

深写法：

```md
围绕报告自动生成研究内容，项目实施后预计形成船体结构疲劳分析报告自动生成系统。该系统以结构化计算结果库、报告模板库和规则校核库为基础，支持典型章节自动填充、图表批量调用、结论语句生成和报告版本管理，可作为项目验收的软件成果进行部署测试，并通过典型船型案例验证其生成完整性和结果一致性。
```

为什么更好：
- 成果是系统，不是抽象能力。
- 交代组成、功能边界和验收方式。

### 示例 2：成果组合

浅写法：

```md
项目将形成平台、模型、数据库和报告等多项成果。
```

深写法：

```md
项目成果可按软件工具、数据资源和文档规范三类组织。软件工具方面，形成报告生成原型系统和结果一致性校核模块，用于支撑报告自动编制和审查复核；数据资源方面，形成疲劳分析结构化结果库、典型章节模板库和规范语句规则库，为系统运行提供可复用底座；文档规范方面，形成系统设计说明、测试验证报告和报告编制规则说明，为后续推广应用和运维迭代提供依据。
```

为什么更好：
- 按成果类型分组。
- 每类成果都有用途。
- 不只是罗列名称。

### 示例 3：成果与研究内容映射

浅写法：

```md
根据研究内容，形成多项技术成果。
```

深写法：

```md
数据结构化处理研究对应形成标准化数据接口和结构化结果库，解决多源计算结果可读取、可追溯的问题；报告生成技术研究对应形成模板引擎和自动生成模块，支撑章节、图表和结论说明的自动编排；一致性校核研究对应形成规则校核模块和测试验证报告，用于检查关键数值、图号、表号和结论语句之间的匹配关系。上述成果共同构成从计算结果到正式报告的工程化交付链路。
```

为什么更好：
- 每项成果都能回指研究内容。
- 成果之间有链路关系。

## 五、主要技术指标示例

### 示例 1：指标有测试条件

浅写法：

```md
报告生成准确率达到95%以上，效率提升50%以上。
```

深写法：

```md
报告生成完整性指标用于衡量系统对规定章节、表格、图示和结论说明的自动覆盖能力。建议以不少于 `【待补】` 个典型疲劳分析案例作为测试集，检查系统是否能够按照模板要求生成完整报告内容，目标值可设定为核心章节覆盖率不低于 `【待补】`。对于尚缺少测试样本规模和人工基准效率的数据，应先保留待校准标记，待试点案例形成后再确定最终验收阈值。
```

为什么更好：
- 没有依据时不伪精确。
- 写清指标定义、测试集和校准方式。

### 示例 2：指标分组

浅写法：

```md
系统响应快、准确率高、稳定性好，满足应用要求。
```

深写法：

```md
技术指标宜按数据处理、报告生成、校核复核和系统运行四类设置。数据处理类指标重点考核典型结果文件的读取成功率、字段提取完整率和版本匹配准确性；报告生成类指标重点考核章节覆盖率、图表调用正确率和模板填充准确性；校核复核类指标重点考核关键数值一致性检查通过率和问题提示有效性；系统运行类指标重点考核单份报告生成时长、批量处理能力和连续运行稳定性。通过分类设置指标，可以避免只用单一准确率概括复杂工程系统能力。
```

为什么更好：
- 指标按对象和能力分组。
- 每类指标都有测量对象。

### 示例 3：伪精确改保守

不建议写法：

```md
系统报告生成时间控制在3分钟以内，准确率达到99.5%，人工工作量降低80%。
```

稳健写法：

```md
在缺少既有人工编制耗时、样本规模和测试环境基准的情况下，报告生成效率指标不宜直接给出过高精确值。可先将指标表述为“在典型报告模板和标准化输入条件下，单份报告生成时间较人工整理流程明显缩短，具体缩短比例以试点案例实测结果为准”，并在指标表中设置 `【待补：人工基准耗时】`、`【待补：测试样本数量】` 和 `【待补：目标阈值】`，待验证阶段补齐。
```

为什么更好：
- 避免看似漂亮但无依据的数字。
- 说明需要哪些数据才能定指标。

## 六、信息不足反馈示例

### 示例 1：缺应用场景

```md
当前缺口：用户 brief 未明确系统优先服务于新造船设计、在役船改造还是科研分析场景。
影响：不同场景会影响报告模板、验收指标和示范案例选择。
需要确认：本项目优先面向哪一类应用场景？
可先保守处理：若暂不确认，正文将按“船舶结构分析通用报告场景”撰写，并在成果验证和指标阈值处保留 `【待补】`。
```

### 示例 2：缺硬指标

```md
当前缺口：brief 要求写主要技术指标，但没有给出样本规模、测试环境和验收阈值。
影响：无法直接写精确准确率、生成时长和效率提升比例。
需要确认：是否已有指定验收指标，还是允许先按行业同类项目口径提出建议值？
可先保守处理：若没有指定值，将采用“建议指标 + 待项目验证阶段校准”的写法，并把具体阈值登记为 `【待补】`。
```

### 示例 3：来源不足但可以继续

```md
当前缺口：某项国外现状仅有行业媒体报道，缺少官方或论文原文支撑。
影响：不能写成确定性强表述，也不能使用“已形成成熟应用”等判断。
处理方式：正文将降级为“公开资料显示，相关方向已有探索”，并在 `facts-ledger.md` 中标记为 `仅作趋势参考`，同时把补充主源列为后续调研项。
```

## 七、从浅段落改深段落的方法

浅段落：

```md
项目将建设智能化平台，整合多源数据，提升报告编写效率，实现业务流程闭环。
```

改写步骤：
- 找对象：什么报告、什么数据、什么业务流程。
- 找问题：为什么现有流程效率低。
- 找机制：平台通过什么模块或规则解决。
- 找输出：形成什么系统、数据库、模板、报告。
- 找验证：通过什么案例或指标证明。

