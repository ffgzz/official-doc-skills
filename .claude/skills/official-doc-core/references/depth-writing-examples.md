# 正式公文深度写作示例

本文件只提供写法样例，不是固定主题模板。使用时应学习其论证方式：先交代对象和问题，再说明原因、机制、输出和验证，不要照抄示例中的项目名称、技术对象或指标数值。

## 一、项目背景示例

### 示例 1：建设背景

浅写法：

```md
随着人工智能技术快速发展，船舶行业也需要加快智能化转型。本项目建设具有重要意义，可以提升企业研发能力，推动产业高质量发展。
```

深写法：

```md
以船舶和海工装备为代表的高端装备制造业，是制造强国和海洋强国建设的重要支撑。随着新一代信息技术向研发设计、仿真分析、生产制造和运维服务等环节持续渗透，船舶工程软件正在由单点工具支撑向全生命周期数据贯通、规范知识复用和协同交付方向演进。然而，现有设计分析流程中仍存在多源数据分散、工具接口不统一、报告与计算结果缺少稳定映射等问题，导致工程人员需要在规范文件、计算结果、历史报告和人工经验之间频繁切换。面对复杂装备研制周期长、变更频繁、数据规模大和合规要求高的特点，有必要围绕工程数据结构化、规则化表达和交付一致性校核开展系统建设，使相关能力从个人经验驱动转向平台化、可追溯的工程支撑。
```

为什么更好：
- 有国家战略和行业场景，但没有停在宏观口号。
- 写出了外部趋势、现实短板和项目切入。
- 结尾直接导向后续建设任务。

### 示例 2：建设意义

浅写法：

```md
本项目意义重大，能够提升技术水平，带动产业发展，促进生态建设。
```

深写法：

```md
在技术层面，项目拟面向船舶工程数据多源异构、版本关系复杂和设计制造链路长等特点，建立覆盖数据模型、接口转换、规则校核和流程追溯的一体化技术体系。该体系不是简单替代单个软件功能，而是通过统一数据源和标准化对象模型，将设计、分析、制造和管理环节中的关键数据纳入同一管理框架，为后续协同设计、状态管控和合规复核提供稳定基础。

在产业层面，项目建设有助于降低关键工业软件和核心数据管理平台对国外产品的依赖，缓解高授权成本、供应不确定和本土化适配不足带来的产业链风险。通过形成面向国内船舶企业业务流程的自主平台，可以支撑设计所、总装厂、供应商和配套单位之间的数据协同，提高产品研制过程中的响应速度和过程可控性。

在生态层面，项目将带动国产 CAD、CAE、CAPP、MES、ERP 等工业软件与船舶数据管理平台的适配集成，推动形成可扩展的行业软件生态。相关研发过程还将沉淀数据标准、接口规范、工程模板和应用经验，为产学研用协同创新、后续推广应用和人才培养提供可复用基础。
```

为什么更好：
- 按技术、产业、生态三个层面展开。
- 每个层面都写清“解决什么、形成什么、带动什么”。
- 避免用一段价值词替代论证。

### 示例 3：国内外发展现状

浅写法：

```md
国外在工业软件方面起步较早，技术比较先进。国内近年来发展很快，但整体水平仍有差距，需要通过本项目进一步追赶。
```

深写法：

```md
从国内发展现状看，船舶行业在设计软件、仿真分析工具和工程数据管理方面已经形成一定基础，部分企业和科研院所也开展了面向船舶产品数据管理、协同设计和数字化交付的系统建设。但在复杂产品全生命周期数据贯通、跨工具接口适配、工程规则沉淀和多单位协同应用方面，国产平台仍面临功能深度不足、场景覆盖有限和用户迁移成本较高等问题，尚未充分形成支撑船舶研制全流程的稳定工程闭环。

从国外发展与应用现状看，头部工业软件企业通常围绕产品生命周期管理、配置管理、协同设计、供应链协作和项目流程控制形成较完整产品体系，并通过与 CAD、CAE、ERP 等工具集成增强平台黏性。其优势不只在单项功能，而在于长期积累的行业模板、数据接口、生态伙伴和工程应用经验。与之相比，国内相关平台需要在本土业务适配、自主可控、安全合规和复杂船舶工程场景深度应用方面形成差异化突破。
```

为什么更好：
- 国内、国外分别写基础、代表性能力和限制。
- 差距落到工程闭环、生态集成和场景深度，而不是简单强弱对比。
- 直接为创新点和项目建设方案提供比较基线。

### 示例 4：痛点分析与发展前景

浅写法：

```md
当前数据管理存在痛点，未来发展前景广阔，因此需要建设统一平台。
```

深写法：

```md
船舶研制过程中，设计、分析、制造和管理环节通常分布在不同软件系统和不同单位之间，数据格式、编码规则、版本状态和权限边界并不完全一致。模型文件、图纸、仿真结果、物料信息和变更记录一旦缺少统一数据模型和关联关系，就容易形成信息孤岛，导致跨专业协同、设计变更追溯和制造准备衔接成本上升。该痛点表面上是数据分散，实质上是缺少能够贯通产品对象、业务流程和工程规则的一体化数据底座。

面向后续发展，船舶工业软件将进一步向全产业链数据一体化管理、多目标多学科协同优化、数字底座平台和人工智能辅助决策方向演进。上述趋势要求平台不仅能够保存数据，还要能够表达数据之间的工程关系、支撑跨系统调用和流程自动化，并在典型船型、典型业务和典型协作场景中持续验证。由此，本项目的建设前景应落到数据标准、接口体系、规则库、协同流程和工程应用生态，而不是泛泛表述为市场空间广阔。
```

为什么更好：
- 痛点写出表现、原因、影响和项目切入。
- 前景写出趋势、承接能力和应用方向。
- 避免空泛展望。

## 二、项目建设方案示例

### 示例 1：建设目标

浅写法：

```md
本项目将建设智能化平台，提升船舶设计效率，形成自主可控能力，带动产业发展。
```

深写法：

```md
本项目拟面向船舶结构疲劳分析报告编制中多源数据分散、工程知识难以复用和报告校核依赖人工的问题，建设覆盖数据结构化、规则知识化、报告自动生成和一致性校核的一体化支撑平台。项目将以典型船型疲劳分析场景为牵引，形成结构化计算结果库、规范规则库、报告模板库和自动生成工具链，支撑分析结果从计算文件向可审查、可追溯报告内容转化。通过在试点业务中开展部署验证，项目预期提升报告编制一致性和复核效率，并为后续国产工程软件集成、行业模板沉淀和应用推广提供基础能力。
```

为什么更好：
- 目标不是口号，而是由问题、建设对象、能力输出和应用验证组成。
- 写出项目要形成的平台、工具链和数据资源。
- 能自然引出后续“主要问题、研发内容、预期成果”等小节。

### 示例 2：项目解决的主要问题

浅写法：

```md
本项目主要解决数据不统一、报告生成效率低、系统集成困难等问题。
```

深写法：

```md
项目首先需要解决多源计算结果难以稳定转化为报告输入的问题。船体结构疲劳分析通常涉及有限元结果文件、应力云图、工况参数、结构部位编号和历史报告文本等多类材料，不同材料之间缺少统一字段、版本关系和语义标识，导致工程人员需要反复人工整理、核对和转录。本项目拟通过建立面向报告生成的字段体系、数据映射规则和异常提示机制，将分散材料转化为可读取、可追溯、可调用的结构化输入，为后续章节生成和一致性校核提供基础。

项目还需要解决报告内容生成与工程审查口径脱节的问题。现有模板化写法往往只完成字段替换，难以保证计算结果、图表编号、结论语句和规范条款之间的一致性。项目拟建立报告章节、图表模板、规范判断和结论语句之间的规则关系，通过规则约束和人工复核接口，使报告生成过程从简单文本拼接转向受控的工程交付过程。
```

为什么更好：
- 每个问题都有表现、原因、影响和解决方向。
- 问题写法直接服务后续研发任务，不是简单罗列痛点。
- 保持正式公文自然段，而不是答题卡式标签。

### 示例 3：项目研发内容

浅写法：

```md
项目研发内容包括数据处理、知识库建设、报告生成、系统开发和应用验证。
```

深写法：

```md
本项目研发内容拟按照“数据底座建设、规则知识建模、报告生成引擎、系统集成验证、应用推广支撑”的主线组织。数据底座建设任务重点面向有限元结果、图表文件、工况参数和历史报告文本，建立统一字段体系、数据接口和版本关联规则，形成后续自动生成所需的标准化输入。规则知识建模任务重点围绕规范条款、章节模板和结论语句之间的映射关系，形成可调用、可校核的规则库。报告生成引擎任务重点研究章节填充、图表调用、结论生成和一致性检查机制，实现从结构化数据到报告初稿的自动编排。系统集成验证任务则面向典型船型案例，完成工具部署、功能测试、人工复核和问题回写，验证研发内容在实际业务流程中的可用性。
```

为什么更好：
- 按建设主线组织任务，不是把名词排成清单。
- 每项任务都能看出对象、动作、输出和后续关系。
- 为技术路线图、成果清单和指标设置留下可追溯依据。

### 示例 4：技术路线

浅写法：

```md
项目采用人工智能、大数据和云平台技术，构建智能化系统，完成开发测试和推广应用。
```

深写法：

```md
项目技术路线拟采用“数据治理层、规则知识层、生成服务层、业务应用层、验证迭代层”的分层架构。数据治理层负责接入计算结果、图表文件、工况参数和历史报告材料，完成字段标准化、版本匹配和质量检查；规则知识层负责建立规范条款、报告章节、图表模板和结论语句之间的映射关系，形成生成过程的约束条件；生成服务层负责调用结构化数据和规则库，完成章节内容编排、图表调用和一致性校核；业务应用层面向工程人员提供报告生成、人工复核、问题修改和版本追溯功能；验证迭代层通过典型船型案例测试生成完整性、图表匹配准确性和关键数值一致性，并将试点反馈回写到规则库和模板库中。
```

为什么更好：
- 技术路线有层次、链路和任务依赖。
- 写出每一层承担的功能和上下游关系。
- 包含测试验证和反馈迭代，不停留在概念架构。

### 示例 5：应用推广、产学研用和开源策略

浅写法：

```md
项目成果将在行业内推广，采用产学研用协同方式，并适时开源部分成果。
```

深写法：

```md
项目应用推广宜按照“内部试点、示范验证、行业复制、反馈迭代”的路径推进。前期选择典型船型和典型报告场景开展内部部署，验证系统在数据接入、报告生成和人工复核环节的可用性；中期依托牵头单位、设计单位和应用单位形成示范窗口，沉淀模板、接口和运维经验；后期结合行业共性需求，将成熟模块、数据规范和应用指南推广至相近业务场景，并通过用户反馈持续完善规则库和模板库。

产学研用合作方面，企业侧负责业务场景、工程数据、试点部署和成果转化，高校或科研院所负责关键方法研究、模型评估和人才支撑，软件开发单位负责系统架构、接口集成和工程实现，应用单位负责场景验证、问题反馈和推广评价。各方通过需求共研、任务共担、成果共验和应用共推形成闭环，避免研发任务与真实使用场景脱节。

攻关成果开源策略可围绕非敏感工具组件、接口规范、数据字典、模板样例和测试用例分级推进。对于涉及企业核心数据、工程图纸、商业规则或安全边界的内容，应采取脱敏、分层授权或不开源处理；对于具备共性推广价值的基础组件和规范文档，可通过版本管理、质量审核、许可证约束和社区反馈机制持续迭代，使开源服务于生态培育，而不是简单公开代码。
```

为什么更好：
- 推广方案有对象、路径、示范和反馈。
- 产学研用写出各方职责和协同接口。
- 开源策略有范围、方式和风险边界。

## 三、创新点示例

### 示例 1：有比较基线的创新点

浅写法：

```md
本项目创新性地构建智能报告生成平台，实现报告自动生成，具有明显创新性。
```

深写法：

```md
区别于传统基于固定模板的字段替换方式，本项目拟构建面向船体结构疲劳分析场景的规则约束型报告生成机制。该机制不是单纯将计算结果填入模板，而是将结构部位、载荷工况、疲劳寿命、云图文件和规范判断语句统一纳入章节语义模型，通过规则库约束生成内容的取值来源、表达口径和校核关系。由此，报告生成过程从“文本拼接”转向“数据、图表、结论同步受控”的工程交付过程，可降低因人工转录和模板复用不当造成的口径偏差。
```

为什么更好：
- 明确比较对象：固定模板字段替换。
- 写出新机制：章节语义模型和规则库约束。
- 创新落到工程价值：降低口径偏差。

### 示例 2：从项目研发内容提升为创新点

项目研发内容式写法：

```md
研究疲劳分析知识库构建技术，形成报告生成所需的知识库。
```

创新点写法：

```md
在知识组织方式上，项目拟将规范条款、历史报告、典型结构部位和计算结果字段统一组织为面向报告生成的疲劳分析知识库。相较于传统以文件夹、人工经验和单份报告模板为主的知识复用方式，该知识库强调规则条款与报告章节、图表类型和结论语句之间的可计算关联，使工程经验能够从个人经验沉淀转为可检索、可调用、可校核的组织级知识资产。
```

为什么更好：
- 不是只说“构建知识库”，而是说明组织方式变化。
- 有旧方式和新方式对比。
- 体现工程知识资产化价值。

### 示例 3：不能强写“首创”

不建议写法：

```md
本项目首次提出船舶报告自动生成技术，达到国内领先水平。
```

稳健写法：

```md
在公开资料尚未充分显示同类能力已形成成熟工程闭环的情况下，本项目将重点探索船体结构疲劳分析报告从数据读取、图表调用、章节生成到一致性校核的端到端实现路径。相关创新应表述为面向特定船舶工程场景的集成创新和流程创新，不宜在缺少充分主源支撑时使用“首次”“领先”等绝对化判断。
```

为什么更好：
- 避免无依据强词。
- 说明能写成什么创新，不能写成什么创新。

### 示例 4：写得太满时如何收口

过满写法：

```md
本项目的实施将全面提升船舶电气设计领域的数字化、智能化、自主化和协同化水平，显著增强产业链安全保障能力、核心技术支撑能力、工程交付能力和行业生态构建能力，为推动高质量发展、服务国家战略和促进产业升级提供坚实支撑。
```

收口后写法：

```md
本项目拟面向船舶电气设计过程中原理图绘制依赖人工、校核重复、知识难以复用的问题，构建集知识抽取、规则校核、模板生成和结果追溯于一体的智能设计支撑能力。项目实施后，预期形成电气原理图生成与校核工具、规则库和模板库，并通过典型船型样例验证其对设计效率和一致性复核的支撑作用。
```

为什么更好：
- 删除了重复价值词，保留了章节真正需要证明的内容。
- 把“支撑什么”落到具体对象、机制、成果和验证。
- 仍然保留正式公文语气，但不再空泛发散。

## 四、主要技术成果示例

### 示例 1：成果实体

浅写法：

```md
形成智能化报告生成能力，提升企业设计水平。
```

深写法：

```md
围绕报告自动生成项目研发内容，项目实施后预计形成船体结构疲劳分析报告自动生成系统。该系统以结构化计算结果库、报告模板库和规则校核库为基础，支持典型章节自动填充、图表批量调用、结论语句生成和报告版本管理，可作为项目验收的软件成果进行部署测试，并通过典型船型案例验证其生成完整性和结果一致性。
```

为什么更好：
- 成果是系统，不是抽象能力。
- 交代组成、功能边界和验收方式。

### 示例 2：成果组合

浅写法：

```md
项目将形成平台、模型、数据库和报告等多项成果。
```

深写法：

```md
项目成果可按软件工具、数据资源和文档规范三类组织。软件工具方面，形成报告生成原型系统和结果一致性校核模块，用于支撑报告自动编制和审查复核；数据资源方面，形成疲劳分析结构化结果库、典型章节模板库和规范语句规则库，为系统运行提供可复用底座；文档规范方面，形成系统设计说明、测试验证报告和报告编制规则说明，为后续推广应用和运维迭代提供依据。
```

为什么更好：
- 按成果类型分组。
- 每类成果都有用途。
- 不只是罗列名称。

### 示例 3：成果与项目研发内容映射

浅写法：

```md
根据项目研发内容，形成多项技术成果。
```

深写法：

```md
数据结构化处理研究对应形成标准化数据接口和结构化结果库，解决多源计算结果可读取、可追溯的问题；报告生成技术研究对应形成模板引擎和自动生成模块，支撑章节、图表和结论说明的自动编排；一致性校核研究对应形成规则校核模块和测试验证报告，用于检查关键数值、图号、表号和结论语句之间的匹配关系。上述成果共同构成从计算结果到正式报告的工程化交付链路。
```

为什么更好：
- 每项成果都能回指项目研发内容或建设任务。
- 成果之间有链路关系。

## 五、主要技术指标示例

### 示例 1：指标有测试条件

浅写法：

```md
报告生成准确率达到95%以上，效率提升50%以上。
```

深写法：

```md
报告生成完整性指标用于衡量系统对规定章节、表格、图示和结论说明的自动覆盖能力。建议以不少于 `【待补】` 个典型疲劳分析案例作为测试集，检查系统是否能够按照模板要求生成完整报告内容，目标值可设定为核心章节覆盖率不低于 `【待补】`。对于尚缺少测试样本规模和人工基准效率的数据，应先保留待校准标记，待试点案例形成后再确定最终验收阈值。
```

为什么更好：
- 没有依据时不伪精确。
- 写清指标定义、测试集和校准方式。

### 示例 2：指标分组

浅写法：

```md
系统响应快、准确率高、稳定性好，满足应用要求。
```

深写法：

```md
技术指标宜按数据处理、报告生成、校核复核和系统运行四类设置。数据处理类指标重点考核典型结果文件的读取成功率、字段提取完整率和版本匹配准确性；报告生成类指标重点考核章节覆盖率、图表调用正确率和模板填充准确性；校核复核类指标重点考核关键数值一致性检查通过率和问题提示有效性；系统运行类指标重点考核单份报告生成时长、批量处理能力和连续运行稳定性。通过分类设置指标，可以避免只用单一准确率概括复杂工程系统能力。
```

为什么更好：
- 指标按对象和能力分组。
- 每类指标都有测量对象。

### 示例 3：伪精确改保守

不建议写法：

```md
系统报告生成时间控制在3分钟以内，准确率达到99.5%，人工工作量降低80%。
```

稳健写法：

```md
在缺少既有人工编制耗时、样本规模和测试环境基准的情况下，报告生成效率指标不宜直接给出过高精确值。可先将指标表述为“在典型报告模板和标准化输入条件下，单份报告生成时间较人工整理流程明显缩短，具体缩短比例以试点案例实测结果为准”，并在指标表中设置 `【待补：人工基准耗时】`、`【待补：测试样本数量】` 和 `【待补：目标阈值】`，待验证阶段补齐。
```

为什么更好：
- 避免看似漂亮但无依据的数字。
- 说明需要哪些数据才能定指标。

## 六、信息不足反馈示例

### 示例 1：缺应用场景

```md
当前缺口：用户 brief 未明确系统优先服务于新造船设计、在役船改造还是科研分析场景。
影响：不同场景会影响报告模板、验收指标和示范案例选择。
需要确认：本项目优先面向哪一类应用场景？
可先保守处理：若暂不确认，正文将按“船舶结构分析通用报告场景”撰写，并在成果验证和指标阈值处保留 `【待补】`。
```

### 示例 2：缺硬指标

```md
当前缺口：brief 要求写主要技术指标，但没有给出样本规模、测试环境和验收阈值。
影响：无法直接写精确准确率、生成时长和效率提升比例。
需要确认：是否已有指定验收指标，还是允许先按行业同类项目口径提出建议值？
可先保守处理：若没有指定值，将采用“建议指标 + 待项目验证阶段校准”的写法，并把具体阈值登记为 `【待补】`。
```

### 示例 3：来源不足但可以继续

```md
当前缺口：某项国外现状仅有行业媒体报道，缺少官方或论文原文支撑。
影响：不能写成确定性强表述，也不能使用“已形成成熟应用”等判断。
处理方式：正文将降级为“公开资料显示，相关方向已有探索”，并在 `facts-ledger.md` 中标记为 `仅作趋势参考`，同时把补充主源列为后续调研项。
```

## 七、从浅段落改深段落的方法

浅段落：

```md
项目将建设智能化平台，整合多源数据，提升报告编写效率，实现业务流程闭环。
```

改写步骤：
- 找对象：什么报告、什么数据、什么业务流程。
- 找问题：为什么现有流程效率低。
- 找机制：平台通过什么模块或规则解决。
- 找输出：形成什么系统、数据库、模板、报告。
- 找验证：通过什么案例或指标证明。

改深后：

```md
项目拟面向船体结构疲劳分析报告编制中数据来源分散、图表整理重复和结论复核依赖人工的问题，建设由结构化结果库、报告模板引擎和一致性校核模块组成的自动生成系统。系统通过统一读取计算结果、绑定报告章节字段和调用标准化图表模板，将原本分散在计算文件、图片目录和历史报告中的信息转化为可追溯的报告生成输入，并通过典型船型案例测试生成完整性、图表匹配准确性和关键数值一致性，从而支撑报告编制流程从人工整理向规则约束下的自动化交付转变。
```
