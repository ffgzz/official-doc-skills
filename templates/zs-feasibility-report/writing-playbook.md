# ZS-项目可行性报告写作 playbook

## 总体配比规则

- 第2章、第3章、第6章是篇幅重心，必须明显重于第7章、第8章、第9章
- 第3章必须展开到“子课题 / 子任务”级别，不能停留在概括性提纲
- 第7章到第9章原则上采用“短正文 + 表”为主，不要把表格职责重新写成长文
- 第10章只做正式收束，不再扩展出新的技术论述

## 正文比例硬约束

以下比例仅统计正文，不含表格单元格、Mermaid 代码和 notes：

- 第1章建议控制在正文总量的 `6%` 至 `12%`
- 第3章应为全稿最重章节，建议占正文总量的 `40%` 至 `50%`
- 第2章和第6章应明显重于第1章，建议各占正文总量的 `10%` 至 `18%`
- 第5章建议控制在正文总量的 `5%` 至 `10%`
- 第4章正文应克制，建议控制在正文总量的 `5%` 至 `8%`
- 第7章到第9章应明显偏短，建议各自控制在正文总量的 `4%` 以内
- 第10章结论建议控制在正文总量的 `3%` 至 `6%`

如果生成稿明显违反上述比例，应在 review 中记为章节配比问题，而不是视为“风格差异”。
在扩充第3章、第6章时，不要以牺牲第1章和第5章的基本信息完整度为代价。

## 文风硬约束

### 推荐风格

- 以工程说明句、判断句、归纳句为主
- 优先保留材料中的技术术语、流程名词、规范口径
- 优先做“保守改写”，不要做宣传式扩写
- 行文应接近正式可研，而不是项目宣讲稿

### 明确禁止

除非源材料原文已出现，否则不要主动生成以下类型表达：

- “标志性工程”
- “彻底打通”
- “一举破解”
- “无懈可击”
- “颠覆性”
- “首次完全实现”
- “完美响应”
- “极大优势”
- “激流勇进”

### 句式约束

- 少用过长排比句
- 少用连续形容词堆叠
- 不要把一个技术判断写成鼓动性口号
- 能拆成两句的长句，不要硬写成一句

## 章节目标与建议写法

### 第1章 概述
- 目标：讲清楚“为什么要立项、想解决什么问题”
- 不要空泛喊口号，要把行业背景和立项意义对应起来
- 推荐结构：`行业背景 -> 业务痛点 -> 立项必要性`
- 这一章虽然不是重章节，但不能薄到只剩背景套话；至少要把行业背景、业务痛点、立项必要性三层写全

### 第2章 项目现状和发展趋势
- 目标：说明现有方法/产品的不足，以及行业下一步的发展方向
- 最好写清“现有流程为什么效率低、为什么口径不一致、为什么需要自动化/智能化”
- 建议采用“现状判断 → 短板痛点 → 行业趋势 → 本项目切入点”的顺序
- 这一章应明显重于第1章，不能只写成两段背景介绍
- 本章至少应覆盖：
  - 现有行业实践
  - 工具或流程短板
  - 技术路径演变
  - 本项目切入理由

### 第3章 研发内容及技术关键
- 目标：做任务分解
- 最好采用“总体研发内容 → 子任务 → 关键技术 → 创新点”的顺序
- 当参考样稿的三级标题不适用于新项目时，应按当前项目内容重新组织
- 必须展开到 `子课题1 / 子课题1.1 / 子课题1.2` 这类层级，优先承接源材料本身的专题结构
- 这一章通常是全稿最长章节
- 每个子课题至少应包含：
  - `要解决的问题`
  - `研究内容`
  - `方法 / 步骤 / 模块`
  - `预期输出`
- 如果材料中已经有步骤、流程、算法、模块名，不要把它们压缩成一段泛化概述
- 这一章不能只写“做什么”，还要写“怎么做”

### 第4章 技术路线和实施方案
- 目标：把项目从输入到交付的技术与实施链条讲清楚
- 这里必须为 `图1` 和 `表1` 留位
- 写法宜克制：技术路线讲整体链条，实施方案讲阶段安排，不要把表1内容全部写回正文
- 本章正文默认不承担大量细节展开，细节应主要留在图1和表1
- 如果第4章正文比第2章或第6章还厚，通常说明配比失衡

### 第5章 现有工作基础和条件
- 目标：回答“团队是否有能力做成这件事”
- 必须体现已有经验、既有成果、已有数据/设备/流程基础
- 不要只留一小段团队介绍然后完全依赖表2；正文至少要覆盖团队基础、技术基础、数据/工具基础三个方面

## 已知信息保留规则

- 如果材料、facts ledger、已生成表格中已经有明确姓名、单位、时间、任务或预算口径，后续章节和装配稿不得把这些已知值回退成 `【待补】`
- `【待补】` 只用于源材料确实没有提供的字段，不得把“已知但尚未抄入当前文件”的信息误写成待补
- 正式总稿中的 `【待补】` 应明显少于工作稿；若总稿中的 `【待补】` 数量高于对应散件文件，视为回退
- 对预算、成员、牵头单位等关键表格字段，若原始材料中本来就是空白，可在正式表格中保留空单元格；不要为了提醒自己而在正式稿中大面积写 `【待补】`

### 第6章 困难评估
- 目标：不是罗列风险词，而是说明“难点是什么、为什么难、怎么应对”
- 推荐采用多组 `问题：...` / `措施：...` 的成组写法
- 这一章应与第2章、第3章一起构成正文重心

### 第7章 预期目标
- 目标：拆成技术成果和应用成效两类，并尽量量化
- 正文宜短，重点由 `表3`、`表4` 承载
- 正文以导语为主，不要展开成长段论证
- 本章正文默认只保留：
  - 1 段总导语
  - `（一）预期技术成果` 下 1 句引表说明
  - `（二）预期成效` 下 1 句引表说明
- 不要把表3、表4中的成果说明和成效说明再重写成多段正文

### 第8章 计划进度安排
- 目标：阶段、时间、工作内容一一对应
- 正文先交代周期和阶段，再留 `表5`
- 不要把表5逐行重写成正文

### 第9章 经费预算
- 目标：说明预算结构、测算依据和口径
- 没有依据的分项金额不能乱写
- 正文先交代预算口径，再留 `表6`
- 没有材料支撑时，正文只说明预算原则，不扩写支出理由长段

### 第10章 研究结论
- 目标：从必要性、可行性、预期价值三方面做收束
- 1 至 2 段即可，不继续展开新论证

## 图1 配套规则

- 图1默认不是线性流水线图，而是“总目标 -> 子课题 1 / 2 / 3 -> 子任务”的分解式框图
- 若未得到用户额外指示，不要把图1画成纯“输入 -> 输出”的通用 pipeline
- 图1的视觉重心应与第3章子课题结构一致，而不是脱离第3章另造一套拓扑
