# 项目背景 Skill

## 先做什么

开始写之前，必须先做网络搜索，并把结果记入 `workspace/plan/<project-slug>/research-sources.md`。

优先搜索：
- 国家战略、产业政策、专项指南、主管部门通知
- 行业协会、头部企业、研究机构公开资料
- 最近 3 年的论文、白皮书、案例、同类项目指南

## 必写链路

本章默认按以下链路组织：

1. 建设背景
2. 国内外现状分析
3. 痛点问题
4. 攻关必要性

若用户还要求“建设意义”“产业链供应链韧性”“安全意义”，可接在必要性之后。

## 具体写法

### 1. 建设背景

先从宏大叙事切入，但必须迅速收束到本项目主题。

推荐顺序：
- 国家战略 / 产业升级 / 技术演进
- 行业转型压力
- 本领域在当前阶段的窗口期
- 本项目切入点

不要只写空泛口号。

### 2. 国内外现状分析

至少拆成：
- 国内现状
- 国外现状
- 国内外差距或趋势判断

写法要求：
- 要写代表性平台、技术路线、应用阶段、工程成熟度
- 要写“国外做到哪一步、国内卡在哪一步”
- 不能只有“国外先进、国内不足”这种空判断

### 3. 痛点问题

把现状分析转成 3 到 5 个具体痛点。

每个痛点都要写清：
- 当前表现
- 对业务/工程的影响
- 为什么靠现有办法解决不了

### 4. 攻关必要性

要从四个角度收束：
- 战略必要性
- 技术必要性
- 工程必要性
- 产业链/供应链/安全必要性

结尾必须回到“为什么现在必须做这个项目”。

## 来源使用要求

- 至少使用 3 类来源：政策/官方、行业/企业、学术/研究
- 搜索结论可以概括，但不能硬编
- 若关键数据没有公开依据，不写精确数字

## 输出检查

写完后确认：
- 是否形成“背景 -> 现状 -> 痛点 -> 必要性”闭环
- 是否有搜索留痕
- 是否已经为后续研究内容和创新点埋好逻辑前提

## 不要做的事

- 不要只有宏观背景，没有行业痛点
- 不要把“现状分析”写成资料堆砌
- 不要把必要性写成宣传标语
