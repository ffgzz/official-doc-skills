# official-doc-project-background

## 定位

这是“项目背景类内容”的规则库，不是固定章模板。

它覆盖的写作对象包括但不限于：
- 建设背景
- 行业背景
- 立项意义
- 国内外现状
- 现状与差距
- 行业发展动向
- 痛点问题
- 攻关必要性
- 建设必要性
- 产业链安全与自主可控意义

如果用户只要求其中一部分，只写那一部分，并按本 skill 的对应规则执行。

## 先做什么

开始写之前先检查调研门禁是否完成，而不是从零开始搜索。

必须先读取：
- `workspace/plan/<project-slug>/research-plan.md`
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-notes.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

如果背景类调研组仍存在以下任一情况，应先回到 `official-doc-research` 补调研，再回来写：
- 只有零散 2 到 3 条来源
- 多数来源早于近 3 年，且没有标记为 `历史基线` / `基础规范`
- 缺少 `A主源`
- 还没有形成可写正文的 `research-notes.md`

本 skill 不直接联网搜索。只有在背景类台账已经基本齐全、但某个小问题仍缺支撑时，才回到 `official-doc-research` 补充调研，由 `official-doc-research` 使用本地 MCP 完成检索、抓取和落账。

补充调研结果必须登记到：
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

这一步必须在本 skill 已显式加载之后执行。

登记时至少补齐以下字段：
- `research-sources.md`：来源标题、来源类型、来源等级、发布机构/作者、发布时间、链接、支撑章节、是否可直接支撑正文断言
- `facts-ledger.md`：事实表述、对应章节、来源编号、主源编号、辅源编号、事实状态、备注

事实状态只允许使用：
- `已核验`
- `待核验`
- `仅作趋势参考`

来源等级只允许使用：
- `A主源`
- `B辅源`
- `C参考`

如果 `using-official-docs` 尚未先调用 `official-doc-research`，或背景类调研仍明显过浅、过旧，就直接开始写背景、现状、痛点、必要性，说明流程执行错误，应先回到“先过调研门禁，再写”的顺序。

补查完成后，本 skill 只读取调研台账和事实台账，不自行改用其他搜索工具。

政策、文号、发布日期、主管部门表述，优先使用官方来源核验。
市场规模、增长率、部署占比、行业排名等数字，优先使用研究机构或企业原始报告核验。
具体平台、模型、产品能力表述，优先使用官方产品页、论文或一手技术资料核验。

背景类事实的默认来源等级口径：
- `A主源`：国务院/部委/地方主管部门/船级社/标准原文/论文原文/企业官方公告或产品页
- `B辅源`：主流行业媒体、研究机构报告、行业协会公开材料
- `C参考`：百科、博客、论坛、自媒体、聚合资讯、转载稿、问答社区

以下来源不得单独支撑 `已核验`：
- 百度百科等百科词条
- 知乎、CSDN 等博客/问答社区
- 资讯转载站、聚合新闻站
- 单篇营销稿、宣传稿

如果某条背景事实只有 `C参考` 来源：
- 不能标记为 `已核验`
- 不能写成具体政策文号、具体发布日期、精确市场数字或明确产品能力断言
- 只能作为检索线索，继续追到 `A主源` 或 `B辅源`

“已核验”的最低门槛是：
- 至少 1 条 `A主源`
- 或 1 条 `A主源` + 1 条 `B辅源`

若只有 `B辅源` 而没有 `A主源`：
- 原则上只能写成保守概括
- 不建议写成强断言

如果没有完成核验：
- 不要写具体文号
- 不要写具体发布日期
- 不要写精确市场数字
- 不要把单一宣传材料中的表述当作确定事实

可以改写为更稳健的概括性表达，例如：
- `近年国家层面持续推动人工智能与制造业深度融合`
- `国外头部工业软件企业正在探索将生成式AI能力引入仿真与工程交付流程`

不要写成：
- `2025年8月，国务院印发……`
- `2025年全球工业AI软件市场规模已达……`

除非这些表述已经在 `facts-ledger.md` 中标记为 `已核验`。

本 skill 默认假定以下前置工作已经完成：
- `official-doc-core` 已执行
- `using-official-docs` 已完成工作区初始化
- 工作区目录已创建
- `project-brief.md`、`research-sources.md`、`facts-ledger.md`、`progress.md` 已存在

因此，本 skill 不负责：
- 创建工作区
- 初始化 plan 文件
- 补建 `project-brief.md`
- 补建 `00-section-plan.md`

本 skill 负责的输出应写入 `workspace/outputs/<project-slug>/` 对应章节文件。
不要把完整背景正文直接输出到终端。
在终端只简要报告：
- 已写入哪个章节文件
- 已登记哪些来源
- 主入口下一步应继续补什么

正式正文默认写成连续自然段，不要把“背景、现状、痛点、必要性”机械拆成项目符号或加粗小标签。需要细分时，优先用二级小标题加自然段展开。

本 skill 写完后，必须把控制权交还给 `using-official-docs`，由主入口继续判断：
- 是否还有其他章节要写
- 是否需要 `official-doc-table`
- 是否需要 `official-doc-figure`
- 是否应进入 `official-doc-review`

至少覆盖三类来源：
- 政策或官方来源
- 行业或企业来源
- 学术或研究来源

优先补充检索：
- 国家战略、主管部门政策、产业专项指南
- 行业协会、头部企业、船级社、研究院公开资料
- 最近 3 年的论文、白皮书、案例、同类项目指南

## 从附件提炼出的共性写法

### 1. 先“势”后“需”，再落到项目

常见展开链路是：
- 宏观技术革命或国家战略
- 行业转型压力与窗口期
- 本领域业务场景的具体困难
- 项目切入点与立项必要性

也就是说，开头可以从大处写，但必须迅速收束到本项目场景，不能一直停留在宏大口号层面。

### 2. 现状分析必须“国内/国外/差距”三段式

附件中的成熟写法通常不是简单罗列资料，而是：
- 国外做到什么程度
- 国内目前做到什么程度
- 两者的差距与趋势判断是什么

因此不要只写“国外先进、国内不足”，而要写：
- 代表性平台、企业、研究团队、技术路线
- 工程成熟度、应用深度、落地范围
- 当前瓶颈和局限

### 3. 痛点要写成“问题表现 -> 影响 -> 现有办法为何不够”

附件里高质量段落的共同特征是：痛点不是抽象批评，而是能直接转化为后续研究任务。

推荐写法：
- 当前问题是什么
- 对效率、质量、协同、合规、数据资产造成什么影响
- 为什么现有工具或人工方式解决不了

### 4. 必要性通常不是单一角度

常见收束角度包括：
- 战略必要性
- 技术必要性
- 工程必要性
- 产业链/供应链/安全必要性
- 数据资产与知识沉淀必要性

## 写作步骤

### 第一步：识别本次只需要哪些子部分

从用户 brief 中先判断要写的是：
- 只有 `建设背景`
- 只有 `国内外现状`
- `现状 + 痛点`
- `痛点 + 必要性`
- 或一整套 `背景 -> 现状 -> 痛点 -> 必要性`

不要因为 skill 覆盖面大，就硬加用户没要求的小节标题。

### 第二步：搭建逻辑闭环

即使用户只要部分内容，也尽量保留逻辑衔接：
- 背景为现状提供时代与行业前提
- 现状为痛点提供证据
- 痛点为必要性提供论据

### 第三步：按正式公文口径行文

语言要求：
- 正式
- 克制
- 工程化
- 可研/立项口径

避免：
- 宣传稿口吻
- 夸张修辞
- 无来源的绝对化判断
- 无核验的具体数字、文号、日期、排名、市场份额

## 事实门禁

### 1. 先筛选事实层级

搜索得到的信息，先按三层处理：
- `政策 / 标准 / 文号层`
- `行业现状 / 市场数据层`
- `案例 / 产品 / 技术路线层`

### 2. 再决定写法强度

只有 `已核验` 的事实，才允许使用“强表述”：
- 具体年份
- 具体月份
- 具体文号
- 精确数字
- 精确比例
- 明确排名

如果只是 `仅作趋势参考`，只允许写成“弱表述”：
- `近年`
- `当前`
- `近年来`
- `头部企业已开始`
- `行业正在从……走向……`

如果事实来源中没有 `A主源`，默认降一档处理：
- 原本想写成“强表述”的，改成“弱表述”
- 原本想写成精确事实的，改成趋势描述或删去

### 3. 先稳健后丰富

背景章节首先要保证可信，再追求信息量。
宁可少写几个数字，也不要写入未经核验的高风险细节。

## 各子部分写法规则

### A. 建设背景 / 行业背景 / 宏观背景

推荐顺序：
1. 国家战略、产业政策、技术演进
2. 行业数字化、智能化、绿色化转型要求
3. 本细分场景面临的新需求或窗口期
4. 收束到本项目主题

从附件提炼出的常见背景框架：
- `全球技术变革 -> 国家战略部署 -> 集团/行业转型要求 -> 具体业务场景`
- `产业升级压力 -> 国外软件依赖或技术代差 -> 本土化与自主可控要求 -> 项目切入`

写法要点：
- 宏大叙事控制在可支撑项目的范围内
- 用 2 到 4 段完成，不要无限铺陈
- 结尾必须明确“为什么本项目现在要做”

### B. 立项意义 / 建设意义

附件中常见的意义展开维度包括：
- 战略价值
- 技术价值
- 降本增效价值
- 数据资产价值
- 合规与标准化价值
- 自主可控与安全价值

推荐写法：
- 不是空泛写“意义重大”
- 而是写“将解决什么、支撑什么、改善什么、沉淀什么”

### C. 国内外现状 / 发展现状 / 现状与差距

推荐结构：
1. 国外现状
2. 国内现状
3. 差距分析
4. 趋势判断

写法要求：
- 写代表性技术路线，而不是百科罗列
- 要交代其成熟度、应用方式、局限
- 对工业软件、AI、智能体、知识图谱、自动化报告等主题，要特别关注“通用能力是否已转化为工程闭环能力”

可直接借鉴的分析角度：
- 通用模型 vs 垂直模型
- 单点智能 vs 工具链闭环
- 实验室验证 vs 工程场景落地
- 辅助分析 vs 合规交付自动化

### D. 行业发展动向 / 趋势判断 / 前景

从附件可归纳的常见趋势方向：
- 高质量数据壁垒成为竞争核心
- 通用模型向垂直专家模型深化
- 多源异构数据向统一数据底座收敛
- 设计、分析、校核、交付形成闭环
- 从单点工具走向平台化、生态化、智能体化
- 自主可控、产业链韧性、数据安全被纳入建设目标

写法要求：
- 趋势要紧扣本项目主题
- 不要写成空泛行业展望
- 趋势结论应自然服务于后续“为什么要攻关”

### E. 痛点问题 / 问题分析

建议列 3 到 6 类痛点，每类至少回答三件事：
- 问题表现
- 对业务/工程的影响
- 现有方法的不足

常见痛点类型可按主题择用：
- 人工依赖强、效率低
- 易出错、复核成本高
- 数据分散、难以复用
- 图模或多系统之间信息断裂
- 规范合规依赖专家经验
- 跨专业协同困难
- 难以沉淀知识与形成可计算规则

### F. 攻关必要性 / 建设必要性

建议从以下 4 到 6 个角度收束：
- 战略层面
- 技术层面
- 工程层面
- 产业链/供应链/安全层面
- 数据资产层面
- 标准化与推广层面

结尾必须回答：
- 为什么不是“以后再做”
- 为什么不能继续依赖现有方式
- 为什么该项目具备现在启动的必要性

## 常用句式骨架

可以借鉴附件中的正式表达方式，但不要机械套话：

- `当前，随着……，本领域正由……向……转变。`
- `与国外同类技术路线相比，国内在……方面仍存在……差距。`
- `现有方法在……场景下主要存在……问题，难以满足……要求。`
- `为破解上述瓶颈，有必要围绕……开展系统性攻关。`
- `项目实施后，将为……提供……支撑。`

## 与后续章节的衔接要求

背景类内容必须为后续章节埋好逻辑前提：
- 为 `研究内容` 提供问题来源
- 为 `创新点` 提供差异化参照系
- 为 `技术成果` 提供产出必要性
- 为 `技术指标` 提供量化依据方向

## 触发样例

以下情形都必须触发本 skill：

### 样例 1

用户写：
- `第1章概述，需要写行业背景、立项意义`

应触发原因：
- 虽然章名是 `概述`，但内容属于背景类规则

### 样例 2

用户写：
- `第2章项目现状和发展趋势，需要写国内外现状、行业发展动向`

应触发原因：
- `国内外现状` 和 `发展动向` 都属于本 skill

### 样例 3

用户写：
- `第二章只写痛点问题和攻关必要性，不单列背景小节`

应触发原因：
- 即使没有出现 `项目背景` 四个字，只要出现 `痛点`、`必要性`，仍然必须调用本 skill

### 样例 4

用户写：
- `请先写国内外发展现状及差距，再写自主可控和产业链安全意义`

应触发原因：
- `现状`、`差距`、`自主可控`、`产业链安全` 全都属于本 skill 的覆盖范围

## 输出检查

写完后确认：
- 是否先做了搜索并登记来源
- 是否按用户要求只写了相关子部分
- 是否形成“背景/现状/痛点/必要性”的合理闭环
- 是否写清国内外差距，而非空泛表态
- 是否给后续研究内容和创新点留出了逻辑接口

## 不要做的事

- 不要只有宏观口号，没有行业和场景落点
- 不要把现状分析写成资料堆砌
- 不要把痛点写成抽象抱怨
- 不要把必要性写成宣传语
- 不要无依据写“国际领先”“国内首创”
