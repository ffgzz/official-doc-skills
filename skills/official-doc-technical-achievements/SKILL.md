# official-doc-technical-achievements

## 定位

这是“技术成果类内容”的规则库。

它要回答的是：
- 这个项目最终交付什么
- 这些成果分别来自哪部分研究内容
- 成果以什么形态验收
- 对后续应用、推广、交付有什么作用

它不是写“预期成效”，也不是写“技术指标”。

本 skill 默认假定：
- `using-official-docs` 已完成工作区和 plan 初始化
- `official-doc-core` 已完成公共校验

本 skill 不负责：
- 创建工作区
- 创建 plan 文件
- 决定是否进入装配

本 skill 的正文结果必须写入 `workspace/outputs/<project-slug>/` 对应章节文件。
不要把完整成果正文直接输出到终端。

本 skill 写完后，必须把控制权交还给 `using-official-docs` 继续推进下游流程。

成果正文应采用“成果名称 + 来源于哪项研究内容 + 作用 + 交付形态”的正式段落表达，不要写成简历式清单。

## 先做什么

开始写之前先检查成果类调研门禁是否完成。

必须先读取：
- `workspace/plan/<project-slug>/research-plan.md`
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-notes.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

如果成果类调研仍存在以下任一情况，应先回到 `official-doc-research` 补调研：
- 还不知道同类项目通常交付哪些成果形态
- 只有项目内部想法，没有外部成果口径参照
- 缺少近年项目指南、任务书、成果清单样例

只有在成果框架已基本明确、但个别成果命名仍需补证时，才允许少量补充检索，例如：
- 同类项目通常形成哪些成果形态
- 行业内常见的交付件、验收件、成果件写法
- 相关成果类别的合理命名方式

记录到：
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

默认搜索工具优先级：
1. 当前 session 已暴露的 MCP 搜索 / connector 搜索工具
2. 其他会话可用的 MCP / connector 搜索工具

不允许使用内置 `Web Search`。
如果当前没有可用的 MCP 搜索工具，或 MCP 搜索报错无法继续：
- 停止联网搜索
- 在 `facts-ledger.md` 中记录搜索阻断原因
- 在 `progress.md` 中记录本章因缺少 MCP 搜索而暂停
- 不得改用内置搜索继续写正文

## 从附件提炼出的共性写法

### 1. 技术成果必须从研究内容推导出来

附件中的成熟文本反复强调：
- `根据以上研究专题，项目实施完毕后，可形成以下技术成果`

也就是说，成果不是凭空追加的愿景，而是研究内容的自然产出。

### 2. 成果常按“系统/模块/数据/报告/标准/知识产权/验证”分组

从附件归纳，常见成果类型包括：
- 平台或系统
- 功能模块、算法模型、工具链
- 数据集、知识库、样本库、规则库
- 技术报告、方案、规范、标准草案
- 专利、软著、论文
- 应用验证成果、示范场景成果

### 3. 成果写法强调“名称 + 作用 + 形态”

好的成果描述通常会写清：
- 成果叫什么
- 它解决或支撑什么
- 以什么形式交付或验收

## 写作原则

### 原则 1：先映射再写成果

先建立一张隐含映射表：
- 研究内容 1 -> 形成哪些成果
- 研究内容 2 -> 形成哪些成果
- 研究内容 3 -> 形成哪些成果

再把这些成果按用户需要的章节结构输出。

### 原则 2：成果要“看得见、交得出、验得到”

优先写：
- 平台原型系统
- 模块工具
- 数据集与知识库
- 报告与标准
- 软件著作权、专利、论文
- 应用验证报告

少写：
- `形成能力`
- `提升水平`
- `推动发展`

这些更像成效，不像成果。

### 原则 3：成果名称应专业化

比起泛称：
- `智能平台`
- `关键模块`

更好的写法是：
- `船体结构疲劳分析报告自动生成系统`
- `目标节点分类模型`
- `自动化应力云图输出工具`
- `疲劳分析知识语料库`
- `合规报告模板引擎`

## 常见成果分组方式

根据附件，可优先采用以下分组：

### A. 软件与平台成果

适用于：
- 系统
- 平台
- 引擎
- 工具链
- 智能体工作流

写法要点：
- 成果名称
- 功能边界
- 主要应用对象
- 交付形态

### B. 模型与算法成果

适用于：
- 分类模型
- 识别算法
- 优化算法
- 生成模型
- 规则引擎

写法要点：
- 与哪项研究内容对应
- 主要解决什么问题
- 在全流程中承担什么角色

### C. 数据与知识成果

适用于：
- 数据集
- 语料库
- 知识图谱
- 规则库
- 模板库

附件里这类成果很常见，尤其适合 AI、工业软件、报告自动化项目。

### D. 文档与标准成果

适用于：
- 技术报告
- 方案报告
- 规范草案
- 标准
- 手册
- 部署文档

### E. 知识产权与推广成果

适用于：
- 发明专利
- 软件著作权
- 论文
- 应用验证报告
- 试点示范成果

## 推荐写法

每个成果项至少回答四件事：
- 成果名称是什么
- 来源于哪项研究内容
- 核心用途是什么
- 以什么形态交付或验收

可采用如下句式：
- `围绕……研究内容，形成……成果，用于……，并以……形式交付。`
- `项目实施后，预计形成……，支撑……场景应用。`

## 与“成效”和“指标”的边界

### 不要把成果写成成效

例如：
- `效率提升 50%`
- `错误率降低 60%`

这是成效或指标，不是成果。

### 不要把成果写成指标

例如：
- `准确率 ≥ 90%`
- `并发用户数 ≥ 100`

这是指标，不是成果。

## 触发样例

以下情形都必须触发本 skill：

### 样例 1

用户写：
- `第7章预期目标，需要写预期技术成果`

应触发原因：
- `预期技术成果` 是本 skill 的直接命中项

### 样例 2

用户写：
- `请列出本项目的交付成果和成果形式`

应触发原因：
- `交付成果`、`成果形式` 都属于成果类规则

### 样例 3

用户写：
- `最后补一节应用成果，说明项目完成后形成哪些系统、模型和报告`

应触发原因：
- `应用成果` 仍然属于技术成果写法，不是单纯成效

### 样例 4

用户写：
- `项目成果一章中，先写研究成果，再写知识产权成果`

应触发原因：
- `研究成果`、`项目成果` 都应触发本 skill

## 输出检查

写完后确认：
- 是否先做了搜索并登记来源
- 是否由研究内容逐项推导成果
- 是否写清了成果名称、用途、交付形态
- 是否把成果、成效、指标区分开
- 是否覆盖软件、数据、报告、标准等合理成果类型

## 不要做的事

- 不要脱离研究内容凭空列成果
- 不要只写“形成一批成果”
- 不要把成效和指标混进成果列表
- 不要所有成果都写成同一句式的空壳条目
