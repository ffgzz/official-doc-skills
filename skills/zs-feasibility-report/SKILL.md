# ZS-项目可行性报告主 Skill

这是 `ZS-项目可行性报告` 的定制化主 Skill。

它的目标是让用户只需说明：
- 要写 ZS-项目可行性报告
- 材料在哪里

然后由你自动完成：
- 套用固定模板
- 初始化输出目录
- 生成骨架
- 分阶段写正文
- 在需要时主动生成表格、图示并完成 `review -> revise -> assemble`

## 一、不要再向用户默认追问参考样稿

本 Skill 的结构、风格和推进方式，优先来自：
- `templates/zs-feasibility-report/outline.md`
- `templates/zs-feasibility-report/writing-playbook.md`
- `templates/zs-feasibility-report/table-catalog.md`
- `templates/zs-feasibility-report/figure-catalog.md`
- `official-doc-core`

除非用户自己明确提供样稿并要求对齐，否则不要把“需要参考样稿”当作默认前提。

## 二、先执行 official-doc-core

在开始本 Skill 的任何正文、表图衔接或阶段推进动作之前，必须先确认：
- 本轮已经单独执行过 `official-doc-core`
- `workspace/plan/progress.md` 已留下 `official-doc-core` 的执行痕迹

如果没有这两点，先执行 `official-doc-core`，再继续本 Skill。

## 三、模板结构（固定）

一级标题固定为以下 10 章：

1. 一、概述
2. 二、项目现状和发展趋势
3. 三、研发内容及技术关键
4. 四、技术路线和实施方案
5. 五、现有工作基础和条件
6. 六、困难评估
7. 七、预期目标
8. 八、计划进度安排
9. 九、经费预算
10. 十、研究结论

二级标题以 `templates/zs-feasibility-report/outline.md` 为准。

## 四、默认写作风格

### 格式刚性
- 必须严格遵守 `ZS-项目可行性报告.docx` 对应的 10 章结构
- 一级标题、二级标题按模板固定，不得自行改名
- 第三章允许下钻，但下钻方式必须服务“子课题 / 子任务”展开
- 第七章到第九章默认采用“短正文 + 表”为主的写法
- 这是一份正式可研/立项材料，不是聊天式方案说明

### 结构风格
- 中等篇幅
- 逻辑主线清晰
- “为什么做 → 现状不足 → 做什么 → 怎么做 → 谁来做 → 目标是什么 → 什么时候做 → 预算如何 → 为什么可行” 这条主线必须完整
- 篇幅重心要向第2章、第3章、第6章倾斜
- 第7章、第8章、第9章正文从简，重心交给表3至表6
- 若用户明确给出总字数，以上结构重心仍保持不变，但各章实际长度要按目标总字数重新分配

### 配比硬约束
- 第1章不应被压缩成只剩背景导语，默认应达到正文总量的 `6%` 至 `12%`
- 第3章必须是全稿最重章节，默认应达到正文总量的 `40%` 至 `50%`
- 第2章与第6章应明显重于第1章，并明显重于第7章到第9章
- 第5章不应被压缩成表2前的一小段过渡文字，默认应达到正文总量的 `5%` 至 `10%`
- 第4章正文必须克制，默认不应超过正文总量的 `8%`
- 第7章、第8章、第9章正文默认各自不超过正文总量的 `4%`
- 在扩充第3章、第6章时，不得通过削薄第1章、第5章来“凑比例”
- 若已生成稿件明显违反上述规则，不得直接进入 `assemble`

### 用户字数要求

若用户明确给出总字数目标，例如：
- `控制在 8000 字左右`
- `总字数 10000 字`
- `不少于 12000 字`
- `不超过 9000 字`

则必须把该要求视为硬约束，并按章节比例把目标总字数切分为各章的预算字数。

默认原则：
- 先保住章节相对轻重关系，再适配总字数
- 优先压缩第4章、第7章、第8章、第9章，不要优先削薄第3章
- 若目标总字数明显低于模板自然体量，应优先保留第2章、第3章、第6章主论证，压缩次要铺陈
- 若目标总字数明显高于模板自然体量，应优先补足第2章、第3章、第6章和第5章，不要把第7章到第9章扩成重章节

### 语言风格
- 申报/可研口径
- 正式、谨慎、可落地
- 不写空泛宣传口号
- 不把预期写成既成事实

### 负面约束

除非源材料原文明确出现，否则不要主动生成以下类型表达：
- “标志性工程”
- “彻底打通”
- “一举破解”
- “无懈可击”
- “颠覆性”
- “完美响应”
- “极大优势”
- “首次完全实现”

默认优先做“保守改写”，不要做宣传式扩写。

### 事实策略
- 没有根据的数字统一写 `【待补】`
- 时间、经费、成果数、成员数不能臆造
- 已在材料、facts ledger、已生成表格中明确出现的姓名、单位、任务、时间、成果，不得在后续回退成 `【待补】`
- 正式总稿中的 `【待补】` 数量不得高于对应散件文件；若出现回退，必须先修正再进入 `assemble`
- 若用户给出总字数要求，必须在 `workspace/plan/project-overview.md` 或 `progress.md` 中记录当前目标总字数

## 五、自动执行流程

### 第 1 阶段：初始化与骨架
必须完成：
1. 读取模板文件
2. 建立或复用 `workspace/outputs/zs-feasibility-report/`
3. 生成 `workspace/outputs/zs-feasibility-report/01-outline.md`
4. 更新 `workspace/plan/project-overview.md`
5. 更新 `workspace/plan/source-materials.md`
6. 更新 `workspace/plan/facts-ledger.md`
7. 更新 `workspace/plan/progress.md`

### 第 2 阶段：优先生成正文主干（1～4 章）
优先写：
- 第1章 概述
- 第2章 项目现状和发展趋势
- 第3章 研发内容及技术关键
- 第4章 技术路线和实施方案

原因：这四章决定整份可行性报告的立项逻辑与技术可信度。

其中必须注意：
- 第2章要写得“厚”，不能只写成一段行业套话
- 第3章是全稿最重章节，必须展开到子课题级别
- 第4章正文应克制，重点为图1和表1留位

### 第 3 阶段：支撑与目标（5～7 章）
继续写：
- 第5章 现有工作基础和条件
- 第6章 困难评估
- 第7章 预期目标

其中必须注意：
- 第5章以团队与基础条件为核心，不要泛泛堆砌履历
- 第6章要写得“厚”，优先采用“问题：... / 措施：...”成组展开
- 第7章正文短写，成果与成效主要靠表3、表4承载
- 第7章正文默认只保留 1 段总导语 + 2 句引表说明；不要扩成两节以上的长正文

### 第 4 阶段：计划、预算、结论（8～10 章）
继续写：
- 第8章 计划进度安排
- 第9章 经费预算
- 第10章 研究结论

其中必须注意：
- 第8章正文只需交代周期与排期逻辑，主体依赖表5
- 第9章正文只需交代预算口径，主体依赖表6
- 第10章收束为 1 至 2 段正式结论，不再扩写成长章节

## 六、各章写法要求

### 第1章 概述
本章回答什么：
- 为什么立项
- 这个项目解决什么核心问题
- 项目的总体目标是什么

必须包含：
- 行业背景
- 核心痛点
- 项目总体目标

禁止写法：
- 只写宏大背景，不落到业务问题
- 开篇堆砌大量宣传词

推荐结构：
- 行业背景 -> 业务痛点 -> 立项意义

必须回答：
- 为什么要做这个项目
- 这个项目要解决什么核心问题
- 项目的总体目标是什么

### 第2章 项目现状和发展趋势
本章回答什么：
- 现有行业或技术做到哪一步
- 当前方案为什么还不够
- 本项目为什么此时切入

必须包含：
- 现状判断
- 痛点或瓶颈
- 技术路径变化
- 本项目切入点

禁止写法：
- 只写两段背景介绍
- 只写“AI 很重要”这类泛化判断

推荐结构：
- 现状判断 -> 痛点拆解 -> 发展趋势 -> 本项目切入点

必须回答：
- 当前行业/技术/流程发展到什么程度
- 当前方案还有什么明显不足
- 下一步发展趋势与本项目的关系是什么

强制要求：
- 不能只写成泛泛的背景介绍
- 至少形成“现状判断 + 痛点拆解 + 发展趋势 + 本项目切入点”四层逻辑
- 当材料足够时，应比第1章明显更厚

### 第3章 研发内容及技术关键
这是核心技术章。
本章回答什么：
- 到底做哪些研发任务
- 每个子课题解决什么问题
- 每个子课题怎么做
- 关键技术和创新点在哪里

每个子课题至少写出：
- `要解决的问题`
- `研究内容`
- `方法 / 步骤 / 模块`
- `预期输出`

禁止写法：
- 把整个子课题压成 1 段概述
- 把材料里已有的子任务名词重新抽象成空泛总结
- 只写“做什么”，不写“怎么做”

推荐展开模板：
- `子课题X ...`
- `子课题X.Y ...`
- `拟解决 ... 问题`
- `具体包括 ...`
- `重点开展 ...`
- `形成 / 输出 ...`

必须尽量写清：
- 研发对象
- 子任务分解
- 每个子任务解决什么问题
- 关键技术与创新点

强制要求：
- 必须展开到“子课题 / 子任务”级别，不能只保留几条概括性列表
- 优先采用 `子课题1 / 子课题1.1 / 子课题1.2` 这类层次展开
- 如果输入材料本身已经给出专题、模块、研究方向，就要据此下钻，而不是重新抽象成空泛概述
- 这一章通常是全稿最长章节

### 第4章 技术路线和实施方案
这是核心方案章。
本章回答什么：
- 整体技术链条如何串起来
- 各阶段实施如何安排

必须包含：
- 技术路线总述
- `图1`
- 实施阶段概述
- `表1`

禁止写法：
- 用长篇正文替代 `图1`
- 把 `表1` 的所有内容整段写回正文
- 把第4章写得比第2章、第6章还厚

正文中必须为以下内容留位：
- `图1 技术路线图`
- `表1 项目任务表`

写法要求：
- `（一）项目技术路线` 用 1 至 2 段解释整体链条，然后留 `图1`
- `（二）实施方案` 用分阶段文字概括实施安排，然后留 `表1`
- 不要在第4章把表1内容整段硬写完

### 第5章 现有工作基础和条件
本章回答什么：
- 团队为什么有能力做成
- 目前已经具备哪些数据、工具、经验和合作基础

必须包含：
- 团队概述
- `表2`
- 既有成果 / 工具 / 数据 / 合作基础

禁止写法：
- 泛泛堆履历
- 只写“团队很强”但没有已有基础

正文中必须为以下内容留位：
- `表2 项目主要成员`

写法要求：
- `（一）项目负责人及团队` 先概述，再留 `表2`
- `（二）相关领域的成果` 重点写已有成果、已有工具、已有数据和合作基础

### 第6章 困难评估
本章回答什么：
- 真正难在哪
- 为什么难
- 对应措施是什么

必须包含：
- 多组 `问题：...`
- 多组 `措施：...`

禁止写法：
- 只列风险名词
- 只写困难不写应对

写法要求：
- 默认采用多组 `问题：...` / `措施：...` 的写法
- 应覆盖技术、数据、流程、模板/生成等核心困难
- 这一章应明显重于第7章到第9章

### 第7章 预期目标
本章回答什么：
- 预期形成哪些技术成果
- 预期带来哪些应用成效

必须包含：
- 简短导语
- `表3`
- `表4`

禁止写法：
- 把本章写成宣传稿
- 用大段正文替代表格

正文中必须为以下内容留位：
- `表3 项目技术成果`
- `表4 项目预期成效`

写法要求：
- `（一）预期技术成果` 只做导语，然后留 `表3`
- `（二）预期成效` 只做导语，然后留 `表4`
- 第7章总正文若超过 1 段导语 + 2 句引表说明，通常视为过写
- 不把本章写成大段宣传性描述

### 第8章 计划进度安排
本章回答什么：
- 项目周期多长
- 分几阶段推进
- 每阶段对应什么任务

必须包含：
- 1 段周期说明
- `表5`

禁止写法：
- 把表5逐行改写成长文

正文中必须为以下内容留位：
- `表5 项目攻关计划进度安排`

写法要求：
- 先用 1 段说明项目周期与阶段划分，再留 `表5`
- 不把阶段计划全文平铺成多段长文

### 第9章 经费预算
本章回答什么：
- 预算口径是什么
- 预算结构如何安排

必须包含：
- 1 段预算口径说明
- `表6`

禁止写法：
- 无依据补金额
- 用很长的预算说明替代表格

正文中必须为以下内容留位：
- `表6 经费预算表`

写法要求：
- 先用 1 段说明预算口径、来源或测算原则，再留 `表6`
- 没有依据的金额和资金来源一律写 `【待补】`

### 第10章 研究结论
本章回答什么：
- 是否值得立项
- 为什么可行
- 预期价值是什么

必须包含：
- 必要性
- 可行性
- 预期价值

禁止写法：
- 再引入新的技术细节
- 写成 3 段以上长收尾

## 八、何时主动调用通用 Skill

### 主动调用 `official-doc-table`
当出现以下任一情况时：
- 正文中已留出表1～表6引用位
- 当前阶段已写到团队、成果、进度、预算等明显需要表格的章节
- 用户说“继续推进”，而当前缺口正好是表格

默认触发顺序：
1. 第4章完成后先补 `表1`
2. 第5章完成后补 `表2`
3. 第7章完成后连续补 `表3`、`表4`
4. 第8章完成后补 `表5`
5. 第9章完成后补 `表6`

### 主动调用 `official-doc-figure`
当出现以下任一情况时：
- 第4章已形成技术链条，可生成 `图1 技术路线图`
- 用户说“继续推进”，而图1仍未生成

默认触发顺序：
1. 第4章 `（一）项目技术路线` 写稳后立即补 `图1`
2. `图1` 应优先于 `表1` 生成，因为第4章先走“路线”，再走“实施方案”

补图额外要求：
- ZS 的 `图1` 默认必须优先还原“总目标 -> 子课题 1 / 2 / 3 -> 子任务”的树状结构
- 如果改画成线性 pipeline，必须有明确理由，并在 figure notes 中说明

### 主动调用 `official-doc-review`
当出现以下任一情况时：
- 一轮正文、表格、图示已基本成型
- 你准备告诉用户“这一轮已完成”
- 用户明确要求检查、复核、体检

默认进入 review 的时点：
- 至少应等 `图1` 和 `表1` 至 `表6` 齐备后再做总 review
- 若只是阶段性推进，也应在当前阶段对应的表图补齐后再 review

### 主动调用 `official-doc-revise`
当出现以下任一情况时：
- `official-doc-review` 已识别出问题项
- 章节配比、标题、表图引用、事实口径需要回修
- 你准备进入正式总稿装配前的最后整理

### 主动调用 `official-doc-assemble`
当出现以下任一情况时：
- review 与 revise 已完成，准备交付完整成稿
- 用户要求“合稿”“正式稿”“总稿”
- 你已经具备第1章到第10章正文、图1、表1至表6

只有在以上条件同时满足时，才允许进入 `official-doc-assemble`。
不要因为局部章节已经 review 完成，就提前装配阶段稿。

## 九、不要做的事

- 不要把 10 章一口气写成一篇长回复却不落文件
- 不要在没依据时补写预算金额
- 不要把表格内容全部揉进正文
