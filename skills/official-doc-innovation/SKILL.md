# official-doc-innovation

## 定位

这是“创新点类内容”的规则库。

它不是单独发明一套漂亮说法，而是负责把：
- 现状差距
- 痛点瓶颈
- 研究内容
- 技术路线

收束成可成立、可辩护、可回指的创新表达。

本 skill 默认假定：
- `using-official-docs` 已完成工作区和 plan 初始化
- `official-doc-core` 已完成公共校验

本 skill 不负责：
- 创建工作区
- 创建 plan 文件
- 决定是否直接结束流程

本 skill 的正文结果必须写入 `workspace/outputs/<project-slug>/` 对应章节文件。
不要把完整创新点正文直接输出到终端。

本 skill 写完后，必须把控制权交还给 `using-official-docs` 继续推进下游流程。

创新点正文默认不用项目符号堆砌“亮点”，而应写成有参照系、有差异来源、有工程价值落点的自然段或少量正式小节。

正式正文写法必须同时遵守：
- `official-doc-core` 中的公文文风规则
- [../official-doc-core/references/formal-doc-style.md](../official-doc-core/references/formal-doc-style.md)
- [../official-doc-core/references/attachment-writing-patterns.md](../official-doc-core/references/attachment-writing-patterns.md) 中的编号方案和创新点写法
- [../official-doc-core/references/depth-writing-rules.md](../official-doc-core/references/depth-writing-rules.md)
- [../official-doc-core/references/depth-writing-examples.md](../official-doc-core/references/depth-writing-examples.md) 中的创新点示例

`attachment-writing-patterns.md` 是硬门禁，也是必须使用的句式模板，不是可选参考。每个创新点必须套用该文件中“创新点写法”的常用写法和句式骨架，尤其是 `区别于传统...方式，本项目拟构建...机制`、`相较于现有...路线，该机制将...统一纳入...`、`由此，...从...转向...，支撑...` 这类句式。每个创新点至少使用一组“比较对象 -> 现有局限 -> 本项目机制 -> 差异成立原因 -> 工程价值”的句式链，并覆盖 `比较对象 / 现有局限 / 本项目机制 / 差异成立原因 / 工程价值 / 对应研究内容或成果指标` 中至少 5 项。若缺少比较对象和差异成立原因，只能写成“技术特点”或“特色做法”，不得写成创新点。

写创新点前，必须先从 `research-evidence.md`、`research-notes.md`、`facts-ledger.md` 中为每个创新点抽出至少 2 个证据点：1 个用于支撑比较基线或现有路线局限，1 个用于支撑本项目机制差异。没有外部比较和差异依据时，不得硬写“创新点成立”。

创新点正文不得裸露内部来源编号，但每个“区别于...”“相较于...”都必须能回溯到证据卡。若创新点段落删掉“区别于/相较于”后仍然成立，说明它本质上只是研究内容摘要或价值判断，应降级为“技术特点”或退回重写。

创新点要克制，不要用“首创、领先、填补空白、重大突破、显著引领”堆气势。优先写机制差异和工程价值，再在结尾保留 1 句稳健收束；若一段超过一半篇幅都在写“先进性、价值、意义”，说明文风过满，必须压缩。

扩写创新点时，不得新增 `00-section-plan.md` 未登记的二级节，也不得用连续流水号标题堆篇幅。需要补深度时，只能加深既有创新点的比较基线、现有局限、项目机制和工程价值，并回指研究内容、成果和指标。

若本章由子代理执行（如 Claude Code 并行任务），该子代理的首个动作必须是显式加载 `official-doc-innovation`，并在开始任何读写前先回报“已显式加载：official-doc-innovation”；未完成这一步，不得读取台账或写稿。该子代理也必须完整遵守本 skill 与 `attachment-writing-patterns.md` 的硬规则；不得把创新点写成无比较基线的摘要或口号。
若运行环境支持 skill 显式调用，子代理必须在开始读取台账和写作前显式加载 `official-doc-innovation`；仅手动读取本文件路径不算完成本 skill 的调用。
子代理任务单应复用 `skills/official-doc-core/references/subagent-task-card-template.md`，并写明本章独占可编辑文件。

## 先做什么

开始写之前先检查创新点所需的比较基线是否已经由 `official-doc-research` 做扎实。

必须先读取：
- `workspace/plan/<project-slug>/research-plan.md`
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/research-notes.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

如果创新类调研仍存在以下任一情况，应先回到 `official-doc-research` 补调研：
- 缺少对“传统方案 / 现有路线 / 同类项目”的近年基线
- 只有本项目视角，没有外部参照系
- 来源还停留在旧资讯稿或单篇宣传稿
- `research-evidence.md` 没有记录外部方案、传统路线或主流工具能力边界的证据点

本 skill 不直接联网搜索。只有在比较基线已形成、但某个创新点仍需补证时，才回到 `official-doc-research` 补充调研，由 `official-doc-research` 使用本地 MCP 完成检索、抓取和落账。补查重点包括：
- 同领域已有方案、竞品、论文、专利、白皮书
- 公开可见的主流技术路线和能力边界
- 同类项目的创新点写法和口径

补充调研结果必须记录到：
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-evidence.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

补查完成后，本 skill 优先读取 `research-evidence.md` 的证据卡，再读取调研台账和事实台账，不自行改用其他搜索工具。

## 写作深度门槛

创新点类正文不能只写“新平台、新方法、新模式”。每个创新点必须形成比较论证链：
- `比较基线`：相对传统人工方式、现有单点工具、通用模型方案、国外平台或国内同类做法中的哪一种。
- `现有局限`：现有路线为什么不能充分解决本项目场景问题。
- `新机制`：本项目的新方法、新模型、新流程或新组织方式是什么。
- `差异成立原因`：这种新机制为什么和现有做法不同，而不是换个说法。
- `工程价值`：差异最终落到效率、质量、合规、标准化、自主可控或交付能力中的哪一项。
- `映射关系`：对应哪项研究内容、技术成果或技术指标。

如果没有外部比较基线，不得写“创新点成立”。应回到 `official-doc-research` 补充 `IN` 组调研，或把文字降级为“特色做法/技术特点”。

## 从附件提炼出的共性写法

写创新点前必须读取 `00-section-plan.md` 中的全文编号方案。创新点若作为二级节，编号服从全文方案；若作为“关键技术及创新点”内部条目，必须沿用本章既定层级，不得与研究内容同级编号混用。

创新点只能写入 `00-section-plan.md` 已登记的创新相关小节。若创新点属于 `（二）关键技术及创新点` 内部内容，不得另起 `（三）技术创新与突破`。需要扩写时，应加深每个创新点的比较基线、现有局限、项目机制和工程价值。

### 1. 创新点不是“夸项目”，而是“写差异”

附件中的高质量创新点，都不是空喊“先进”“领先”，而是强调：
- 相对传统方法改变了什么
- 相对现有路线突破了什么
- 新增了什么机制、闭环或能力

### 2. 创新表达通常带有“旧 -> 新”的对照关系

附件里常见的有效表达方式包括：
- `从……转为……`
- `由……升级为……`
- `实现……一体化`
- `构建……闭环`
- `突破传统……模式`

这类表达之所以有效，是因为它天然带有参照系。

### 3. 创新点必须能回指研究内容和现状差距

不能出现以下问题：
- 创新点和前文现状分析无关
- 创新点和研究内容不对应
- 创新点和成果、指标无法挂钩

### 4. 附件式创新点骨架

创新点应按 `比较对象 -> 现有局限 -> 本项目机制 -> 差异成立原因 -> 工程价值` 写成完整段落。

可直接套用的段落骨架：

```md
区别于传统...方式，本项目拟构建...机制。相较于现有...路线，该机制将...、...和...统一纳入...，通过...实现...。由此，...从...转向...，为...提供...。
```

若没有足够证据支撑“创新”，应降级写成 `技术特点` 或 `特色做法`，不得强写 `首创`、`领先`、`填补空白`。

正式可研/立项材料中的创新点不得压缩成一句评审摘要。每个创新点建议 220 到 350 字，至少包含 2 个自然句：第一句写比较基线和现有局限，第二句写本项目机制、差异成立原因和工程价值。若 review 要求“压缩创新点”，只能删除重复修饰，不能删除比较基线、技术机制和价值落点。

## 写作原则

### 原则 1：先找参照系

创新点必须明确基于什么作比较：
- 传统人工方式
- 现有单点工具
- 通用大模型方案
- 海内外主流技术路线
- 本领域既有工程流程

### 原则 2：写“机制创新”，不是只写“效果很好”

优先写：
- 技术机制怎么变了
- 数据组织方式怎么变了
- 流程闭环怎么建立了
- 交付方式怎么升级了

效果可以写，但应放在机制之后。

### 原则 3：创新点数量适中

一般写 3 到 8 点即可。

每一点都要足够完整，宁可少而实，不要多而虚。

## 推荐结构

每个创新点建议回答四件事：
- 现有方法的局限是什么
- 本项目提出了什么新机制/新方法/新系统
- 与现有路线相比差异在哪里
- 对工程实践带来什么直接价值

可以采用如下表达骨架：
- `针对……长期依赖……的问题，项目提出……`
- `区别于传统……方式，本项目通过……`
- `相较于现有……路线，本项目实现……`
- `从而形成……能力/闭环/优势`

## 常见创新点类型

根据附件内容，可以优先从以下类型中提炼：

### A. 数据与知识组织创新

例如：
- 非结构化资料向结构化知识库转化
- 统一数据底座或统一语义模型
- 规范、历史案例、工程规则的可计算化

### B. 算法与模型创新

例如：
- 混合式分类模型
- 多模态特征融合
- 知识图谱增强的大模型或智能体
- 约束驱动与生成式算法结合

### C. 工具链与流程闭环创新

例如：
- 从数据提取到分析再到交付的一体化闭环
- 设计、校核、修正、交付自动联动
- 自动化脚本 + 模板引擎 + 智能体的协同流程

### D. 工程交付创新

例如：
- 从“辅助分析”升级为“合规交付”
- 从单点出图升级为标准化批量生成
- 从专家经验依赖升级为规则驱动和知识驱动

### E. 应用模式与生态创新

例如：
- 自主可控
- 国产工具链适配
- 多单位协同与可推广机制

## 如何避免把创新点写成研究内容重复

如果一句话只是：
- `研究……`
- `开发……`
- `构建……`

那往往还只是研究内容，不是创新点。

要把它提升为创新点，必须加上：
- 相对谁不同
- 为什么这种不同成立
- 带来什么新能力

## 与其他章节的关系

创新点必须同时满足三种映射：
- 能回指 `项目背景` 里的差距或痛点
- 能回指 `研究内容` 里的具体任务或关键技术
- 能回指 `技术成果/技术指标` 里的交付与验证

如果做不到这三点，这个创新点大概率是不成立的。

## 触发样例

以下情形都必须触发本 skill：

### 样例 1

用户写：
- `某一章关键技术及创新点`

应触发原因：
- 章名直接命中 `创新点`

### 样例 2

用户写：
- `请单独写本项目的差异化优势和特色亮点`

应触发原因：
- 即使没有出现 `创新点`，`差异化优势`、`特色亮点` 仍应触发本 skill

### 样例 3

用户写：
- `在研究内容后补一节技术创新，重点说明和传统方案相比的新机制`

应触发原因：
- `技术创新` 明确属于本 skill，且要求正是“旧 -> 新”的对照写法

### 样例 4

用户写：
- `预期成果后面再写创新价值`

应触发原因：
- 即使创新内容出现在成果章节之后，只要要求写创新价值或创新性，仍然要调用本 skill

## 输出检查

写完后确认：
- 是否先做了搜索并登记来源
- 每个创新点是否有参照系
- 是否写清“旧 -> 新”的机制变化
- 是否能映射到研究内容、成果和指标
- 是否避免了空泛的“国际领先”“国内首创”表述

## 不要做的事

- 不要把研究内容原样改名当创新点
- 不要只写“先进、智能、高效”
- 不要写没有比较对象的创新点
- 不要写无法落到成果或指标上的创新点
- 不要无依据宣称唯一性或首创性
