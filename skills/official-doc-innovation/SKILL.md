# official-doc-innovation

## 定位

这是“创新点类内容”的规则库。

它不是单独发明一套漂亮说法，而是负责把：
- 现状差距
- 痛点瓶颈
- 研究内容
- 技术路线

收束成可成立、可辩护、可回指的创新表达。

本 skill 默认假定：
- `using-official-docs` 已完成工作区和 plan 初始化
- `official-doc-core` 已完成公共校验

本 skill 不负责：
- 创建工作区
- 创建 plan 文件
- 决定是否直接结束流程

本 skill 的正文结果必须写入 `workspace/outputs/<project-slug>/` 对应章节文件。
不要把完整创新点正文直接输出到终端。

本 skill 写完后，必须把控制权交还给 `using-official-docs` 继续推进下游流程。

## 先做什么

开始写之前先检查创新点所需的比较基线是否已经由 `official-doc-research` 做扎实。

必须先读取：
- `workspace/plan/<project-slug>/research-plan.md`
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/research-notes.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

如果创新类调研仍存在以下任一情况，应先回到 `official-doc-research` 补调研：
- 缺少对“传统方案 / 现有路线 / 同类项目”的近年基线
- 只有本项目视角，没有外部参照系
- 来源还停留在旧资讯稿或单篇宣传稿

只有在比较基线已形成、但某个创新点仍需补证时，才允许少量补充检索，例如：
- 同领域已有方案、竞品、论文、专利、白皮书
- 公开可见的主流技术路线和能力边界
- 同类项目的创新点写法和口径

记录到：
- `workspace/plan/<project-slug>/research-sources.md`
- `workspace/plan/<project-slug>/facts-ledger.md`

默认搜索工具优先级：
1. 当前 session 已暴露的 MCP 搜索 / connector 搜索工具
2. 其他会话可用的 MCP / connector 搜索工具

不允许使用内置 `Web Search`。
如果当前没有可用的 MCP 搜索工具，或 MCP 搜索报错无法继续：
- 停止联网搜索
- 在 `facts-ledger.md` 中记录搜索阻断原因
- 在 `progress.md` 中记录本章因缺少 MCP 搜索而暂停
- 不得改用内置搜索继续写正文

## 从附件提炼出的共性写法

### 1. 创新点不是“夸项目”，而是“写差异”

附件中的高质量创新点，都不是空喊“先进”“领先”，而是强调：
- 相对传统方法改变了什么
- 相对现有路线突破了什么
- 新增了什么机制、闭环或能力

### 2. 创新表达通常带有“旧 -> 新”的对照关系

附件里常见的有效表达方式包括：
- `从……转为……`
- `由……升级为……`
- `实现……一体化`
- `构建……闭环`
- `突破传统……模式`

这类表达之所以有效，是因为它天然带有参照系。

### 3. 创新点必须能回指研究内容和现状差距

不能出现以下问题：
- 创新点和前文现状分析无关
- 创新点和研究内容不对应
- 创新点和成果、指标无法挂钩

## 写作原则

### 原则 1：先找参照系

创新点必须明确基于什么作比较：
- 传统人工方式
- 现有单点工具
- 通用大模型方案
- 海内外主流技术路线
- 本领域既有工程流程

### 原则 2：写“机制创新”，不是只写“效果很好”

优先写：
- 技术机制怎么变了
- 数据组织方式怎么变了
- 流程闭环怎么建立了
- 交付方式怎么升级了

效果可以写，但应放在机制之后。

### 原则 3：创新点数量适中

一般写 3 到 8 点即可。

每一点都要足够完整，宁可少而实，不要多而虚。

## 推荐结构

每个创新点建议回答四件事：
- 现有方法的局限是什么
- 本项目提出了什么新机制/新方法/新系统
- 与现有路线相比差异在哪里
- 对工程实践带来什么直接价值

可以采用如下表达骨架：
- `针对……长期依赖……的问题，项目提出……`
- `区别于传统……方式，本项目通过……`
- `相较于现有……路线，本项目实现……`
- `从而形成……能力/闭环/优势`

## 常见创新点类型

根据附件内容，可以优先从以下类型中提炼：

### A. 数据与知识组织创新

例如：
- 非结构化资料向结构化知识库转化
- 统一数据底座或统一语义模型
- 规范、历史案例、工程规则的可计算化

### B. 算法与模型创新

例如：
- 混合式分类模型
- 多模态特征融合
- 知识图谱增强的大模型或智能体
- 约束驱动与生成式算法结合

### C. 工具链与流程闭环创新

例如：
- 从数据提取到分析再到交付的一体化闭环
- 设计、校核、修正、交付自动联动
- 自动化脚本 + 模板引擎 + 智能体的协同流程

### D. 工程交付创新

例如：
- 从“辅助分析”升级为“合规交付”
- 从单点出图升级为标准化批量生成
- 从专家经验依赖升级为规则驱动和知识驱动

### E. 应用模式与生态创新

例如：
- 自主可控
- 国产工具链适配
- 多单位协同与可推广机制

## 如何避免把创新点写成研究内容重复

如果一句话只是：
- `研究……`
- `开发……`
- `构建……`

那往往还只是研究内容，不是创新点。

要把它提升为创新点，必须加上：
- 相对谁不同
- 为什么这种不同成立
- 带来什么新能力

## 与其他章节的关系

创新点必须同时满足三种映射：
- 能回指 `项目背景` 里的差距或痛点
- 能回指 `研究内容` 里的具体任务或关键技术
- 能回指 `技术成果/技术指标` 里的交付与验证

如果做不到这三点，这个创新点大概率是不成立的。

## 触发样例

以下情形都必须触发本 skill：

### 样例 1

用户写：
- `第3章关键技术及创新点`

应触发原因：
- 章名直接命中 `创新点`

### 样例 2

用户写：
- `请单独写本项目的差异化优势和特色亮点`

应触发原因：
- 即使没有出现 `创新点`，`差异化优势`、`特色亮点` 仍应触发本 skill

### 样例 3

用户写：
- `在研究内容后补一节技术创新，重点说明和传统方案相比的新机制`

应触发原因：
- `技术创新` 明确属于本 skill，且要求正是“旧 -> 新”的对照写法

### 样例 4

用户写：
- `预期成果后面再写创新价值`

应触发原因：
- 即使创新内容出现在成果章节之后，只要要求写创新价值或创新性，仍然要调用本 skill

## 输出检查

写完后确认：
- 是否先做了搜索并登记来源
- 每个创新点是否有参照系
- 是否写清“旧 -> 新”的机制变化
- 是否能映射到研究内容、成果和指标
- 是否避免了空泛的“国际领先”“国内首创”表述

## 不要做的事

- 不要把研究内容原样改名当创新点
- 不要只写“先进、智能、高效”
- 不要写没有比较对象的创新点
- 不要写无法落到成果或指标上的创新点
- 不要无依据宣称唯一性或首创性
