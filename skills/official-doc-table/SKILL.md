# 公文表格 Skill

## 适用时机

以下任一情况出现时，应主动使用本 Skill：
- 用户明确说要补表
- 当前正文已出现表格引用位
- 当前章节天然依赖表格来承载结构化信息
- 入口 Skill 或主 Skill 判断当前下一步最合理的是补表

## 核心目标

把“表标题 + 字段定义 + 数据来源 + 待补项”生成标准表，而不是继续塞回正文段落。
本 Skill 必须以 `templates/<template>/table-catalog.md` 为主驱动，而不是临时凭感觉决定先补哪张表。

## 使用前必做

1. 确认当前模板是 `zs-feasibility-report` 还是 `full-research-template`
2. 读取对应 `table-catalog.md`
3. 识别 catalog 中的优先级与章节绑定关系
4. 锁定表 caption
5. 核对正文中该表的引用位置
6. 核对 `plan/facts-ledger.md` 中相关口径

## 选择逻辑

### 先决定“补哪张表”

默认按以下顺序判断：
1. catalog 中标为“必出”或“高优先”的表
2. 当前正文已经出现引用位的表
3. 当前阶段对应的重章节所依赖的表
4. 用户明确点名要求的表

### 再决定“补到什么粒度”

默认按 catalog 中给出的内容来定：
- 所属章节
- 作用
- 建议表头
- 可选补充列
- 主要材料来源
- 常见待补项
- 装配要求

不要跳过这些目录信息，直接凭经验写一个“差不多”的表。

## 输出格式

每张表至少输出：
- `tables/<template>/<table-file>.md`
- `tables/<template>/<table-file>-notes.md`

其中 `notes` 必须写清：
- 当前表服务哪个章节
- 当前表在 catalog 中的优先级
- 每列字段来自哪里
- 哪些值是 `【待补】`
- 哪些口径已与正文核对
- 装配时应插入到正文哪个位置

## 强制规则

- caption 不改
- 表头尽量贴近模板字段
- 不擅自新增“看起来合理”的指标列
- 数值、单位、时间必须与正文一致
- 不知道的字段直接写 `【待补】`
- 必须优先服从 catalog 中的“位置、作用、建议表头、装配要求”
- 更新 `plan/progress.md`

## ZS-项目可行性报告优先表格

- 表1～表6 默认为固定必出表
- 其中表3～表6属于配比刚性表，应优先服务第7章到第9章的“短正文 + 表”结构
- 第4章优先补表1，第5章优先补表2，第7章优先补表3和表4，第8章优先补表5，第9章优先补表6

## 完整科研项目模板优先表格

优先处理以下三层：
1. 第3章团队类高优先表
2. 第4章成果与任务分配类高优先表
3. 第8章资金与投资测算类高优先表

第一轮优先表默认包括：
- 表3-2 团队承担项目情况表
- 表3-3 项目领导团队成员
- 表3-4 项目核心技术成员
- 表4-1 主要报告类成果分布表
- 表4-2 交付专利及软著成果分布表
- 表4-3 交付标准成果分布表
- 表4-4 联合体成员成果指标分配表
- 资金来源结构表
- 分年度投资安排表
- 主要投资估算表

中优先和扩展表只有在正文已经下钻到对应专题层级时再补，不要一上来就把测试用例、缺陷统计、验证统计类表全部生成。

## notes 写法建议

每张 `-notes.md` 建议至少包含：
1. 表 caption 与所属章节
2. catalog 规定的优先级和用途
3. 当前采用的表头
4. 各列字段来源
5. 待补项
6. 与正文核对结果
7. 装配提示

## 与章节配比的关系

- 当章节被模板定义为“偏短章节”时，表格应承担更多信息量，正文保持克制
- 当章节是“重章节”时，表格主要起结构化补充作用，不要反过来替代正文主论证
- 对大模板，优先保证第3章、第4章、第8章的高优先表，而不是平均铺开所有章节

## 交付前检查

- 表号与正文引用一致
- 表头与章节语义一致
- 与正文口径冲突的地方已在 notes 中标出
- 已遵守 catalog 中的优先级和装配要求
