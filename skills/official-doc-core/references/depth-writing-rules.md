# 正式公文深度写作规则

本文件用于解决“材料搜到了，但正文仍然浅”的问题。正式项目公文的深度不来自术语堆砌，而来自清晰的论证链、可追溯依据、工程机制和可验证输出。

配套示例见 [depth-writing-examples.md](./depth-writing-examples.md)。写作时优先学习示例的论证方式，不要照抄示例主题、项目名称或指标数值。

## 一、深度写作的基本定义

正文不能只回答“有什么”，还要回答：
- 为什么现在需要做。
- 现有状态的差距在哪里。
- 差距产生的原因是什么。
- 本项目准备用什么技术机制解决。
- 预期形成什么可交付成果。
- 如何证明成果有效、可验收、可推广。

如果一段文字只是在重复“建设平台、提升能力、形成体系、赋能业务”，但没有写清对象、场景、机制、输出和验证方式，默认视为浅写法。

## 二、写作前必须建立“章节论证链”

五类共性章节正式起草前，先从 `research-plan.md`、`research-evidence.md`、`research-notes.md`、`facts-ledger.md` 和 `00-section-plan.md` 中抽取本章论证链。论证链至少包含：
- 核心判断：本章要证明什么。
- 事实依据：哪些已核验事实可以支撑。
- 差距或问题：现有做法哪里不足。
- 技术机制：项目采用什么方法、模型、流程或工具链。
- 输出结果：形成什么成果、能力、指标或交付件。
- 边界条件：哪些内容只能保守写，哪些仍需 `【待补】`。

若上述六项中少于四项能被填实，不得直接扩写正文。应回到 `official-doc-research` 补调研，或按“信息不足反馈规则”向用户问一个关键问题。

## 三、段落信息密度

正式段落通常应包含三个以上“信息原子”：
- 对象：围绕哪个系统、设备、工艺、流程、数据或场景。
- 问题：当前瓶颈、风险、低效或不确定性是什么。
- 原因：为什么现有方式不能充分解决。
- 方法：采用什么技术路线、组织方式或工程措施。
- 机制：方法为什么能起作用。
- 输出：形成什么模块、模型、报告、规范、平台或数据资源。
- 验证：通过什么测试、指标、场景或交付件证明有效。
- 约束：安全、合规、标准、兼容、成本、周期等边界。

单段只写“项目将提升效率、优化流程、增强能力”而没有对象和机制，默认不合格。

## 四、信息不足时如何处理

不要用泛泛的“请提供更多资料”打断用户。只在缺口会改变写作方向时提问，且一次只问一个问题。

应提问的情况：
- 不确定最终文体，可能导致章节口径不同。
- 不确定项目技术对象或应用场景，可能导致研究内容写偏。
- 不确定必须写入的硬指标、周期、单位、验收口径。
- 用户要求写具体数字、政策文号、预算、负责人，但台账没有依据。

不必提问的情况：
- 只是缺少一个非关键数字，可先写保守表述并标 `【待补】`。
- 只是某个来源没有达到强断言标准，可降级为趋势性表述。
- 用户没有指定子标题，但 `00-section-plan.md` 已能推导出合理结构。

反馈格式应具体：

```md
当前缺口：缺少项目适用船型和试点单位。
影响：会影响应用场景、指标阈值和成果验收口径。
需要补充：请确认本项目优先面向新造船设计、在役船改造，还是两者兼顾。
可先保守处理：若暂不确认，正文将按“船舶设计与分析通用场景”写，并在指标处保留 `【待补】`。
```

## 五、浅写法禁区

以下表达不是绝对禁用，但若没有对象、机制和依据支撑，默认需要改写：
- `提升效率`
- `降低成本`
- `优化流程`
- `赋能业务`
- `打造平台`
- `构建体系`
- `形成闭环`
- `达到先进水平`
- `实现智能化`
- `推动高质量发展`

错误写法：

```md
项目将打造智能化平台，赋能船舶设计业务，显著提升设计效率，形成完整闭环。
```

较好写法：

```md
项目拟面向船舶设计过程中规范查询、方案校核和报告编制效率不足的问题，构建由知识抽取、规则推理、模板生成和结果校核组成的自动化处理链路。该链路以船级社规范、企业历史案例和标准报告模板为输入，形成可复用的规则库和报告生成工具，使设计人员能够在统一平台内完成资料检索、计算结果整理和合规性复核，从而减少人工转录和重复校对环节。
```

## 六、五类章节的深度门槛

### 项目背景

至少形成以下链路：
`宏观背景 -> 细分场景 -> 国内外现状 -> 差距原因 -> 业务影响 -> 攻关必要性`。

不得只写“国外先进、国内不足”。必须按技术路线、工程成熟度、应用场景、标准合规、数据积累等维度比较。

### 研究内容

每项研究内容至少写清：
`问题来源 -> 研究对象 -> 子课题/子任务 -> 关键技术机制 -> 输入输出 -> 与其他任务的关系 -> 验证方式`。

不得把研究内容写成名词清单，也不得把子课题写成只有标题的一句话。

### 创新点

每个创新点至少写清：
`比较基线 -> 现有局限 -> 本项目新机制 -> 差异成立原因 -> 工程价值 -> 对应研究内容或成果`。

不得使用无依据的“首创、领先、填补空白、国际先进”等强词。

### 主要技术成果

每项成果至少写清：
`来源研究内容 -> 成果名称 -> 功能边界 -> 交付形态 -> 应用场景 -> 验收或使用方式`。

不得把“提升能力、推动应用、形成体系”当作成果名称。

### 主要技术指标

每项指标至少写清：
`指标对象 -> 指标定义 -> 目标值或区间 -> 测试条件 -> 验证方法 -> 依据或校准来源`。

没有依据的精确数字不能强写。可使用区间、阶段校准或 `【待补】`。

## 七、写作后的自检

每个章节写完后至少自查：
- 是否有论证链，而不是材料堆砌。
- 是否有现状差距和原因分析，而不只是背景介绍。
- 是否写出了技术机制，而不只是功能愿望。
- 是否能从研究内容推到成果，再从成果推到指标。
- 是否仍有口号式短句、项目符号正文、加粗小标签。
- 是否对未核验信息降级处理或标记 `【待补】`。
