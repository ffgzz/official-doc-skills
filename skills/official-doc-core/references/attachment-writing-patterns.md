# 公文结构与句式规则

## 一、文档级编号方案

编号必须由 `using-official-docs` 在初始化阶段统一确定，并写入 `project-brief.md` 与 `workspace/outputs/<project-slug>/00-section-plan.md`。五个专项章节 skill 只执行该方案，不得各自临场选择编号。

默认按文体选择：

- 可研报告、立项申请书、项目建议书、技术总结：采用正式报告型编号。一级章用 `一、`，二级节用 `（一）`，三级项用 `1.`，四级点用 `（1）`。
- 项目指南、攻关任务书、任务清单型材料：采用指南任务型编号。一级任务用 `1.`，二级要求用 `（1）`，三级动作或技术点用 `（a）`。
- 研究内容需要深度拆成课题/子课题时：章级仍服从全文编号，在“研究内容/主要攻关内容”内部可采用 `课题1`、`子课题1.1`、`1.1.1` 的技术分解编号。

硬规则：

- 同一层级不得混用 `（1）` 和 `1.1`。
- 同一章节内若已采用 `课题1 / 子课题1.1 / 1.1.1`，则不要再把同级条目写成 `（1）`。
- 编号只用于标题层级和必要的表格/指标项，不得把正文写成编号清单。
- 标题下必须有完整自然段展开，不能只有标题和一句功能愿望。
- 若用户或附件模板已经固定编号，优先服从模板；若未固定，按上述默认规则选型，并在计划文件中留痕。
- 扩写章节时不得靠连续追加 `### 11.`、`### 12.`、`### 13.` 这类流水号标题增加篇幅。正式可研/立项/项目建议书的主体章节应优先在既定二级节内用自然段、表格和少量必要三级标题展开。
- 普通章节不得出现 `### 一、...`、`### 二、...` 这类把一级编号降级使用的标题。正式报告型正文中，三级标题用 `1.`，四级标题用 `（1）`；研究内容内部的 `课题1 / 子课题1.1` 是唯一例外。
- 同一章节不得出现重复标题。若需要补充相近内容，应合并到原有标题下，不得另起同名或近似同名小节。

章节计划中必须同时记录“用户章节标记”和“正式输出标题”。例如用户提示词写“第一章 概述”，但目标文体为可行性报告且编号方案为正式报告型时，正式输出标题应写成 `一、概述`，不得在正式稿中输出 `第一章 概述`。`第一章/第二章` 只能作为用户原始标记或内部排序信息，不能覆盖 `一、/二、` 的正式输出编号。

正式稿不得新增 `00-section-plan.md` 中没有登记的二级节。若用户只要求第三章包含 `（一）项目研究的内容`、`（二）关键技术及创新点`，不得为了补字数追加 `（三）技术创新与突破`、`（四）研究方案可行性分析`、`（五）课题设置与任务分工` 等计划外小节。字数不足时，只能在既定小节内加深“问题、机制、输出、验证”和表图说明，不能靠增设新节凑篇幅。

三级、四级标题也要服从当前层级方案。正式报告型正文中，三级标题用 `1.`、四级标题用 `（1）`；只有研究内容内部的课题化拆解可以使用 `子课题1.1` 或 `1.1.1`。在实施方案、风险、成果、指标等普通章节中，不得出现 `11.1`、`13.2` 这类十进制目录标题。

## 二、附件式写法硬门禁

本文件中的“常用写法”是五个正文 skill 的硬约束，不是可选参考。凡涉及 `official-doc-project-background`、`official-doc-research-content`、`official-doc-innovation`、`official-doc-technical-achievements`、`official-doc-technical-indicators` 的正文段落，写完后必须自检是否满足以下要求；不满足时先回修，再交给 review 或 assemble。

每个关键段落至少覆盖以下 6 个要素中的 4 个：

`对象 / 问题 / 动作 / 机制 / 输出 / 验证`

重章节和成果、指标章节原则上应覆盖 5 个以上要素。若段落只能看出“做什么”，看不出“为什么做、怎么做、形成什么、如何验”，视为不合格。

五类正文的强制句式骨架如下：

- 背景与必要性：`在...层面，...对...提出更高要求；落实到...场景，现有...存在...问题，影响...；因此，有必要围绕...开展系统性攻关，形成...能力，为...提供支撑。`
- 研究内容：`针对...问题，本研究拟开展...，构建/开发...，通过...机制实现...；在...方面形成...；最终输出...，支撑...。`
- 关键技术：`...的关键不在于...，而在于...；项目拟通过...建立...，实现...；该技术位于...与...之间，承担...作用。`
- 创新点：`区别于...，本项目拟...；相较于...，该机制将...纳入...，由此实现...，支撑...。`
- 技术成果：`围绕...研究内容，形成...成果，用于...场景，以...形式交付，并通过...验收。`
- 技术指标：`在...条件下，实现...，...达到...；通过...测试/抽样复核/示范应用验证...。`

禁止把上述骨架压缩成口号式短句。每个正文 skill 可以根据章节实际调整措辞，但不能删掉“问题、机制、输出、验证”的核心信息。

扩写规则：

- 字数不足时，优先补充事实依据、差距原因、技术机制、成果边界、测试条件和验收口径。
- 不得新增 `00-section-plan.md` 没有登记的二级节。
- 不得新增一组连续流水号三级标题来替代正文扩写。
- 不得追加 `历史意义`、`时代价值`、`远景展望`、`政策建议`、`社会责任`、`一带一路`、`国际合作机遇`、`可持续发展规划`、`人才培养` 等与用户章节计划无关的泛化标题。
- 需要组织长内容时，优先使用“总述段 + 表格 + 表后说明 + 少量必要三级标题 + 自然段”的附件式结构。

## 三、正文骨架

正式公文的常用骨架不是“概念堆叠”，而是把对象、问题、研究动作和效果串成闭环。

### 1. 背景与必要性

推荐链路：

`宏观形势/政策要求 -> 行业或单位场景 -> 现有问题 -> 影响后果 -> 项目切入 -> 建设意义`

常用写法：

- `在...层面，...正在推动...，对...提出了更高要求。`
- `在...场景下，现有...仍主要依赖...，存在...问题。`
- `上述问题直接影响...，也制约...。`
- `因此，有必要围绕...开展系统性攻关，形成...能力，为...提供支撑。`

意义类内容宜按 `战略价值 / 技术价值 / 工程价值 / 数据资产价值 / 自主可控价值` 组织，每个角度都要写“解决什么、支撑什么、沉淀什么”，不要只写“意义重大”。

### 2. 研究内容与关键技术

研究内容应先有总述，再按任务包下钻。附件中常见形态是：

`总体研究思路 -> 研究内容表 -> 内容关系说明 -> 课题/子课题逐项展开 -> 关键技术提炼`

推荐句式：

- `本项目围绕...目标，拟按照“...、...、...”的主线组织研究内容。`
- `针对...问题，本研究拟开展...研究，开发/构建...，实现...。`
- `在...方面，研究...方法，形成...规则/模型/接口；在...基础上，开发...模块，实现...。`
- `最终形成...，为后续...提供...。`

子课题段落至少包含：

`针对的问题 -> 研究/开发/构建的对象 -> 技术机制 -> 形成的输出 -> 支撑的后续环节`

关键技术不等于研究内容换名。关键技术段落宜写成：

`技术难点 -> 核心机制 -> 链路位置 -> 工程作用`

可用句式：

- `...的关键不在于...，而在于...。`
- `项目拟通过...，建立...，实现...。`
- `该技术位于...与...之间，承担从...到...的转换作用。`

## 四、创新点写法

创新点必须带比较基线。附件式写法通常是“传统方式/现有路线”与“本项目机制”的对照。

推荐链路：

`比较对象 -> 现有局限 -> 本项目新机制 -> 差异成立原因 -> 工程价值 -> 对应研究内容/成果/指标`

常用句式：

- `区别于传统...方式，本项目拟构建...机制。`
- `相较于现有...路线，该机制将...统一纳入...，实现...。`
- `由此，...从...转向...，支撑...。`

不得在缺少主源支撑时写 `首创`、`领先`、`填补空白` 等绝对判断。

## 五、技术成果写法

成果必须由研究内容推导，建议先建立隐含映射，再输出正文或表格。

附件中常见开头：

`根据以上研究专题，项目实施完毕后，可形成以下技术成果。`

成果项至少写清：

`成果名称 -> 对应研究内容 -> 功能边界 -> 交付形态 -> 应用场景 -> 验收方式`

优先使用具体成果名：

- 系统/平台/引擎/工具链
- 模型/算法/规则引擎
- 数据集/样本库/知识库/规则库/模板库
- 技术报告/测试报告/规范草案/操作手册
- 专利/软著/论文/示范应用报告

若用户允许表格，成果表建议字段为：

`序号 | 技术成果 | 对应研究内容 | 交付形态 | 主要作用/验收方式`

## 六、技术指标写法

指标必须可测试、可验证，不应只列百分比。附件式指标通常包含输入条件和验收口径。

推荐句式：

- `在...输入信息准确、完整的前提下，实现...，...达到...。`
- `在...测试场景下，系统能够...，...准确率/完整率/通过率达到...。`
- `通过...测试、抽样复核或示范应用，验证...。`

每条指标至少包含：

`指标对象 -> 指标定义 -> 目标值/区间 -> 测试条件 -> 验证方法 -> 对应成果`

若用户允许表格，指标表建议字段为：

`指标类别 | 指标名称 | 目标值/区间 | 测试条件 | 验证方式 | 对应成果`

没有证据支撑的精确值用保守区间或 `【待补】`，不要伪造高精度数字。

## 七、常见不合格形态

出现以下情况时应回修：

- 各章标题层级自己选择编号，导致同一份稿件混用 `（1）`、`1.1`、`1.1.1`。
- 用大量连续的三级、四级标题堆出篇幅，正文没有在既定小节内形成连续论证。
- 研究内容只有名词和功能愿望，没有“针对问题、研究动作、形成输出”。
- 关键技术写成“采用AI技术、实现智能化”这类空话。
- 创新点没有比较对象，只写“创新、先进、高效”。
- 成果写成“提升能力、形成体系”，没有可交付实体。
- 指标只写“准确率高、效率提升”，没有测试条件和验收方式。
